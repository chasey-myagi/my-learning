1CLI 入口架构：4683 行的启动引擎

src/main.tsx 是整个 Claude Code 的心脏。它不是一个简单的命令行解析器，而是一个精心编排的启动流水线，将性能优化、安全检查、特性开关、会话恢复、多模式入口融合在一起。

Side-Effect-First 启动模式

文件的前 20 行就展示了 Claude Code 的性能哲学 -- 在 import 还没完成时就启动异步工作：

// Line 1-20: 三个并行 side-effect 在 import 阶段启动
profileCheckpoint('main_tsx_entry');       // 性能计时起点
startMdmRawRead();                         // MDM 子进程(plutil/reg)并行启动
startKeychainPrefetch();                   // macOS Keychain 双读并行(~65ms 优化)
性能洞察

Keychain prefetch 将两次串行的同步 spawn 操作（OAuth token + legacy API key）变成并行异步，节省约 65ms 的 macOS 启动时间。这种"import 阶段就开始干活"的策略，在后续 100+ 行的 import 完成时，子进程已经跑完了。

Commander.js 命令架构

main.tsx 使用 @commander-js/extra-typings（类型安全增强版 Commander）构建 CLI，通过 preAction hook 实现延迟初始化：

// Line 907: preAction hook -- 只在真正执行命令时初始化，help 展示时跳过
program.hook('preAction', async thisCommand => {
  await Promise.all([ensureMdmSettingsLoaded(), ensureKeychainPrefetchCompleted()]);
  await init();                          // 核心初始化(configs, env vars, telemetry)
  runMigrations();                       // 版本化配置迁移
  void loadRemoteManagedSettings();      // 企业远程配置(非阻塞)
  void loadPolicyLimits();               // 组织策略限制(非阻塞)
});

Dead Code Elimination 与 Feature Gates

通过 Bun 的 feature() API 实现编译期特性开关：

// 编译期消除：只有 COORDINATOR_MODE feature flag 开启时才引入 coordinator 模块
const coordinatorModeModule = feature('COORDINATOR_MODE')
  ? require('./coordinator/coordinatorMode.js')
  : null;

// KAIROS (Assistant Mode) -- 同样模式
const assistantModule = feature('KAIROS')
  ? require('./assistant/index.js')
  : null;

这种模式确保未启用的特性完全不进入打包产物，不增加任何运行时开销。目前已知的 feature flags 包括：COORDINATOR_MODE、KAIROS、DIRECT_CONNECT、SSH_REMOTE、LODESTONE、UDS_INBOX、BG_SESSIONS、TRANSCRIPT_CLASSIFIER、UPLOAD_USER_SETTINGS。

4 种 Hook 执行类型

通过 Zod discriminated union 定义，每种类型有独立的 schema 验证：

command (Shell)
执行 shell 命令。支持 bash/zsh/sh/powershell，可设置超时、自定义 spinner 消息、once（一次性）、async（后台）、asyncRewake（后台+exit code 2 唤醒模型）。
prompt (LLM)
用 LLM 评估 prompt。$ARGUMENTS 占位符注入 hook input JSON。可指定模型（默认小型快速模型）。适合做内容审查、分类判断。
http (Webhook)
POST hook input JSON 到指定 URL。支持自定义 headers，header 值可引用环境变量（需通过 allowedEnvVars 白名单）。适合外部系统集成。
agent (AI Verifier)
启动 AI Agent 验证器。给定 prompt 描述验证目标，Agent 自主执行验证。默认使用 Haiku 模型，60s 超时。适合自动化质量检查。

Hook 配置 Schema（Zod 验证）

// src/schemas/hooks.ts - Hook 配置 schema
const HooksSchema = z.partialRecord(
  z.enum(HOOK_EVENTS),           // 25+ 事件类型作为 key
  z.array(HookMatcherSchema())   // 每个事件可有多个 matcher
);

// Matcher 支持工具名匹配
const HookMatcherSchema = z.object({
  matcher: z.string().optional(),    // e.g. "Write", "Bash(git *)"
  hooks: z.array(HookCommandSchema())  // 匹配时执行的 hook 列表
});

// Hook 命令是 discriminated union
const HookCommandSchema = z.discriminatedUnion('type', [
  BashCommandHookSchema,   // type: 'command'
  PromptHookSchema,        // type: 'prompt'
  AgentHookSchema,         // type: 'agent'
  HttpHookSchema,          // type: 'http'
]);

Hook 执行管线

Event 触发 → if 条件过滤 → Matcher 匹配 → 执行 Hook → JSON 输出解析 → 决策分发
Hook 输出可控制：block（阻断）、additionalContext（注入上下文）、preventContinuation（停止后续）、updatedMCPToolOutput（修改输出）
if 条件过滤

if 字段使用权限规则语法（如 "Bash(git *)"），在 spawn hook 进程之前就过滤掉不匹配的调用。这避免了为每个 Bash 调用都启动 hook 子进程的开销。

Hook 来源优先级

Hook 可从多个来源注册，策略层面可控制哪些来源被允许：

  • settings.json -- 用户/项目级配置
  • Plugin hooks -- 已安装插件注册的 hooks（可被 allowManagedHooksOnly 策略禁止）
  • Session hooks -- 运行时动态注册（function hooks, SDK callback hooks）
  • Managed hooks -- 企业管理的 hooks（策略设置优先级最高）

shouldDisableAllHooksIncludingManaged 是全局 kill switch，--bare 模式跳过所有 hooks。

4多代理协调器模式

src/coordinator/coordinatorMode.ts 实现了一种全新的 AI 编排范式：Coordinator-Worker 架构。Coordinator 是一个只负责编排的 "管理者" Agent，通过 AgentTool 和 SendMessageTool 指挥多个 Worker Agent 并行工作。

模式激活

// 通过 bun:bundle feature flag + 环境变量双重控制
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE);
  }
  return false;
}

Coordinator System Prompt 设计

Coordinator 的 system prompt 是一份精心设计的约 350 行指令，核心原则：

明确角色分离
Coordinator 只有 3 个工具：AgentTool（spawn worker）、SendMessageTool（continue worker）、TaskStopTool（stop worker）。不直接写代码。
Phase-based 工作流
Research → Synthesis → Implementation → Verification。Coordinator 在 Synthesis 阶段必须理解研究结果，不能 lazy delegation。
并行是超能力
"Parallelism is your superpower" -- 独立任务必须并行启动。读操作自由并行，写操作按文件域串行。
Worker Prompt 自包含
"Workers can't see your conversation" -- 每个 prompt 必须包含完整的文件路径、行号、具体变更描述。禁止 "based on your findings" 这种 lazy 委托。

Continue vs Spawn 决策矩阵

场景机制原因
研究了要编辑的文件Continue (SendMessage)Worker 已有文件上下文
研究广泛但实现聚焦Spawn fresh (Agent)避免探索噪声拖累
纠正失败或扩展工作ContinueWorker 有错误上下文
验证其他 worker 的代码Spawn fresh验证者需要新鲜眼光
第一次尝试方向完全错误Spawn fresh清除锚定效应

5认证与安全链路

Claude Code 实现了一个完整的 OAuth 2.0 + PKCE 认证流，同时支持 API Key、Bedrock、Vertex、Foundry 等多种认证方式。安全不仅仅是登录 -- 它贯穿了从启动到运行的每一个环节。

OAuth 2.0 Authorization Code Flow with PKCE

generateCodeVerifier() → buildAuthUrl(PKCE) → AuthCodeListener (localhost) → Browser redirect → exchangeCodeForTokens() → fetchProfileInfo()

OAuthService 支持两种授权码获取方式：

  • 自动流：打开浏览器，重定向到 localhost 监听器自动捕获 code
  • 手动流：用户复制粘贴 code（非浏览器环境）
  • SDK 模式：skipBrowserOpen 选项，由 SDK 客户端控制显示

7会话持久化与恢复

Claude Code 的会话不是一次性的 -- 它支持 --resume、--continue、--fork-session 等多种恢复模式，以及 teleport（跨机器迁移会话）能力。

会话恢复矩阵

模式CLI Flag行为
Continue-c / --continue恢复当前目录最近的会话
Resume-r / --resume [id]按 session ID 恢复，或交互式选择
Fork--fork-session创建新 session ID 但继承历史
Resume at--resume-session-at <msg-id>恢复到指定消息（时间旅行）

会话存储

  • Transcript 文件：JSON 格式的完整消息历史
  • Session ID：UUID v4，可通过 --session-id 指定
  • PID 文件：~/.claude/sessions/ 下注册，用于并发会话检测
  • Session Title：可缓存，支持自定义 -n/--name
  • Agent Settings：保存 agent/mode/advisor 等设置
  • Worktree State：当使用 git worktree 时保存 worktree 路径和分支

SessionStart Hook 触发矩阵

不同恢复方式触发不同的 hook source：

processSessionStartHooks('startup')  // 新会话启动
processSessionStartHooks('resume')   // --resume/--continue 恢复
processSessionStartHooks('clear')    // /clear 命令
processSessionStartHooks('compact')  // 上下文压缩后

关键细节：resume/continue 模式跳过 main.tsx 中的 startup hooks（由 conversationRecovery.ts 触发 'resume' 源），防止 hooks 执行两次（gh-30825）。

9配置迁移与 Schema 验证

Claude Code 维护一个版本化配置迁移系统，确保用户升级时配置无缝衔接。当前迁移版本为 11。

迁移版本追踪

// main.tsx:325 - 版本化迁移
const CURRENT_MIGRATION_VERSION = 11;

function runMigrations(): void {
  if (getGlobalConfig().migrationVersion !== CURRENT_MIGRATION_VERSION) {
    migrateAutoUpdatesToSettings();
    migrateBypassPermissionsAcceptedToSettings();
    migrateEnableAllProjectMcpServersToSettings();
    resetProToOpusDefault();
    migrateSonnet1mToSonnet45();
    migrateLegacyOpusToCurrent();
    migrateSonnet45ToSonnet46();
    migrateOpusToOpus1m();
    migrateReplBridgeEnabledToRemoteControlAtStartup();
    if (feature('TRANSCRIPT_CLASSIFIER')) {
      resetAutoModeOptInForDefaultOffer();
    }
    saveGlobalConfig(prev => ({
      ...prev,
      migrationVersion: CURRENT_MIGRATION_VERSION
    }));
  }
}

10工程启示：Harness 设计的核心模式

深入 Claude Code 源码后，可以提炼出以下 AI Harness 工程化的核心设计模式：

1. 启动性能：每毫秒都是战场

Pattern: Parallel Prefetch

在 import 阶段就启动子进程（MDM plutil、Keychain 读取），让 I/O 与 135ms 的 module evaluation 并行。startDeferredPrefetches() 在首次渲染后才启动非关键工作（user context、git status、tips），避免与 critical path 竞争事件循环。

2. 安全：分层信任模型

安全不是一个开关，而是一个渐进式信任链：

  • Boot 层：anti-debugging、Windows PATH 防护
  • Config 层：safe env vars → trust dialog → full env vars
  • Runtime 层：Sandbox、Permission modes（plan/auto/bypassPermissions）、per-tool 权限规则
  • Hook 层：if 条件过滤、managedHooksOnly 策略、async hook 限制

3. 可扩展性：事件驱动 + 四种执行器

Hook 系统的 command/prompt/http/agent 四种类型覆盖了从本地脚本到远程 Webhook 再到 AI 自主验证的全部扩展场景。if 条件过滤和 matcher 模式匹配确保 hooks 只在必要时执行。

4. 弹性：Fail-Open 哲学

远程配置、策略限制、GrowthBook feature flags -- 所有外部依赖都遵循 fail-open 原则。网络故障时 CLI 继续工作（只是可能没有最新的企业策略）。这对一个开发者工具来说至关重要 -- 宁可用旧配置，不可阻塞开发流程。

5. 迁移：Versionized + Idempotent

每次迁移都是幂等的（检查当前值再决定是否写入），通过 CURRENT_MIGRATION_VERSION 全局版本号批量管理。新模型发布时只需添加一个迁移函数并 bump 版本号。@[MODEL LAUNCH] 注释标记提醒开发者需要关注的位置。

6. 多模态运行时：一套代码，N 种形态

通过 clientType + feature() + isNonInteractiveSession 三维判断，同一个 main.tsx 入口支撑了 CLI、SDK、MCP、Bridge、SSH、Direct Connect、Assistant 等所有运行模式。代码通过条件分支而非独立入口文件实现，保证了行为一致性。

7. 编排抽象：Coordinator 不写代码

Coordinator 模式的核心洞察是理解与执行分离。Coordinator 必须理解 Worker 的研究结果（Synthesis），然后写出自包含的 Worker prompt。这不是 delegation -- 是 comprehension-based orchestration。

总结

Claude Code 的 Harness Engineering 展示了一个成熟的 AI Agent 运行时应该长什么样：极致的启动性能（parallel prefetch + deferred work），分层的安全模型（从 boot 到 runtime 的渐进信任），可编程的生命周期（25+ hook 事件 x 4 种执行器），弹性的外部集成（fail-open 远程配置），以及统一的多模态入口（一个 main.tsx 支撑所有运行形态）。这不仅仅是一个 CLI 工具 -- 它是一个完整的 AI Agent 操作系统。
