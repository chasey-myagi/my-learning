01查询引擎总体架构

QueryEngine 是 Claude Code 整个对话生命周期的核心管控器，一个 QueryEngine 实例对应一次完整会话，每次 submitMessage() 启动一个新的"轮次"。

核心类 QueryEngine

src/QueryEngine.ts

QueryEngine 是一个 AsyncGenerator 驱动的状态机。它的 submitMessage() 方法返回 AsyncGenerator<SDKMessage>，调用者通过 for await 逐步消费产出的 SDK 消息。

export class QueryEngine {
  private config: QueryEngineConfig
  private mutableMessages: Message[]       // 会话历史（跨轮次持久化）
  private abortController: AbortController // 中断控制
  private permissionDenials: SDKPermissionDenial[]
  private totalUsage: NonNullableUsage     // 累计 token 用量
  private readFileState: FileStateCache    // 文件状态缓存
  private discoveredSkillNames: Set<string>()

  async *submitMessage(prompt, options): AsyncGenerator<SDKMessage> {
    // 1. 构建系统提示词
    // 2. 处理用户输入（斜杠命令、附件等）
    // 3. 调用 query() 进入主循环
    // 4. 消费并转发消息给 SDK 调用者
  }
}

02主查询循环 -- while(true) 状态机

query.ts 中的 queryLoop() 是一个无限循环的 AsyncGenerator，通过 State 对象在迭代间传递可变状态，用 Continue 语义标记转换原因。

State 结构

每次循环迭代开始时解构 State，读取当前状态：

type State = {
  messages: Message[]                    // 消息数组（含工具结果）
  toolUseContext: ToolUseContext          // 工具执行上下文
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number   // max_output_tokens 恢复尝试次数
  hasAttemptedReactiveCompact: boolean   // 是否已尝试响应式压缩
  maxOutputTokensOverride: number | undefined
  pendingToolUseSummary: Promise<ToolUseSummaryMessage | null> | undefined
  stopHookActive: boolean | undefined
  turnCount: number                      // 当前轮次计数
  transition: Continue | undefined       // 上一次迭代的转换原因
}

循环流程

while (true) { ┌─────────────────────────────────────────────────┐ │ 1. 解构 State，初始化查询链追踪 │ │ 2. applyToolResultBudget() - 工具结果预算控制 │ │ 3. snipCompactIfNeeded() - 历史裁剪 │ │ 4. microcompactMessages() - 微压缩 │ │ 5. applyCollapsesIfNeeded() - 上下文折叠 │ │ 6. autoCompactIfNeeded() - 自动压缩 │ ├─────────────────────────────────────────────────┤ │ 7. callModel() → 流式接收 SSE 事件 │ │ ├─ 工具块 → StreamingToolExecutor.addTool() │ │ ├─ 文本块 → yield 给调用者 │ │ └─ 错误块 → 判断是否 withhold（PTL/MOT） │ ├─────────────────────────────────────────────────┤ │ 8. 后模型阶段 (needsFollowUp == false?) │ │ ├─ PTL 恢复: collapse drain → reactive compact│ │ ├─ MOT 恢复: escalate 8k→64k → multi-turn │ │ ├─ Stop Hooks 评估 │ │ └─ Token Budget 检查 │ │ → return { reason: 'completed' } │ ├─────────────────────────────────────────────────┤ │ 9. 工具执行阶段 (needsFollowUp == true) │ │ ├─ getRemainingResults() / runTools() │ │ ├─ getAttachmentMessages() - 文件变更/记忆 │ │ ├─ 队列命令注入 │ │ └─ maxTurns 检查 │ │ → state = { ...next, turnCount++ } │ │ → continue │ └─────────────────────────────────────────────────┘ }

状态转换（Transition）类型

每次 continue 时，State 的 transition 字段记录了为什么需要继续循环：

Transition 原因触发条件说明
next_turn工具返回结果后正常的下一轮工具循环
reactive_compact_retryPrompt Too Long 后响应式压缩后重试
collapse_drain_retry上下文折叠后排空折叠队列后重试
max_output_tokens_escalate8k 输出不够升级到 64k 输出限制
max_output_tokens_recovery输出截断注入恢复消息让模型继续
stop_hook_blockingStop Hook 阻止停止钩子要求重试
token_budget_continuation预算未用完nudge 消息推动模型继续
设计亮点

queryLoop 使用 显式 State 对象 + continue 替代递归调用。避免递归导致的栈溢出和 GC 压力，同时 transition 字段让测试可以断言恢复路径是否触发——无需检查消息内容。

流式回退处理

当 onStreamingFallback 触发时，query.ts 执行清理：

  • 为已产出的孤立助手消息 yield tombstone，通知 UI 和 transcript 移除
  • 清空 assistantMessages、toolResults、toolUseBlocks
  • 调用 streamingToolExecutor.discard() 丢弃进行中的工具
  • 创建新的 StreamingToolExecutor 实例
Withhold 机制

某些错误消息（Prompt Too Long、Max Output Tokens、Media Size Error）不会立即 yield 给调用者，而是被 withhold（扣留），等待恢复逻辑判定。这防止了 SDK 消费者（如 Desktop 端）在收到 error 后提前终止会话，而恢复循环仍在运行的情况。

04StreamingToolExecutor -- 流式并行工具编排

当模型流式输出多个 tool_use 块时，StreamingToolExecutor 在流接收过程中就开始执行工具，大幅缩短整体延迟。

并发控制模型

src/services/tools/StreamingToolExecutor.ts

模型流输出: [text][tool_use A][tool_use B][tool_use C][message_stop] │ │ │ ▼ ▼ ▼ addTool(A) addTool(B) addTool(C) │ │ │ ▼ ▼ ▼ ┌──────────────────────────────────────┐ │ processQueue() │ │ │ │ A: isConcurrencySafe? ───┐ │ │ B: isConcurrencySafe? ───┤ 并行 │ │ C: !isConcurrencySafe │ 执行 │ │ ▼ │ │ [A 执行] [B 执行] │ │ ↓ ↓ │ │ [A 完成] [B 完成] │ │ ↓ │ │ [C 排队等待] → [C 执行] │ └──────────────────────────────────────┘

ToolStatus 状态流转

type ToolStatus = 'queued' | 'executing' | 'completed' | 'yielded'

// 状态流转:
// queued → executing → completed → yielded
//   ↓ (abort/error)
// queued → completed (synthetic error) → yielded

并发安全判定

每个工具通过 isConcurrencySafe(parsedInput) 方法声明自己是否可以并发执行：

  • 并发安全工具：Read、Glob、Grep 等只读工具可以彼此并行
  • 非并发工具：Write、Edit、Bash 等需要独占执行
  • 判定规则：所有当前执行中的工具都是并发安全 + 新工具也是并发安全 = 允许并行

错误级联

Bash 错误级联

只有 Bash 工具的错误会触发兄弟取消（siblingAbortController.abort('sibling_error')）。Bash 命令通常有隐式依赖链（如 mkdir 失败后续命令无意义），而 Read/WebFetch 等独立工具的失败不应影响其他工具。

Progress 消息即时传递

工具执行过程中产生的 progress 类型消息不等待工具完成，而是通过 pendingProgress 队列 + progressAvailableResolve Promise 信号立即传递给调用者，实现实时进度反馈。

// getRemainingResults() 中的等待逻辑
if (this.hasExecutingTools() && !this.hasCompletedResults() && !this.hasPendingProgress()) {
  const executingPromises = this.tools
    .filter(t => t.status === 'executing' && t.promise)
    .map(t => t.promise!)
  const progressPromise = new Promise<void>(resolve => {
    this.progressAvailableResolve = resolve
  })
  // 任一工具完成 OR 任一 progress 可用 → 继续
  await Promise.race([...executingPromises, progressPromise])
}

中断行为分类

中断原因行为说明
user_interrupted按 interruptBehaviorcancel 型工具立即终止，block 型继续
sibling_error生成合成错误Bash 兄弟出错，取消排队中的工具

05重试与错误恢复 -- withRetry 全解析

withRetry 是一个 AsyncGenerator，将 API 调用包装为具有指数退避、模型降级、Fast Mode 回退、持久重试等能力的健壮调用层。

核心参数

src/services/api/withRetry.ts

const DEFAULT_MAX_RETRIES = 10
const FLOOR_OUTPUT_TOKENS = 3000
const MAX_529_RETRIES = 3
const BASE_DELAY_MS = 500

重试决策树

API Error │ ├── 用户中断 (AbortError) ─────────→ 抛出，不重试 │ ├── Fast Mode 活跃 + 429/529? │ ├── retry-after < 20s ───────→ 等待后重试（保持 Fast Mode） │ ├── overage 被禁用 ──────────→ 永久关闭 Fast Mode │ └── 其他 ────────────────────→ 触发冷却期，降级标准速度 │ ├── 529 + 非前台查询源? ───────────→ 立即放弃（防止级联放大） │ ├── 529 + 连续 >= 3次? │ ├── 有 fallbackModel ────────→ 抛出 FallbackTriggeredError │ └── 无 fallbackModel ────────→ 抛出 CannotRetryError │ ├── Max Tokens 溢出 (400)? │ └── 计算 adjustedMaxTokens ──→ 带 maxTokensOverride 重试 │ ├── 持久模式 (UNATTENDED_RETRY)? │ └── 429/529 无限重试 ────────→ 心跳式 yield（30s 间隔） │ └── 正常重试 └── 指数退避: min(500ms * 2^(attempt-1), 32s) + 25% jitter

模型降级（Fallback）

当连续 529 错误达到 MAX_529_RETRIES（3次）时：

  1. 抛出 FallbackTriggeredError
  2. query.ts 捕获该错误，将 currentModel 切换为 fallbackModel
  3. 为已产出的孤立消息生成 tool_result 错误块
  4. 清空所有中间状态，用新模型重新发起请求
  5. 如果消息中有 thinking 签名块，需要 stripSignatureBlocks（签名与模型绑定）
  6. yield 系统消息通知用户模型已切换

持久重试模式

通过 CLAUDE_CODE_UNATTENDED_RETRY 环境变量启用，专为无人值守场景设计：

429 速率限制

读取 anthropic-ratelimit-unified-reset 头部获取重置时间，最多等待 6 小时。每 30 秒产出心跳 SystemAPIErrorMessage，防止宿主环境判定会话空闲。

529 过载

指数退避最高到 5 分钟，不使用模型降级。for 循环的 attempt 被钳位（clamped）以防止终止，使用独立的 persistentAttempt 计数器。

06推测执行 -- Speculation 机制

Speculation 是 Claude Code 的一项前沿优化：在用户还未确认发送时，预测性地执行模型建议的操作，接受后直接注入结果，跳过等待时间。

核心流程

src/services/PromptSuggestion/speculation.ts

用户停止输入 → promptSuggestion 生成建议 → 用户看到建议 │ ▼ startSpeculation() │ ┌───────────────────┤ ▼ ▼ 创建 overlay 目录 runForkedAgent() (copy-on-write) (最多 20 轮/100 消息) │ │ │ ┌──────────────┤ │ │ canUseTool 检查 │ │ ├─ Read/Glob/Grep → 允许 │ │ ├─ Write/Edit → 检查权限模式 │ │ │ ├─ acceptEdits/bypass → overlay 隔离后允许 │ │ │ └─ 其他 → boundary(edit)，中止 │ │ ├─ Bash(只读) → 允许 │ │ ├─ Bash(写入) → boundary(bash)，中止 │ │ └─ 其他工具 → boundary(denied_tool)，中止 │ │ │ ▼ │ boundary 到达或完成 ▼ 用户按 Tab 接受 │ ┌─────────┤ ▼ ▼ copyOverlayToMain() prepareMessagesForInjection() (overlay → 主目录) (过滤 thinking/中断消息) │ │ └──────────┬───────────┘ ▼ 注入消息到会话历史 │ ┌──────┤ ▼ ▼ complete? incomplete? │ │ 无需查询 需要继续查询 (Tab 直达) (模型接续)

安全边界（Boundary）

Boundary 类型触发条件行为
completeAgent 自然完成全部结果可注入，不需后续查询
bash非只读 Bash 命令中止推测，保留已完成工具结果
edit文件编辑且权限不足中止推测，等用户确认
denied_tool未知/不允许的工具中止推测，保留只读结果

Overlay 文件隔离

推测执行中的文件写入不直接修改工作目录，而是写入临时 overlay 目录：

  • 写入路径：/tmp/claude/speculation/{pid}/{id}/{相对路径}
  • Copy-on-Write：首次写入时将原始文件复制到 overlay
  • 读取重定向：如果文件已在 overlay 中写入过，后续读取也重定向到 overlay
  • 接受时：copyOverlayToMain() 将 overlay 文件复制回主目录
  • 中止/错误时：safeRemoveOverlay() 删除整个 overlay 目录

流水线建议（Pipelined Suggestion）

当推测执行完成后，系统不闲等——它立即开始为下一轮生成建议。用户接受当前推测后，下一个建议已经准备好，形成连续的推测链。

性能影响

推测执行可以节省每次工具调用的完整往返时间。对于典型的 Read + Grep + Edit 工作流，这可以节省 5-15 秒。会话累计节省时间通过 speculationSessionTimeSavedMs 跟踪，并展示给用户。

07令牌管理与成本追踪

Claude Code 维护了多层次的令牌跟踪系统——从 API 响应的精确 usage 到跨压缩边界的估算，再到美元成本上限控制。

tokenCountWithEstimation

src/utils/tokens.ts

这是最核心的令牌估算函数，在自动压缩阈值判断、阻塞限制检查等场景广泛使用：

export function tokenCountWithEstimation(messages: readonly Message[]): number {
  // 1. 从后往前找到最近一个有 usage 的助手消息
  // 2. 处理同一 API 响应被拆分为多条消息的情况（相同 message.id）
  // 3. 使用精确 usage 数据：input + cache_creation + cache_read + output
  // 4. 对 usage 之后的消息使用粗略估算：
  //    roughTokenCountEstimationForMessages(messages.slice(i + 1))
  //    （大致是 chars / 4）
  return getTokenCountFromUsage(usage)
       + roughTokenCountEstimationForMessages(messages.slice(i + 1))
}

三层成本控制

1. maxBudgetUsd -- 美元预算

QueryEngine.submitMessage 每消费一条消息后检查 getTotalCost() >= maxBudgetUsd，超出则终止并返回 error_max_budget_usd 结果。

2. taskBudget -- API 令牌预算

通过 output_config.task_budget 传递给 API，让服务端感知预算。跨压缩边界时，客户端计算 remaining = total - preCompactContext。

3. Token Budget -- 输出令牌预算

Feature-gated 的 TOKEN_BUDGET 机制。checkTokenBudget() 在每轮结束时检查：如果输出令牌未达预算的 90%，注入 nudge 消息让模型继续；如果连续 3 次增量 < 500 tokens，判定为收益递减并停止。

Usage 数据流

API Response (SSE) │ ├── message_start: usage.input_tokens, cache_*_tokens ├── message_delta: usage.output_tokens (增量) └── message_stop: 触发累加 │ ▼ currentMessageUsage = updateUsage(current, delta) │ ▼ (message_stop) totalUsage = accumulateUsage(total, currentMessage) │ ▼ addToTotalSessionCost(calculateUSDCost(usage, model)) │ ▼ cost-tracker.ts → getTotalCost() / getModelUsage()

finalContextTokensFromLastResponse

用于 task_budget 的跨压缩计算。优先使用 usage.iterations[-1]（包含服务端工具循环后的最终窗口），否则回退到 input_tokens + output_tokens（不含缓存令牌）。

08思考模式集成

Extended Thinking 让模型在生成最终回答前进行内部推理。Claude Code 精心管理思考块的生命周期，遵循严格的 API 规则以避免"一整天的调试和拔头发"。

The rules of thinking are lengthy and fortuitous. They require plenty of thinking of most long duration and deep meditation for a wizard to wrap one's noggin around. -- query.ts 源码注释

三大铁律

  1. 开启规则：包含 thinking 或 redacted_thinking 块的消息必须属于 max_thinking_length > 0 的请求
  2. 位置规则：thinking 块不能是消息中的最后一个块
  3. 保持规则：thinking 块必须在助手轨迹期间保持——如果该轮包含 tool_use，则思考块必须保留到后续的 tool_result 和下一个助手消息

思考配置类型

type ThinkingConfig =
  | { type: 'disabled' }               // 完全关闭
  | { type: 'adaptive' }               // 自适应（模型自行决定）
  | { type: 'enabled'; budgetTokens: number }  // 显式预算

模型能力匹配

paramsFromContext 中的思考参数构建遵循严格的模型能力检查：

if (hasThinking && modelSupportsThinking(model)) {
  if (modelSupportsAdaptiveThinking(model)) {
    // 支持自适应：不设预算，让模型自行决定
    thinking = { type: 'adaptive' }
  } else {
    // 不支持自适应：使用显式预算
    let thinkingBudget = getMaxThinkingTokensForModel(model)
    // 预算不能超过 maxOutputTokens - 1
    thinkingBudget = Math.min(maxOutputTokens - 1, thinkingBudget)
    thinking = { type: 'enabled', budget_tokens: thinkingBudget }
  }
}

思考清除（Thinking Clear）

当距上次 API 调用超过 1 小时（CACHE_TTL_1HOUR_MS），系统会启用 context_management 中的 thinking clear 策略——指示 API 清除历史中的思考块以释放缓存空间。这通过 sticky-on latch 实现：一旦启用就在会话内持续，避免 header 翻转导致缓存失效。

推测执行中的思考处理

推测执行的 prepareMessagesForInjection() 会过滤掉所有 thinking 和 redacted_thinking 块。因为推测消息要注入到主会话历史中，而思考块的签名与模型绑定，跨模型注入会导致 API 400 错误。

QueryEngine 层的终止条件

QueryEngine.submitMessage 在消费 query() 的输出时，还有额外的终止检查：

  • maxBudgetUsd：每条消息后检查 getTotalCost() >= maxBudgetUsd，产出 error_max_budget_usd
  • Structured Output 重试上限：JSON Schema 模式下，如果 StructuredOutput 工具调用超过 MAX_STRUCTURED_OUTPUT_RETRIES（默认 5），产出 error_max_structured_output_retries
  • max_turns_reached 附件：query.ts yield 的特殊附件消息，QueryEngine 转为 error_max_turns 结果

chainId 和 depth 贯穿所有 analytics 事件，用于关联同一轮次中的多次 API 调用，追踪压缩、降级、恢复等操作的因果链。

Output Tokens 恢复循环

当模型输出被截断（max_output_tokens stop reason）时，系统最多尝试 3 次恢复：

  1. Escalate（一次性）：如果使用了 8k 的 capped 默认值，直接升级到 64k——不注入消息，不增加轮次
  2. Multi-turn recovery（最多 3 次）：注入 "Output token limit hit. Resume directly -- no apology, no recap..." 让模型从断点继续
  3. 放弃：恢复次数耗尽后，surface 扣留的错误消息
设计哲学

整个查询引擎的设计遵循"fail-open with graceful degradation"原则。每一层恢复机制都有明确的退出条件，避免无限循环。布尔守卫（如 hasAttemptedReactiveCompact）确保每种恢复策略只尝试一次。transition 字段让测试可以验证恢复路径是否按预期触发，无需深入检查消息内容。
