CHAPTER 02

技能格式规范：SKILL.md 与 Frontmatter

技能的物理形态是一个目录，包含 SKILL.md 文件。该文件使用 YAML Frontmatter + Markdown Body 的双层结构。

目录结构

.claude/skills/
  my-skill/
    SKILL.md          # 唯一必需文件
    helper.sh         # 可选的辅助文件
    templates/        # 可选的子目录
      prompt.txt

完整 Frontmatter Schema

---
# 基础信息
description: "技能的描述（展示在列表中）"
name: "显示名称（可选，默认用目录名）"
when_to_use: "模型何时应该使用此技能（详细场景描述）"
version: "1.0.0"

# 调用控制
user-invocable: true          # 用户可否通过 /skill-name 调用
disable-model-invocation: false  # 禁止模型主动调用
argument-hint: "file path"    # 参数提示（灰色显示在命令后）
arguments:                    # 命名参数列表
  - file_path
  - output_dir

# 工具与模型
allowed-tools:                # 技能激活时可用的工具
  - Bash(npm:*)
  - Read
  - Edit
model: sonnet                 # 模型覆盖（haiku/sonnet/opus/inherit）
effort: high                  # 思考努力级别

# 执行模式
context: fork                 # inline（默认）或 fork（子 Agent）
agent: Bash                   # fork 时使用的 Agent 类型

# 条件激活
paths:                        # gitignore 风格的路径模式
  - "src/**/*.ts"
  - "tests/"

# Hooks
hooks:
  PreToolUse:
    - matcher: Bash
      hooks:
        - command: "echo pre-hook"
          type: command

# Shell
shell: bash                   # !` ` 块使用的 shell（bash/powershell）
---

# 技能正文（Markdown）

这里是技能的 prompt 内容...

支持变量替换：
- ${1}、${2} — 位置参数
- ${file_path} — 命名参数
- ${CLAUDE_SKILL_DIR} — 技能所在目录
- ${CLAUDE_SESSION_ID} — 当前会话 ID

支持内联 Shell 执行：
!`ls -la`

```! bash
echo "这会在技能加载时执行"
```

Frontmatter 解析流程

源码位于 src/utils/frontmatterParser.ts，解析分三步：

❶ 正则匹配 --- 分隔符
→
❷ YAML 解析 frontmatter 块
→
❸ 类型校验 + 默认值填充

CHAPTER 03

技能加载与发现引擎

技能加载系统是整个扩展架构的核心。源码主体在 src/skills/loadSkillsDir.ts（约 1000 行），负责从 4 个层级的目录中发现、加载、去重和管理技能。

四层技能源

层级路径SettingSource说明
Managed{managedPath}/.claude/skills/policySettings企业策略推送的技能，最高优先级
User~/.claude/skills/userSettings用户全局技能
Project.claude/skills/（项目目录到 home 的每一级）projectSettings项目级技能
Additional--add-dir 指定的额外目录projectSettings外挂目录技能

加载流程

❶ getSkillDirCommands(cwd)
→
❷ 并行扫描 4 层目录
→
❸ realpath 去重
→
❹ 条件/无条件分离
// src/skills/loadSkillsDir.ts - 核心加载逻辑
const [managedSkills, userSkills, projectSkillsNested,
       additionalSkillsNested, legacyCommands] = await Promise.all([
  loadSkillsFromSkillsDir(managedSkillsDir, 'policySettings'),
  loadSkillsFromSkillsDir(userSkillsDir, 'userSettings'),
  Promise.all(projectSkillsDirs.map(dir =>
    loadSkillsFromSkillsDir(dir, 'projectSettings'))),
  Promise.all(additionalDirs.map(dir =>
    loadSkillsFromSkillsDir(join(dir, '.claude', 'skills'), 'projectSettings'))),
  loadSkillsFromCommandsDir(cwd),  // 兼容旧的 /commands/ 目录
])

去重机制

通过 realpath() 解析符号链接到规范路径，确保同一文件不会通过不同路径（如符号链接、重叠的父目录）被重复加载。这比 inode 比较更可靠，避免了虚拟文件系统/NFS 上 inode 不准确的问题。

条件技能（Conditional Skills）

带有 paths frontmatter 的技能不会立即激活。它们被存储在 conditionalSkills Map 中，等待模型触及匹配路径时才激活：

// 当模型操作文件时触发
activateConditionalSkillsForPaths(filePaths, cwd)
// 使用 gitignore 风格匹配
if (skillIgnore.ignores(relativePath)) {
  dynamicSkills.set(name, skill)      // 移入动态技能
  conditionalSkills.delete(name)       // 移出条件池
  activatedConditionalSkillNames.add(name)
}
性能优化：技能加载结果通过 lodash memoize 缓存，以 cwd 为 key。但可用性检查（meetsAvailabilityRequirement、isCommandEnabled）每次调用 getCommands() 都会重新评估，确保认证状态变化（如 /login）立即生效。

动态技能发现

除了条件技能，系统还支持运行时动态发现技能目录。新发现的技能通过信号（createSignal）通知缓存清除，避免循环依赖。

CHAPTER 04

SkillTool 执行引擎

SkillTool 是模型调用技能的唯一入口，定义在 src/tools/SkillTool/SkillTool.ts。它是一个标准的 Tool，通过 buildTool() 构建，拥有输入验证、权限检查、执行三阶段。

输入 Schema

z.object({
  skill: z.string()  // 技能名称，如 "commit", "review-pr", "pdf"
  args: z.string().optional()  // 可选参数
})

执行流程

❶ validateInput
→
❷ checkPermissions
→
❸ call (inline/fork)
→
❹ 返回结果

两种执行模式

📄 Inline 模式（默认）

技能 prompt 直接展开到当前对话上下文中。模型在同一个 token 预算内处理技能内容。

// 返回结果
{
  success: true,
  commandName: "skill-name",
  allowedTools: [...],
  status: "inline"
}
🔌 Fork 模式

技能在隔离的子 Agent 中运行，拥有独立的 token 预算。通过 runAgent() 驱动，支持进度回报。

// context: 'fork' 触发
{
  success: true,
  commandName: "skill-name",
  status: "forked",
  agentId: "uuid",
  result: "执行结果文本"
}

Prompt 构建

技能列表被嵌入到 SkillTool 的 prompt 中，通过 system-reminder 消息展示给模型。列表有预算控制：

// src/tools/SkillTool/prompt.ts
const SKILL_BUDGET_CONTEXT_PERCENT = 0.01  // 上下文窗口的 1%
const MAX_LISTING_DESC_CHARS = 250         // 每条描述最大字符数

// Bundled skills 描述永不截断
// 其他技能描述按预算等比截断
// 极端情况下非 bundled 只显示名称

权限检查

SkillTool 支持基于规则的权限匹配，规则格式支持：

  • 精确匹配：Skill(commit)
  • 前缀通配：Skill(review:*) — 匹配 review-pr、review-code 等
  • 官方插件信任：来自官方 marketplace 的技能可配置自动信任
CHAPTER 05

内置技能（Bundled Skills）

内置技能编译到 CLI 二进制中，无需磁盘文件，所有用户开箱即用。注册入口在 src/skills/bundled/index.ts。

注册模式

// src/skills/bundledSkills.ts
export function registerBundledSkill(definition: BundledSkillDefinition): void {
  // definition 包含：name, description, whenToUse,
  // allowedTools, model, getPromptForCommand, files, ...
  const command: Command = { type: 'prompt', source: 'bundled', ... }
  bundledSkills.push(command)
}

// 技能可附带引用文件（延迟提取到磁盘）
files?: Record<string, string>  // 相对路径 -> 内容

文件提取安全机制

Bundled Skills 的附带文件（files 字段）在首次调用时延迟提取到磁盘。安全措施包括：

  • 每进程随机 nonce 目录（防预创建攻击）
  • O_NOFOLLOW | O_EXCL — 不跟随符号链接，拒绝已存在文件
  • 0o700 / 0o600 权限 — 仅所有者可访问
  • 路径遍历检查 — 拒绝 .. 和绝对路径
Built-in Plugins 与 Bundled Skills 的区别：Bundled Skills 始终可用且无法禁用；Built-in Plugins 出现在 /plugin UI 中，用户可以开关（持久化到 settings）。一个 Built-in Plugin 可以包含多个 skills、hooks 和 MCP servers。
CHAPTER 06

插件系统与 Marketplace

插件系统是 Claude Code 最重量级的扩展机制，涉及约 40 个源文件（src/utils/plugins/），覆盖安装、加载、缓存、版本控制、安全策略等完整生命周期。

插件组成

一个 LoadedPlugin 可以包含以下组件：

📝 Skills

skillsPath / skillsPaths 指向包含 SKILL.md 的目录

🔌 Hooks

hooksConfig: HooksSettings — PreToolUse / PostToolUse / Stop 等事件钩子

🚀 MCP Servers

mcpServers: Record<string, McpServerConfig> — MCP 协议服务器

📖 LSP Servers

lspServers: Record<string, LspServerConfig> — Language Server Protocol 服务器

三种插件来源

来源标识符格式特点
Marketplaceplugin-name@marketplace-name通过 Git 仓库分发，支持版本控制、自动更新、策略封锁
Built-inplugin-name@builtin编译到 CLI 中，可通过 /plugin UI 开关
Session--plugin-dir 参数会话级别，仅当前运行有效

Marketplace 架构

~/.claude/plugins/
  known_marketplaces.json    # 已知 marketplace 配置
  marketplaces/              # marketplace 缓存
    my-marketplace/          # Git 克隆的 marketplace 仓库
      .claude-plugin/
        marketplace.json     # marketplace 清单
  installed/                 # 已安装插件的版本缓存

官方保留的 marketplace 名称：

claude-code-marketplace claude-code-plugins anthropic-marketplace anthropic-plugins agent-skills

加载流程

❶ loadAllPlugins()
→
❷ marketplace + session + builtin 并行加载
→
❸ mergePluginSources() 去重
→
❹ 依赖验证 + 策略过滤

两种加载变体：

  • loadAllPlugins() — 完整加载，可能触发 Git clone（用于 /plugins、显式刷新）
  • loadAllPluginsCacheOnly() — 仅缓存，从 installed_plugins.json 读取（用于启动时，不阻塞网络）

后台安装管理器

PluginInstallationManager（src/services/plugins/）在启动后异步 reconcile marketplace：

  • 发现 missing 的 marketplace → 后台 clone
  • 检测 sourceChanged → 后台更新
  • 新安装完成 → 自动 refresh 插件列表
  • 更新完成 → 设置 needsRefresh，提示用户 /reload-plugins
安全防线：插件系统有多层安全机制 — marketplace 名称反仿冒检测（正则 + Unicode 同形字检查）、策略封锁列表（pluginPolicy.ts）、非 ASCII 字符拦截、企业 managed settings 锁定。
CHAPTER 07

命令系统架构

命令系统定义在 src/commands.ts（约 750 行）和 src/types/command.ts。它是三大扩展系统的统一调度层。

Command 类型体系

// src/types/command.ts
type Command = CommandBase & (PromptCommand | LocalCommand | LocalJSXCommand)

type CommandBase = {
  name: string
  description: string
  aliases?: string[]
  availability?: ('claude-ai' | 'console')[]  // 认证要求
  isEnabled?: () => boolean                     // 条件启用
  isHidden?: boolean                            // 隐藏于 typeahead
  loadedFrom?: 'commands_DEPRECATED' | 'skills' | 'plugin' | 'managed' | 'bundled' | 'mcp'
  kind?: 'workflow'
  whenToUse?: string
  // ...
}

三种命令类型

类型执行方式用途示例
prompt 展开为 prompt 文本发送给模型 技能、自定义命令 /commit, /review, 所有 skills
local 本地 TypeScript 函数执行，返回文本结果 简单本地操作 /compact, /cost, /clear
local-jsx 渲染 Ink（React-like）UI 组件 交互式界面 /help, /config, /doctor, /plugin

命令注册与合并

getCommands(cwd) 是命令系统的核心入口，合并顺序决定优先级：

// src/commands.ts - loadAllCommands()
return [
  ...bundledSkills,        // 1. 内置技能（最高优先）
  ...builtinPluginSkills,  // 2. Built-in 插件技能
  ...skillDirCommands,     // 3. 文件技能（/skills/ 和 /commands/）
  ...workflowCommands,     // 4. Workflow 命令
  ...pluginCommands,       // 5. 插件命令
  ...pluginSkills,         // 6. 插件技能
  ...COMMANDS(),           // 7. 内置命令（最低优先）
]

命令生命周期

用户输入 /xxx
→
findCommand() 名称/别名匹配
→
类型判断
→
执行
  • prompt 型：调用 getPromptForCommand(args, context) → 内容注入对话 → 模型处理
  • local 型：load() 延迟加载模块 → call(args, context) → 返回文本
  • local-jsx 型：load() → call(onDone, context, args) → 渲染 React 组件

可用性过滤

每次 getCommands() 调用都会重新评估两层过滤：

  1. meetsAvailabilityRequirement() — 认证类型匹配（claude-ai / console）
  2. isCommandEnabled() — feature flag / 环境变量 / 平台检查
远程安全：REMOTE_SAFE_COMMANDS 白名单控制 --remote 模式下可用的命令。BRIDGE_SAFE_COMMANDS 控制从 Mobile/Web 客户端发来的命令。prompt 类型天然安全（只产生文本），local-jsx 类型全部封锁（渲染 Ink UI）。

插件如何贡献技能

插件通过 skillsPath / skillsPaths 字段指向包含 SKILL.md 的目录。加载时：

  1. getPluginSkills() 扫描已启用插件的技能目录
  2. 使用与文件技能相同的 parseFrontmatter + createSkillCommand 流程
  3. 技能名称加上插件前缀：plugin-name:skill-name
  4. 进入统一注册表，模型可通过 SkillTool 调用

MCP 如何贡献技能

MCP 服务器可以暴露 prompts，这些 prompts 可以被标记为技能（loadedFrom: 'mcp'）：

  • 通过 mcpSkillBuilders 注册表（write-once pattern）获取创建函数
  • 使用与文件技能相同的 parseSkillFrontmatterFields 解析
  • 安全限制：MCP 技能的 Markdown 中不执行内联 shell 命令（!`...`）
  • 通过 getMcpSkillCommands() 过滤，只有 loadedFrom === 'mcp' 的才算技能

用户自定义扩展指南

创建用户技能

最简单的扩展方式 — 在 ~/.claude/skills/ 下创建技能目录：

# 创建技能目录
mkdir -p ~/.claude/skills/my-skill

# 编写 SKILL.md
cat > ~/.claude/skills/my-skill/SKILL.md << 'EOF'
---
description: "我的自定义技能"
when_to_use: "当用户需要做 XX 时使用"
allowed-tools:
  - Bash
  - Read
  - Edit
---

# 技能 Prompt

你的指令内容...
EOF

创建项目技能

# 项目级技能（提交到 Git）
mkdir -p .claude/skills/deploy

cat > .claude/skills/deploy/SKILL.md << 'EOF'
---
description: "部署到生产环境"
user-invocable: true
context: fork
allowed-tools:
  - Bash(npm:*)
  - Bash(docker:*)
---
执行部署流程...
EOF

技能扩展决策树

需要扩展 Claude Code?
     │
     ├─ 只需要一个 prompt? ──────────▶ 创建 SKILL.md
     │     ├─ 全局可用? ──────────────▶ ~/.claude/skills/
     │     ├─ 项目特定? ──────────────▶ .claude/skills/
     │     └─ 按文件类型激活? ────────▶ 加 paths frontmatter
     │
     ├─ 需要 hooks + MCP + 多技能? ──▶ 创建 Plugin
     │     ├─ 团队共享? ─────────────▶ 发布到 Marketplace
     │     └─ 个人使用? ─────────────▶ 本地 --plugin-dir
     │
     └─ 需要 UI 交互? ──────────────▶ 贡献内置命令（需修改源码）
最佳实践：
1. 优先使用 context: fork 运行复杂技能，避免消耗主对话的 token 预算
2. 用 allowed-tools 精确控制技能可用的工具，遵循最小权限原则
3. 写好 when_to_use 帮助模型正确匹配技能，这比 description 更重要
4. 条件技能（paths）是零成本的 — 不匹配时不占用技能列表空间
