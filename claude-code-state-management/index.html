1架构全景：状态管理的核心拓扑

Claude Code 的状态管理并没有使用 Redux、MobX 或 Zustand 等第三方库，而是从零实现了一个仅 34 行的微型状态容器，搭配 React 18 的 useSyncExternalStore API 实现响应式绑定。整个架构可以用下面的拓扑图概括：


  ┌─────────────────────────────────────────────────────────────┐
  │                    createStore<AppState>                     │
  │  ┌────────────┐   ┌────────────┐   ┌────────────────────┐  │
  │  │  getState() │   │  setState() │   │   subscribe()      │  │
  │  │  → AppState │   │  (updater)  │   │   → unsubscribe    │  │
  │  └────────────┘   └─────┬──────┘   └────────┬───────────┘  │
  │                         │                    │              │

  │              │    next) bailout   │          │              │
  │              └──────────┬─────────┘          │              │
  │                         │                    │              │

核心设计哲学

不依赖外部状态管理库，34 行代码实现完整的 Store 原语。用 TypeScript 的 DeepImmutable 泛型在编译期保证不可变性，运行时零开销。

2状态容器实现：34 行的极简 Store

核心实现在 src/state/store.ts，仅导出两个类型和一个工厂函数：

src/state/store.ts
type Listener = () => void
type OnChange<T> = (args: { newState: T; oldState: T }) => void

export type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}

export function createStore<T>(
  initialState: T,
  onChange?: OnChange<T>,
): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()

  return {
    getState: () => state,

    setState: (updater: (prev: T) => T) => {
      const prev = state
      const next = updater(prev)
      if (Object.is(next, prev)) return  // ← bailout 优化
      state = next
      onChange?.({ newState: next, oldState: prev })
      for (const listener of listeners) listener()
    },

    subscribe: (listener: Listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}

关键设计决策

特性实现方式对比 Zustand
Bailout 机制Object.is(next, prev) 引用相等跳过相同
更新方式仅 updater function，无直接赋值Zustand 支持 partial 对象和 updater
Side Effects内置 onChange 回调在状态更新后同步执行需 subscribe 或中间件
中间件无中间件系统支持 immer / persist / devtools
Selector在 React 侧通过 useSyncExternalStore 实现内置 selector
体积~34 行，0 依赖~2KB gzipped
为什么不用 Zustand？

Claude Code 是一个 CLI 终端应用（基于 Ink/React），不需要 devtools、持久化中间件等 Web 场景特性。自研 34 行 Store 足够满足需求，还能精确控制 onChange 的执行时机——这对权限模式同步到远程 CCR 服务至关重要。

4DeepImmutable 类型系统：编译期不可变性守卫

DeepImmutable<T> 是一个递归泛型，对对象的每一层属性施加 readonly，将 Map 替换为 ReadonlyMap，将 Set 替换为 ReadonlySet，将数组替换为 ReadonlyArray。它定义在 src/types/utils.ts（构建时编译进 bundle）。

典型的 DeepImmutable 递归结构

标准递归不可变类型（推断的定义）
type DeepImmutable<T> =
  T extends Map<infer K, infer V>
    ? ReadonlyMap<DeepImmutable<K>, DeepImmutable<V>>
    : T extends Set<infer U>
      ? ReadonlySet<DeepImmutable<U>>
      : T extends Array<infer U>
        ? ReadonlyArray<DeepImmutable<U>>
        : T extends object
          ? { readonly [K in keyof T]: DeepImmutable<T[K]> }
          : T

为什么需要 & 交叉排除？

AppState 中的 tasks、mcp、replContext 等字段包含函数类型（如 abortController、vmContext、handler），对函数递归 readonly 会导致类型错误。因此 Claude Code 采用交叉类型分治策略：

// DeepImmutable 包裹纯数据字段
type AppState = DeepImmutable<{ settings: ..., verbose: ..., ... }>
  // 交叉混入含函数类型的字段
  & { tasks: { [id: string]: TaskState }, ... }

这种设计在 Tool.ts 中同样被使用：

src/Tool.ts
// Apply DeepImmutable to the imported type
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory>
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  shouldAvoidPermissionPrompts?: boolean
  // ...
}>
注意事项

DeepImmutable 是纯编译期保护——运行时 JavaScript 对象仍然可变。所有状态更新必须通过 setState(prev => ({ ...prev, field: newValue })) 的展开运算符创建新引用，这也是 Object.is bailout 的前提。

5React 绑定：选择器订阅与 useSyncExternalStore

React 侧的集成在 src/state/AppState.tsx 中完成，核心是三个 hooks：

useAppState —— 选择器订阅

src/state/AppState.tsx
/**
 * Subscribe to a slice of AppState. Only re-renders when
 * the selected value changes (compared via Object.is).
 *
 * Do NOT return new objects from the selector -- Object.is
 * will always see them as changed.
 */
export function useAppState(selector) {
  const store = useAppStore()
  const get = () => {
    const state = store.getState()
    return selector(state)
  }
  return useSyncExternalStore(store.subscribe, get, get)
}

使用方式非常直观：

// 独立订阅不同字段 — 各自独立 re-render
const verbose = useAppState(s => s.verbose)
const model = useAppState(s => s.mainLoopModel)

// 订阅子对象引用（不要创建新对象！）
const { text, promptId } = useAppState(s => s.promptSuggestion) // OK
// const bad = useAppState(s => ({ a: s.verbose }))  // 每次都 re-render!

useSetAppState —— 纯写入 hook

export function useSetAppState() {
  return useAppStore().setState  // 稳定引用，永不触发 re-render
}

useAppStateMaybeOutsideOfProvider —— 安全降级

// 在 AppStateProvider 外部调用时返回 undefined，不抛异常
export function useAppStateMaybeOutsideOfProvider(selector) {
  const store = useContext(AppStoreContext)
  return useSyncExternalStore(
    store ? store.subscribe : NOOP_SUBSCRIBE,
    () => store ? selector(store.getState()) : undefined
  )
}

useSettings —— 专用配置 hook

在 src/hooks/useSettings.ts 中进一步封装：

export type ReadonlySettings = AppState['settings'] // DeepImmutable wrapped

export function useSettings(): ReadonlySettings {
  return useAppState(s => s.settings)
}
React Compiler 优化

AppState.tsx 的编译产物使用了 react/compiler-runtime 的 _c 函数做自动 memo 缓存（$[0], $[1]... 的 slot 数组），说明 Claude Code 已经启用了 React Compiler。这意味着 useAppState 中的 selector 闭包和 get 函数都会被自动缓存，避免不必要的 re-subscribe。

6状态变更回调：onChangeAppState 副作用链

createStore 的第二个参数 onChange 接收 onChangeAppState 函数，它是所有状态变更的单一副作用管道。每次 setState 产生新状态后同步调用。

src/state/onChangeAppState.ts 副作用链
  1. 权限模式同步 — 检测 toolPermissionContext.mode 变化，通知 CCR 远程服务和 SDK status stream。先 externalize 模式名（bubble/ungated auto 等内部模式映射到 default），避免向外部暴露内部模式。
  2. 模型设置持久化 — mainLoopModel 变化时写入 userSettings，并同步 setMainLoopModelOverride。设为 null 时清除设置。
  3. 视图状态持久化 — expandedView 变化时写入 globalConfig（showExpandedTodos / showSpinnerTree），保证重启后恢复上次视图。
  4. verbose 持久化 — 同步到 globalConfig.verbose。
  5. 配置变更缓存清理 — settings 引用变化时清除 API Key Helper、AWS/GCP 凭证缓存，重新应用环境变量。
设计亮点

权限模式变更的注释特别提到，在引入 onChange 之前，有 8+ 个不同的 mutation 路径需要各自手动通知 CCR，其中只有 2 个路径实际做了通知。集中到 onChange 后，任何 setAppState 调用只要改变了 mode，都会自动同步——这是"把副作用集中化"的典型案例。

7选择器模式：纯函数状态投影

src/state/selectors.ts 定义了独立于 React 的纯选择器函数，用于非 hook 场景（工具执行、任务路由等）：

getViewedTeammateTask

export function getViewedTeammateTask(
  appState: Pick<AppState, 'viewingAgentTaskId' | 'tasks'>
): InProcessTeammateTaskState | undefined {
  const { viewingAgentTaskId, tasks } = appState
  if (!viewingAgentTaskId) return undefined
  const task = tasks[viewingAgentTaskId]
  if (!task || !isInProcessTeammateTask(task)) return undefined
  return task
}

getActiveAgentForInput —— 输入路由选择器

export type ActiveAgentForInput =
  | { type: 'leader' }
  | { type: 'viewed'; task: InProcessTeammateTaskState }
  | { type: 'named_agent'; task: LocalAgentTaskState }

export function getActiveAgentForInput(appState: AppState): ActiveAgentForInput {
  const viewedTask = getViewedTeammateTask(appState)
  if (viewedTask) return { type: 'viewed', task: viewedTask }
  // fallback: check for named agent (local_agent type)
  const { viewingAgentTaskId, tasks } = appState
  if (viewingAgentTaskId) {
    const task = tasks[viewingAgentTaskId]
    if (task?.type === 'local_agent') return { type: 'named_agent', task }
  }
  return { type: 'leader' }
}

注意选择器使用 Pick<AppState, ...> 约束入参，只依赖必要的字段。这使得选择器可以在测试中用最小对象调用，也为未来的 selector 缓存提供了精确的依赖声明。

teammateViewHelpers —— 视图切换的状态变迁

src/state/teammateViewHelpers.ts 包含 enterTeammateView、exitTeammateView、stopOrDismissAgent 三个函数，是状态更新的 "action" 层。它们通过 setAppState(prev => ({ ...prev, ... })) 模式保持不可变性：

// 进入队友视图：设 retain=true 阻止驱逐，清 evictAfter
// 切换时释放前一个队友（retain=false, messages 清空）
export function enterTeammateView(taskId, setAppState) {
  setAppState(prev => {
    let tasks = prev.tasks
    if (switching || needsRetain) {
      tasks = { ...prev.tasks }  // 浅拷贝 tasks 对象
      if (switching) tasks[prevId] = release(prevTask)
      if (needsRetain) tasks[taskId] = { ...task, retain: true }
    }
    return { ...prev, viewingAgentTaskId: taskId, tasks }
  })
}

SettingsSchema —— Zod 校验

src/utils/settings/types.ts 中定义了完整的 SettingsSchema（基于 Zod v4），包含 50+ 个可选字段。关键设计原则在注释中明确声明：

允许：添加可选字段、添加 enum 值、放宽验证。
禁止：删除字段、删除 enum 值、必选化可选字段、收紧类型。

配置热更新流水线

  1. changeDetector — chokidar 监听所有 settings 文件目录，awaitWriteFinish 去抖（1s 稳定性检查）。额外处理 delete-and-recreate 模式（grace period）。
  2. ConfigChange hooks — 变更事件先经过 hooks 系统，exit code 2 可阻止变更生效。
  3. fanOut — 重置 settings 缓存一次（避免 N 个 subscriber 各自重置造成 N 次磁盘读取），然后 emit signal。
  4. useSettingsChange — React hook 监听 signal，调用 applySettingsChange。
  5. applySettingsChange — 重新读取 settings，重新加载权限规则，更新 hooks 快照，推入 AppState。
  6. onChangeAppState — 检测 settings 引用变化，清除 auth 缓存，重新应用环境变量。

迁移设计原则

  • 幂等性：每个迁移检查当前状态，只在需要时执行。大多数通过 getSettingsForSource('userSettings') 读取未合并的配置，避免被其他层覆盖。
  • 单源修改：只修改 userSettings，不碰 project/local/policy 配置。注释明确说明原因——修改 merged settings 会导致"静默提升到全局"。
  • 一次性守卫：复杂迁移使用 globalConfig 中的布尔标志（如 hasResetAutoModeOptInForDefaultOffer）防止重复执行。
  • 用户分群：多个迁移按用户类型（ant/external）、订阅等级（Pro/Max/Team）、API Provider（firstParty/bedrock）做条件分支。
无集中版本号

与传统数据库迁移不同，Claude Code 没有全局版本号或迁移注册表。每个迁移函数独立判断是否需要执行，依赖当前配置状态而非版本戳。这种设计简化了分支合并和回滚，但也意味着迁移之间的执行顺序依赖调用点的排列。

11Provider 架构：从 Store 到 UI 的数据流

AppStateProvider 是整个应用的状态根节点，创建 Store 并通过 React Context 向下传递：

简化的 Provider 结构
const AppStoreContext = React.createContext<AppStateStore | null>(null)
const HasAppStateContext = React.createContext<boolean>(false)

function AppStateProvider({ children, initialState, onChangeAppState }) {
  // 防止嵌套
  if (useContext(HasAppStateContext)) {
    throw new Error("AppStateProvider can not be nested")
  }

  // 创建 store — 仅在初始化时
  const [store] = useState(() =>
    createStore(initialState ?? getDefaultAppState(), onChangeAppState)
  )

  // Mount 时检查权限模式
  useEffect(() => {
    if (isBypassPermissionsModeDisabled()) {
      store.setState(prev => ({
        ...prev,
        toolPermissionContext: createDisabledBypassPermissionsContext(...)
      }))
    }
  }, [])

  // 监听配置文件变更
  useSettingsChange(source => applySettingsChange(source, store.setState))

  return (
    <HasAppStateContext.Provider value={true}>
      <AppStoreContext.Provider value={store}>
        <MailboxProvider>
          <VoiceProvider>{children}</VoiceProvider>
        </MailboxProvider>
      </AppStoreContext.Provider>
    </HasAppStateContext.Provider>
  )
}

关键组合

  • HasAppStateContext：布尔 Context，防止 Provider 嵌套导致多 Store 实例。
  • MailboxProvider：Agent Swarm 的消息邮箱系统。
  • VoiceProvider：语音模式（ant-only feature），通过 bun:bundle 的 feature() 做死代码消除。
DCE (Dead Code Elimination)

Claude Code 使用 Bun 的 feature() 做条件编译。例如 feature('VOICE_MODE') 在外部构建中编译为 false，使 VoiceProvider 变成 passthrough，其完整实现被 tree-shake 移除。同样的模式用于 COMMIT_ATTRIBUTION、CONTEXT_COLLAPSE、TRANSCRIPT_CLASSIFIER 等。

12总结：设计取舍与架构评价

优势

  • 极致轻量：34 行 Store 实现，零外部依赖，启动零开销。
  • 类型安全：DeepImmutable 在编译期阻止意外修改，交叉类型优雅处理函数字段。
  • 副作用集中化：onChangeAppState 作为单一 choke point，消除了散落的手动同步。
  • 配置热更新：chokidar 监听 + signal 通知 + 缓存重置一体化，文件变更到 UI 刷新的延迟约 1.5s。
  • 向后兼容：Zod 校验 + 保留无效字段 + 无版本号迁移，对用户极其友好。

值得关注的取舍

  • 状态树扁平化：80+ 个顶层字段（特别是 15+ 个 replBridge*），缺少领域分组的中间层。但对于 CLI 应用而言，这种简单性是合理的——避免了深层 selector 的复杂性。
  • 无 selector 缓存：依赖 useSyncExternalStore 的 Object.is 比较，如果 selector 返回新对象会导致无效 re-render。但注释已明确警告，且 React Compiler 帮助缓解了这个问题。
  • 迁移无编排：每个迁移自行判断是否执行，没有中央迁移注册表。对于 CLI 产品（而非数据库）这是合理的简化。
  • sessionStorage 单文件：5000+ 行的巨型模块，集中了所有 JSONL 读写逻辑。可能是技术债，但也避免了模块间的循环依赖。
