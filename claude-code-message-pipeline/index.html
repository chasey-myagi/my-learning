01

全局架构总览

Claude Code 的消息管道是一条从用户按下回车到 API 响应渲染回终端的完整数据通路。整个管道可以抽象为以下六大阶段：

用户输入
→
QueryEngine
→
normalizeMessagesForAPI
→
queryModel (claude.ts)
→
withRetry + Streaming
→
AssistantMessage
核心文件架构
入口层 QueryEngine.ts → query.ts (queryLoop)
规范化层 utils/messages.ts → normalizeMessagesForAPI()
API 层 services/api/claude.ts → queryModel() / queryModelWithStreaming()
客户端层 services/api/client.ts → getAnthropicClient() (@anthropic-ai/sdk)
可靠性层 services/api/withRetry.ts → withRetry() / CannotRetryError
成本追踪 cost-tracker.ts + utils/modelCost.ts + utils/tokens.ts

整条链路中，query.ts 中的 queryLoop 是外层 while(true) 循环，负责工具调用的多轮迭代；而 claude.ts 中的 queryModel 则负责单次 API 请求的构建、发送与流式解析。两者通过 AsyncGenerator 协议连接，形成了可中断、可恢复的消息流。

02

消息类型体系

Claude Code 内部定义了丰富的消息类型层次，远超 Anthropic API 的简单 user/assistant 二元结构。这些类型全部定义在 types/message.ts 中（编译产物）。

核心消息类型

类型说明API 映射
UserMessage用户输入、工具执行结果（tool_result）role: "user"
AssistantMessage模型响应，含 text、tool_use、thinking 等 content blocksrole: "assistant"
SystemMessage系统信息：compact boundary、api_error、local_command、permission retry 等多种 subtype被 normalizeMessagesForAPI 过滤
AttachmentMessageCLAUDE.md、memory 文件、MCP 指令等上下文注入重排序后合并到 user message
ProgressMessage工具执行进度（UI 专用）被过滤，不发送到 API
TombstoneMessage标记被删除的消息（undo 操作）被过滤

AssistantMessage 的丰富字段

从源码 baseCreateAssistantMessage 函数可以看到，一条 AssistantMessage 不只是 API 的 BetaMessage，还携带了大量元数据：

{
  type: 'assistant',
  uuid: randomUUID(),           // 内部唯一标识
  timestamp: ISO8601,
  requestId: string,            // API 请求 ID，用于链式追踪
  message: {                    // Anthropic SDK 的 BetaMessage
    id, model, role, stop_reason,
    usage: { input_tokens, output_tokens,
             cache_creation_input_tokens,
             cache_read_input_tokens,
             server_tool_use, service_tier,
             cache_creation: { ephemeral_1h, ephemeral_5m },
             inference_geo, iterations, speed },
    content: BetaContentBlock[]
  },
  isApiErrorMessage?: boolean,  // 是否为合成的错误消息
  apiError?: string,            // 错误分类标记
  errorDetails?: string,        // 原始 API 错误文本
  isVirtual?: true,             // 仅 UI 展示，不发送到 API
  costUSD?: number              // 该次请求的美元成本
}

合成消息 (Synthetic Messages)

系统使用 SYNTHETIC_MODEL = '<synthetic>' 标记非 API 返回的消息，包括：

  • INTERRUPT_MESSAGE -- 用户中断请求
  • CANCEL_MESSAGE -- 用户取消工具操作
  • REJECT_MESSAGE -- 用户拒绝工具执行
  • NO_RESPONSE_REQUESTED -- 不需要 API 响应

这些合成消息通过 isSyntheticMessage() 和 isSyntheticApiErrorMessage() 识别，在 token 统计时被排除。

SDK 消息映射

utils/messages/mappers.ts 负责内部消息与 SDK 消息格式之间的双向转换：

  • toSDKMessages() -- 内部 Message[] 转 SDKMessage[]（发送给外部 SDK 消费者）
  • toInternalMessages() -- SDK 消息转内部格式（接收外部消息）
  • 转换过程中处理 compact_boundary 消息的元数据映射（camelCase <=> snake_case）

消息规范化流水线

normalizeMessagesForAPI() 是整个管道中最复杂的函数之一（位于 utils/messages.ts），负责将内部的混合消息数组转换为 API 可接受的 [UserMessage | AssistantMessage][] 格式。

规范化步骤

reorderAttachments
→
filterVirtual
→
stripErrorMedia
→
mergeMessages
→
normalizeToolInput
→
ensureToolResultPairing
  1. Attachment 重排序 -- reorderAttachmentsForAPI() 将 attachment 消息向上冒泡，直到遇到 tool_result 或 assistant 消息为止，确保上下文注入不打断工具调用链。
  2. 过滤虚拟消息 -- 移除 isVirtual: true 的消息（仅 UI 展示用途）。
  3. 移除错误引发的媒体 -- 如果前一次 API 调用因 PDF 太大或图片超限报错，从对应的 user message 中剥离相关 block。
  4. 消息合并 -- 相邻同角色消息合并（mergeUserMessages, mergeAssistantMessages），确保 user/assistant 严格交替。
  5. 工具输入规范化 -- normalizeToolInputForAPI() 确保 tool_use 的 input 是合法 JSON 对象（处理 streaming 拼接的字符串残留）。
  6. Tool Result 配对修复 -- ensureToolResultPairing() 为孤立的 tool_use 补充合成的 error tool_result，剥离孤立的 tool_result。

post-normalization 处理

在 claude.ts 中，规范化后的消息还会经历：

  • Tool Search 字段清理 -- 当模型不支持 tool search 时，剥离 tool_reference 块和 caller 字段
  • Advisor 块清理 -- 无 advisor beta header 时剥离 advisor 相关 blocks
  • 媒体数量限制 -- stripExcessMediaItems() 确保不超过 100 个媒体项（API_MAX_MEDIA_PER_REQUEST），超限时从最旧的开始移除
  • 签名块清理 -- stripSignatureBlocks() 移除 thinking 块中的签名
设计洞察

normalizeMessagesForAPI 被 ~20 个调用方使用（analytics、feedback、sharing 等），因此有意不接受 model 参数，保持接口简洁。模型相关的后处理（如 tool_reference 剥离）在调用方侧完成。

Beta Headers 动态管理

Claude Code 使用 "粘性锁存 (sticky-on latch)" 策略管理动态 beta headers，避免会话中途切换导致 prompt cache 失效：

  • afkHeaderLatched -- AFK 模式 beta，首次激活后保持整个会话
  • fastModeHeaderLatched -- 快速模式 beta，同上
  • cacheEditingHeaderLatched -- 缓存编辑 beta，同上
  • thinkingClearLatched -- 思维清除，1h 缓存 TTL 过期后触发
关键设计

这些 latch 在 /clear 和 /compact 时被 clearBetaHeaderLatches() 重置。切换不当会浪费 50-70K tokens 的 prompt cache。

05

Prompt Caching 策略

Prompt Caching 是 Claude Code 成本控制的核心机制。系统在多个层面精心设计了缓存策略，确保多轮对话中的重复内容不被重复计费。

Cache Control 注入

getCacheControl() 函数根据上下文动态决定缓存策略：

function getCacheControl({ scope, querySource }) {
  return {
    type: 'ephemeral',
    ...(should1hCacheTTL(querySource) && { ttl: '1h' }),
    ...(scope === 'global' && { scope }),  // 全局缓存范围
  }
}

缓存断点位置 (addCacheBreakpoints)

系统在消息数组中策略性地放置 cache_control 标记：

  • User Message 尾部 -- 每个 user message 的最后一个 content block 添加 cache_control
  • Assistant Message 尾部 -- 类似，但排除 thinking 和 redacted_thinking blocks
  • System Prompt -- 构建 system prompt blocks 时添加
  • Tool Schemas -- 当使用全局缓存范围时，在工具定义上添加 scope: 'global'

1 小时 TTL 策略

should1hCacheTTL() 决定是否使用 1 小时 TTL（默认 5 分钟），条件：

  1. 用户类型是 ant（内部员工）或 Claude AI 订阅用户且未使用超量
  2. 查询来源匹配 GrowthBook 配置的 allowlist（如 repl_main_thread*, sdk）
  3. Bedrock 3P 用户可通过 ENABLE_PROMPT_CACHING_1H_BEDROCK 环境变量启用
⚠ 会话稳定性保护

缓存资格在会话启动时 锁存，防止中途 GrowthBook 更新或超量状态变化导致 TTL 翻转——每次翻转浪费 ~20K tokens。

  • setPromptCache1hEligible() -- 锁存用户资格
  • setPromptCache1hAllowlist() -- 锁存 allowlist 配置

Prompt Cache Break 检测

promptCacheBreakDetection.ts 实现了详尽的缓存击穿检测——记录每次请求的 system prompt hash、tool schemas hash、model、betas 等状态，在下次请求时对比变化。它甚至能精确到具体是哪个 tool 的 schema 发生了变化（77% 的 tool break 由此引起）。

Cached Microcompact (Cache Editing)

进阶特性：通过 API 的 cache_edits 机制，在不破坏缓存的前提下对已缓存的消息做增量修改（如删除旧工具结果以压缩上下文），避免完整的 compact 操作导致缓存全部失效。

流式响应处理

Claude Code 使用 Anthropic SDK 的原始流（Raw Stream）而非高级 BetaMessageStream 封装，以避免 O(n^2) 的 partial JSON 解析开销。整个流处理引擎在 claude.ts 的 queryModel 函数内实现。

SSE 事件类型与处理

事件类型处理逻辑关键操作
message_start初始化 partialMessage记录 TTFB 时间、初始化 usage、提取 research 元数据
content_block_start在 contentBlocks[index] 中创建新块根据类型（text/tool_use/thinking/server_tool_use）初始化不同结构
content_block_delta增量拼接内容text_delta 拼接 text，input_json_delta 拼接 JSON 字符串，thinking_delta 拼接思维链
content_block_stop完成单个 content block创建 AssistantMessage 并 yield，解析 tool_use 的 JSON input
message_delta更新 stop_reason 和 usage累加 output_tokens、记录 response headers
message_stop标记流结束触发 cost 计算、telemetry 上报

content_block_stop 的核心处理

case 'content_block_stop': {
  const m: AssistantMessage = {
    message: {
      ...partialMessage,
      content: normalizeContentFromAPI(
        [contentBlock],   // 单块规范化
        tools,
        options.agentId,
      ),
    },
    type: 'assistant',
    uuid: randomUUID(),
    timestamp: new Date().toISOString(),
    costUSD,
  }
  newMessages.push(m)
  yield m                   // 实时 yield 给调用方
}

注意每个 content block 完成时就立即 yield 一个 AssistantMessage，而不是等到整个消息完成。这使得 UI 可以即时展示工具调用结果。

流式超时与 Watchdog

系统实现了双层流超时保护：

  • Idle Watchdog -- 可配置的空闲超时（默认 90s），如果没有任何 chunk 到达就强制终止流。通过 CLAUDE_STREAM_IDLE_TIMEOUT_MS 和 CLAUDE_ENABLE_STREAM_WATCHDOG 控制。
  • Stall Detection -- 每个 chunk 到达时检查距上个 chunk 的间隔，超过 30s 记录为 stall（但不终止）。

非流式 Fallback

当流式请求遇到 529 (Overloaded) 错误时，系统会降级为非流式请求（executeNonStreamingRequest），超时设为 300s（远程会话 120s）。非流式模式限制 max_tokens 以控制延迟。

Claude Code 实现了从 per-request 到 per-session 的完整成本追踪体系，精确到每个模型和每种 token 类型。

累积追踪链路

addToTotalSessionCost() 是成本追踪的核心函数，每次 API 响应完成后调用：

  1. 按模型累加 token 统计（addToTotalModelUsage）

  • getTokenCountFromUsage(usage) -- input + cache_creation + cache_read + output
  • tokenCountFromLastAPIResponse(messages) -- 从最近一条 API 响应获取 context 大小
  • finalContextTokensFromLastResponse(messages) -- 使用 usage.iterations 的最后一条（用于 task_budget.remaining 计算）

重试与错误处理

services/api/withRetry.ts 实现了一个复杂的重试引擎，它不只是简单的指数退避，而是一个考虑了 fast mode fallback、429/529 差异化处理、持久模式无限重试等多维度的决策系统。

核心类型

export async function* withRetry<T>(
  getClient: () => Promise<Anthropic>,
  operation: (client, attempt, context: RetryContext) => Promise<T>,
  options: RetryOptions,
): AsyncGenerator<SystemAPIErrorMessage, T>

关键：withRetry 是一个 AsyncGenerator，在等待重试期间 yield SystemAPIErrorMessage 给 UI 层展示。

重试决策矩阵

错误类型HTTP Status处理策略
Rate Limit429订阅用户不重试；企业/PAYG 用户重试；Fast Mode 触发 cooldown
Overloaded529前台查询最多 3 次后 fallback 到 fallback model；背景查询立即放弃
Auth Error401清除 API key 缓存、刷新 OAuth token 后重试
Token Revoked403OAuth token 吊销，强制刷新后重试
Context Overflow400解析错误信息中的 token 数，动态减小 max_tokens 后重试
Timeout408正常重试
Connection ResetECONNRESET禁用 Keep-Alive、重建连接后重试
Server Error500+正常重试

退避策略

// 指数退避 + 抖动
baseDelay = min(BASE_DELAY_MS * 2^(attempt-1), maxDelayMs)
jitter    = random() * 0.25 * baseDelay
delayMs   = baseDelay + jitter

// 常量
BASE_DELAY_MS = 500
DEFAULT_MAX_RETRIES = 10
MAX_529_RETRIES = 3

Fast Mode Fallback

当 fast mode 请求遇到 429/529 时，系统根据 retry-after 头的长短做分叉决策：

  • 短延迟 (<20s) -- 保持 fast mode 等待重试（保护 prompt cache，同一模型名）
  • 长延迟 (>20s 或未知) -- 进入 cooldown 切换到标准速度（最少 10 分钟）
  • 超量不可用 -- 检查 anthropic-ratelimit-unified-overage-disabled-reason header，永久禁用 fast mode

529 Fallback Model

连续 3 次 529 错误后，非订阅用户触发 FallbackTriggeredError，外层 query loop 捕获后切换到 fallback model（通常从 Opus 降级到 Sonnet）。

持久重试模式

通过 CLAUDE_CODE_UNATTENDED_RETRY 环境变量启用（内部功能），针对无人值守的远程会话：

  • 429/529 无限重试，退避上限 5 分钟
  • 尊重 anthropic-ratelimit-unified-reset header 等待到重置时间
  • 长等待时每 30 秒 yield 一个心跳消息，防止宿主环境标记会话为空闲
  • 重置上限 6 小时

前台 vs 后台查询差异

通过 FOREGROUND_529_RETRY_SOURCES 集合区分——只有用户正在等待的查询（repl_main_thread, agent, compact, side_question, auto_mode 等）才重试 529。后台任务（summary, title, suggestion, classifier）立即放弃，避免容量雪崩时的 3-10x 重试放大。

09

消息持久化与恢复

Claude Code 的消息不是一次性的——它们被持久化到磁盘，支持 /resume 恢复会话、/compact 压缩上下文、以及跨设备的远程会话同步。

持久化存储

utils/sessionStorage.ts 负责会话的序列化与反序列化。每条消息以 SerializedMessage 格式存储，保留完整的内部类型信息。

会话恢复流程 (conversationRecovery.ts)

  1. 加载会话日志文件（loadTranscriptFile / loadFullLog）
  2. 构建会话链（buildConversationChain），处理多分支和 compact 边界
  3. 一致性校验（checkResumeConsistency）
  4. 迁移旧版 attachment 类型（migrateLegacyAttachmentTypes）
  5. 过滤孤立的 thinking-only 消息（filterOrphanedThinkingOnlyMessages）
  6. 过滤未解决的 tool_use（filterUnresolvedToolUses）
  7. 恢复 file history 快照和 plan 状态

Compact 机制

services/compact/ 目录实现了多层压缩策略：

  • Auto Compact -- 当 token 数接近 context window 时自动触发（阈值 = 有效窗口 - 13K buffer）
  • Reactive Compact -- prompt_too_long 错误后的响应式压缩
  • Session Memory Compact -- 提取会话记忆到独立存储
  • Snip Compact -- 历史记录裁剪
  • Cached Microcompact -- 不破坏 prompt cache 的增量修改

Compact 后在消息数组中插入 SystemCompactBoundaryMessage，getMessagesAfterCompactBoundary() 确保只有边界之后的消息被发送到 API。

Tool Result 存储优化

utils/toolResultStorage.ts 实现了 per-message 的工具结果大小预算（applyToolResultBudget）。超大的工具输出会被替换为压缩摘要，替换记录被持久化以支持恢复。

SDK 消息同步

对于远程会话（Claude Desktop、mobile apps），utils/messages/mappers.ts 负责内部消息与 SDK 格式的双向转换。特别处理：

  • local_command 输出转换为合成的 assistant message（Android 和 session-ingress 不支持 local_command_output 子类型）
  • Plan 内容注入到 ExitPlanModeV2 的 tool_input 中（V2 工具从文件读取 plan，但 SDK 消费者期望 tool_input.plan 存在）
  • Rate limit 信息映射（toSDKRateLimitInfo）

设计亮点与工程启示

从 Claude Code 的消息管道中可以提炼出多项值得借鉴的工程实践。

1. AsyncGenerator 作为贯穿协议

从 queryLoop 到 queryModel 再到 withRetry，整条链路使用 AsyncGenerator 连接。这不是偶然选择——AsyncGenerator 天然支持：

  • 流式数据传递（yield 事件逐个送达）
  • 背压控制（消费者暂停时生产者自动阻塞）
  • 错误冒泡（throw 会穿透 yield* 链）
  • 取消传播（.return() 沿链路关闭所有 generator）

2. 会话级锁存防止缓存抖动

多处使用 "一旦激活就不关闭" 的 latch 策略（beta headers、1h TTL 资格、allowlist）。这种 "宁可多发一个 header 也不翻转" 的思路，源于一个量化指标：每次翻转浪费 20-70K tokens 的 prompt cache。

3. 分层错误处理

🛠 三层错误处理
  • withRetry 层 -- 网络级重试、auth 刷新、rate limit 退避
  • queryModel 层 -- 流式错误降级、context overflow 调整、fallback model
  • query loop 层 -- tool 执行错误处理、auto compact、reactive compact

4. 前台/后台查询差异化

通过 QuerySource 标记每个查询的来源，在重试策略、缓存策略、telemetry 粒度上做精细区分。背景查询的重试被严格限制，避免容量雪崩时的放大效应。

5. 防御性消息修复

normalizeMessagesForAPI 不信任输入数据的完整性。它会：

  • 修复 tool_use/tool_result 的配对缺失（生产事故后的安全网）
  • 去重同 ID 的 tool_use blocks（CC-1212 bug 的修复）
  • 为空白 assistant message 添加占位符（防止 API 收到以 assistant 开头的消息序列）
  • 跨消息的 tool_use ID 去重（orphan handler 重推导致的重复）

6. 指纹追踪与归因

computeFingerprintFromMessages() 从首个 user message 计算指纹，注入到 system prompt 的 attribution header 中。每条 assistant message 的 requestId 形成链式追踪，支持跨请求的分析。

7. 成本意识的工程文化

从代码注释中可以看到量化思维无处不在：

  • "每次 TTL 翻转浪费 ~20K tokens"
  • "BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures"
  • "77% of tool breaks" 由 schema 变化引起
  • "3-10x gateway amplification" 量化了背景重试的风险

这些数据直接驱动了工程决策，而不是凭直觉。
