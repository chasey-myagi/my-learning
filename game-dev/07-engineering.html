01 / 项目结构与文件组织

良好的项目结构是工程效率的基础。一个组织清晰的项目能让新成员快速上手、减少合并冲突、加速构建流程。混乱的文件组织则会随着项目规模增长而不断拖慢开发节奏。

通用目录结构原则

无论使用什么引擎，游戏项目的目录结构都应遵循几个核心原则：

  • 按功能/模块分类，而不是按文件类型分类
  • 分离源码与资源，构建时明确哪些需要编译、哪些需要打包
  • 配置与数据外置，不硬编码在代码中
  • 第三方依赖隔离，明确哪些是外部代码
  • 文档就近放置，模块的文档跟着模块走

资源命名规范

一致的命名规范可以大幅减少团队协作中的混乱。以下是业界常用的命名约定：

通用命名规则

  • 小写 + 下划线（snake_case）：player_idle_01.png，这是最安全的跨平台方案
  • 前缀标识类型：T_ 贴图，M_ 材质，SM_ 静态模型，SK_ 骨骼模型
  • 后缀标识变体：_diffuse，_normal，_01，_lod2
  • 禁止空格和特殊字符：会在命令行和脚本中造成问题
  • 禁止中文路径：很多工具链不兼容

02 / 版本控制

版本控制是现代游戏开发的基石。即使是独立开发者，也必须使用版本控制 —— 它不仅能防止灾难性的数据丢失，还是理解代码演变、回退错误改动、多设备协作的基础设施。

Git 在游戏开发中的使用

Git LFS 处理大文件

Git LFS（Large File Storage）是游戏项目使用 Git 的必备扩展。它将大文件的实际内容存储在专用服务器上，Git 仓库中只保存指针文件。

Git LFS 常用操作
# 安装 Git LFS
$ git lfs install

# 追踪特定文件类型
$ git lfs track "*.png"
$ git lfs track "*.fbx"
$ git lfs track "*.wav"

# 查看 LFS 追踪的文件
$ git lfs ls-files

# 查看 LFS 存储状态
$ git lfs status

# 迁移已有仓库的大文件到 LFS
$ git lfs migrate import --include="*.png,*.fbx,*.wav"

# 只拉取需要的 LFS 文件（节省带宽）
$ git lfs fetch --include="assets/textures/**"

# 清理旧的 LFS 缓存
$ git lfs prune
注意
Git LFS 有存储和带宽限制（GitHub 免费版每月 1GB 带宽、1GB 存储）。大型项目建议使用自托管的 LFS 服务器或 GitLab。考虑使用 git lfs fetch --recent 只拉取近期文件。

分支策略

游戏项目的分支策略取决于团队规模和发布节奏。两种主流策略的对比：

Git Flow

适合有明确发布周期的项目。

  • main - 发布版本
  • develop - 开发主线
  • feature/* - 功能分支
  • release/* - 发布准备
  • hotfix/* - 紧急修复

优点：版本清晰、适合同时维护多个版本
缺点：分支多、合并复杂、流程重

Trunk-Based Development

适合持续交付和小团队。

  • main - 唯一主线
  • 短生命周期的功能分支
  • Feature Flag 控制功能开关
  • 频繁合并，保持主线可用

优点：简单、减少合并冲突、快速反馈
缺点：需要更好的测试覆盖、CI/CD 支持

独立开发者建议
独立开发者或 2-3 人小团队推荐 Trunk-Based。保持 main 分支始终可运行，需要实验性功能时开短期分支，完成后立即合并。不要过度设计分支策略。

美术资源的版本控制

美术资源是游戏版本控制中最棘手的部分。核心挑战：

  • 文件体积大：一个 PSD 源文件可能几百 MB
  • 二进制格式：无法 diff，无法 merge
  • 更新频繁：美术迭代快，历史版本占用大量空间
  • 源文件 vs 导出文件：需要决定哪些入库

推荐方案

  1. 导出文件入 Git LFS：.png, .fbx, .wav 等导出结果，通过 LFS 管理
  2. 源文件独立存储：.psd, .blend, .max 等源文件放在 NAS / 云盘 / 专用 Perforce 服务器
  3. 建立资产管线：明确从源文件到游戏引擎的导出流程，最好自动化
  4. 文件锁定：使用 git lfs lock 防止多人同时修改同一个二进制文件
文件锁定操作
# 锁定文件，防止他人修改
$ git lfs lock "assets/textures/T_player_body_diffuse.png"

# 查看当前锁定的文件
$ git lfs locks

# 解锁文件
$ git lfs unlock "assets/textures/T_player_body_diffuse.png"

03 / 构建系统与 CI/CD

持续集成/持续部署（CI/CD）在游戏开发中被低估了。自动化构建不仅能尽早发现问题，还能让团队始终有一个可运行的最新版本用于测试。

自动化构建的价值

  • 早发现问题：每次提交自动编译，编译错误立即暴露
  • 随时可测试：QA 团队和策划可以随时获取最新版本
  • 一致的构建环境：消除"在我机器上能跑"的问题
  • 多平台同时构建：一次提交，PC / 移动 / Web 同步出包
  • 自动化测试门禁：测试不通过就不能合并
  • 发布流程标准化：一键出正式包，杜绝手工操作失误

版本号管理策略

推荐使用语义化版本号（Semantic Versioning）：MAJOR.MINOR.PATCH

版本号约定

  • 0.x.x — 开发阶段，API 可能频繁变化
  • 1.0.0 — 首次正式发布
  • MAJOR — 重大更新、DLC、资料片
  • MINOR — 新功能、新内容
  • PATCH — Bug 修复、小调整

额外信息：

  • Build Number：自动递增，标识具体构建，如 1.2.3+456
  • Git Hash：在构建信息中嵌入 commit hash，方便追溯
  • 渠道标识：1.2.3-beta，1.2.3-rc.1
自动版本号注入脚本
#!/bin/bash
# version.sh - 从 Git tag 生成版本信息

VERSION=$(git describe --tags --always --dirty)
COMMIT=$(git rev-parse --short HEAD)
BUILD_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
BUILD_NUMBER=${CI_PIPELINE_ID:-"local"}

# 写入版本文件
cat > src/version_info.h << EOF
#pragma once
#define GAME_VERSION     "${VERSION}"
#define GAME_COMMIT      "${COMMIT}"
#define GAME_BUILD_DATE  "${BUILD_DATE}"
#define GAME_BUILD_NUM   "${BUILD_NUMBER}"
EOF

echo "Version: ${VERSION} (${COMMIT}) Build: ${BUILD_NUMBER}"

构建产物管理

构建产物的管理策略直接影响团队的工作效率：

  • Artifact 保留策略：开发版保留 7-14 天，里程碑版本永久保留
  • 产物命名规范：game-name_v1.2.3_windows_x64_build456.zip
  • 分发渠道：内部用 S3/Minio/共享盘，外部用 itch.io / Steam Upload
  • 符号文件归档：保留 .pdb / .dSYM 文件用于 crash dump 分析

04 / 测试策略

游戏测试比传统软件测试更具挑战性：输出不确定（随机性）、依赖实时交互（帧率、输入时序）、"正确"的定义模糊（手感、体验）。但这并不意味着游戏不能测试 —— 而是需要更聪明的测试策略。

游戏测试的特殊挑战

  • 不确定性：随机数、物理模拟、AI 行为导致每次运行结果不同
  • 实时性：60fps 意味着 16.67ms 的帧时间窗口，时序敏感
  • 主观性：手感好不好、画面美不美、难度是否合适无法自动化判断
  • 组合爆炸：装备 x 技能 x 地形 x 敌人的组合难以穷举
  • 状态空间巨大：玩家可以在任何时候做任何事
  • 视觉回归：渲染改动的正确性验证需要视觉比对

CPU 性能分析

CPU 侧的常见瓶颈与对策：

常见 CPU 瓶颈

  • 大量 GC 分配：每帧创建临时对象，触发垃圾回收 → 用对象池、struct 代替 class
  • 复杂的 AI 计算：每帧跑寻路和决策树 → 分帧计算、LOD AI
  • 物理模拟：过多碰撞体 → 简化碰撞形状、分组过滤
  • 字符串操作：频繁拼接字符串 → StringBuilder、缓存
  • 序列化：每帧序列化大数据 → 增量更新、异步处理
  • 不合理的更新频率：不需要每帧更新的逻辑放在 Update → 降低频率或事件驱动
分帧处理示例（避免单帧卡顿）
// 将大量工作分散到多帧执行
public class FrameDistributor
{
    private Queue<Action> _workQueue = new();
    private const float MAX_MS_PER_FRAME = 2.0f;

    public void EnqueueWork(IEnumerable<Action> tasks)
    {
        foreach (var task in tasks)
            _workQueue.Enqueue(task);
    }

    public void ProcessFrame()
    {
        var sw = Stopwatch.StartNew();

        while (_workQueue.Count > 0
            && sw.Elapsed.TotalMilliseconds < MAX_MS_PER_FRAME)
        {
            var task = _workQueue.Dequeue();
            task.Invoke();
        }
    }
}

GPU 性能分析

GPU 瓶颈通常表现为：CPU 空闲但帧率上不去。关键指标：

  • Draw Call 数量：每个 Draw Call 都有 CPU 开销，目标控制在 1000 以内（移动端 200-300）
  • Overdraw：同一像素被重复绘制 → 合理排序、遮挡剔除
  • Shader 复杂度：复杂的片段着色器 → LOD 材质、简化远处效果
  • 纹理带宽：大量高分辨率纹理 → Mipmap、纹理压缩、纹理集
  • 后处理：全屏后处理效果消耗大 → 合并 Pass、降低分辨率

GPU 优化清单

  1. 开启静态/动态合批（Batching）减少 Draw Call
  2. 使用 GPU Instancing 渲染大量相同物体
  3. 实现视锥体剔除和遮挡剔除（Frustum/Occlusion Culling）
  4. 使用 LOD（Level of Detail）减少远处物体面数
  5. 纹理使用 ASTC（移动）/ BC7（PC）压缩格式
  6. Atlas 合并小纹理，减少纹理切换
  7. 后处理合并为更少的 Pass
  8. 阴影使用级联阴影贴图（Cascaded Shadow Maps）

内存分析

内存问题往往隐蔽但致命。关键关注点：

  • 峰值内存：在内存最紧张的场景（通常是内容最丰富的关卡）测量
  • 泄漏检测：反复进出场景，观察内存是否持续上涨
  • 大头分析：排序 top N 的内存消耗者，通常纹理占 60-80%
  • 碎片化：长时间运行后分配/释放效率下降

常见性能陷阱

  1. 在循环中查找节点：find_node() / GetComponent() 每帧调用 → 缓存引用
  2. 过度使用 Raycast：大量射线检测 → 合理设置频率和距离
  3. 透明物体排序：大量半透明粒子 → 减少粒子数量、使用不透明替代
  4. 实时光源过多：多个动态光源 → 烘焙静态光照、限制动态光源数量
  5. 高面数模型无 LOD：远处物体和近处一样多的三角面 → 设置 LOD Group
  6. 不必要的物理计算：屏幕外物体仍在模拟 → 睡眠/禁用远处物理体
  7. Log 输出：发布版本仍有大量 Debug.Log → 条件编译或 log level 控制
  8. 每帧创建 Lambda/闭包：产生 GC 压力 → 提升到类级别

优化优先级决策

不是所有性能问题都值得花同样的精力。决策框架：

  1. 先确认目标：目标平台的目标帧率（60fps? 30fps?），目标分辨率
  2. 量化瓶颈：Profiler 数据驱动，找到耗时 top 5
  3. 评估投入产出：某优化能省 5ms 但需要重写整个系统 → 先做能快速生效的
  4. 平台优先级：如果移动端是主要平台，先优化移动端
  5. 用户感知优先：掉帧/卡顿 > 加载慢 > 内存占用高

资源加载与卸载策略

预加载（Pre-loading）

在场景转换的 Loading Screen 期间一次性加载所有需要的资源。适用于关卡制游戏。

懒加载（Lazy Loading）

资源在首次使用时才加载。适用于 UI 资源、不确定是否会用到的内容。

引用计数卸载

追踪每个资源被多少对象引用，引用归零时卸载。防止正在使用的资源被意外释放。

资源管理器实现
class ResourceManager:
    var _cache: Dictionary = {}    # path -> { resource, ref_count }
    var _loading: Dictionary = {}  # path -> Array[callback]

    ## 同步加载（小资源）
    func load_sync(path: String) -> Resource:
        if _cache.has(path):
            _cache[path]["ref_count"] += 1
            return _cache[path]["resource"]

        var res = load(path)
        if res:
            _cache[path] = {
                "resource": res,
                "ref_count": 1,
                "size_bytes": _estimate_size(res)
            }
        return res

    ## 异步加载（大资源）
    func load_async(path: String, callback: Callable) -> void:
        if _cache.has(path):
            _cache[path]["ref_count"] += 1
            callback.call(_cache[path]["resource"])
            return

        if _loading.has(path):
            _loading[path].append(callback)
            return

        _loading[path] = [callback]
        ResourceLoader.load_threaded_request(path)
        # 在 _process 中检查加载状态

    ## 释放引用
    func release(path: String) -> void:
        if not _cache.has(path):
            return
        _cache[path]["ref_count"] -= 1
        if _cache[path]["ref_count"] <= 0:
            _cache.erase(path)  # 引用归零，释放

    ## 强制清理所有无引用资源
    func purge_unused() -> int:
        var freed = 0
        var to_remove: Array = []
        for path in _cache:
            if _cache[path]["ref_count"] <= 0:
                freed += _cache[path]["size_bytes"]
                to_remove.append(path)
        for path in to_remove:
            _cache.erase(path)
        return freed

流式加载（Streaming）

大型开放世界游戏无法一次加载所有内容。流式加载的核心思想是只加载玩家附近的内容。

流式加载策略

  • 基于距离：以玩家为中心，加载 N 格范围内的区块（Chunk），卸载远处区块
  • 基于视线：加载玩家朝向的区域，背后的可以延迟加载
  • 分层加载：地形先加载低精度，靠近后替换为高精度
  • 优先级队列：玩家移动方向的区块优先加载
区块流式加载示例
class WorldStreamer:
    const CHUNK_SIZE = 64.0       # 每个区块 64x64 单位
    const LOAD_RADIUS = 3        # 加载周围 3x3 区块
    const UNLOAD_RADIUS = 5      # 超出 5 格卸载

    var loaded_chunks: Dictionary = {}  # Vector2i -> ChunkData
    var player_chunk: Vector2i

    func update(player_pos: Vector3) -> void:
        var new_chunk = Vector2i(
            floori(player_pos.x / CHUNK_SIZE),
            floori(player_pos.z / CHUNK_SIZE)
        )

        if new_chunk == player_chunk:
            return

        player_chunk = new_chunk

        # 加载需要的区块
        for x in range(-LOAD_RADIUS, LOAD_RADIUS + 1):
            for z in range(-LOAD_RADIUS, LOAD_RADIUS + 1):
                var coord = player_chunk + Vector2i(x, z)
                if not loaded_chunks.has(coord):
                    _load_chunk_async(coord)

        # 卸载远处的区块
        var to_unload: Array = []
        for coord in loaded_chunks:
            var dist = (coord - player_chunk).length()
            if dist > UNLOAD_RADIUS:
                to_unload.append(coord)

        for coord in to_unload:
            _unload_chunk(coord)

内存碎片化

频繁的 malloc/free 会导致内存碎片化 —— 总空闲内存够用，但找不到连续的大块空间。在 C/C++ 游戏引擎中尤为突出。

减少碎片化的策略

  • 固定大小分配器（Pool Allocator）：相同大小的对象从同一个池分配
  • 栈分配器（Stack Allocator）：LIFO 顺序，适合每帧临时数据
  • 双端栈分配器（Double-Ended Stack）：一端放持久数据，一端放临时数据
  • 预分配大块内存：启动时分配好，运行时内部管理
  • 避免运行时 new/delete：游戏循环中尽量零分配

对象池实现

对象池是游戏开发中最常用的内存优化模式。核心思想：预先创建一批对象，用完不销毁而是回收，需要时取出复用。

通用对象池（C#）
public class ObjectPool<T> where T : class, IPoolable, new()
{
    private readonly Stack<T> _pool;
    private readonly int _maxSize;
    private int _totalCreated;

    public ObjectPool(int initialSize, int maxSize = 1000)
    {
        _maxSize = maxSize;
        _pool = new Stack<T>(initialSize);

        // 预热：创建初始对象
        for (int i = 0; i < initialSize; i++)
        {
            var obj = new T();
            obj.OnCreated();
            _pool.Push(obj);
            _totalCreated++;
        }
    }

    public T Get()
    {
        T obj;
        if (_pool.Count > 0)
        {
            obj = _pool.Pop();
        }
        else if (_totalCreated < _maxSize)
        {
            obj = new T();
            obj.OnCreated();
            _totalCreated++;
        }
        else
        {
            return null; // 池已满
        }

        obj.OnGetFromPool();
        return obj;
    }

    public void Return(T obj)
    {
        obj.OnReturnToPool();
        _pool.Push(obj);
    }

    public int AvailableCount => _pool.Count;
    public int TotalCreated => _totalCreated;
}

public interface IPoolable
{
    void OnCreated();         // 首次创建
    void OnGetFromPool();    // 从池中取出时重置
    void OnReturnToPool();   // 回收时清理
}

// 使用示例：子弹对象池
public class BulletManager
{
    private ObjectPool<Bullet> _bulletPool =
        new(initialSize: 200, maxSize: 2000);

    public Bullet SpawnBullet(Vector3 pos, Vector3 dir)
    {
        var bullet = _bulletPool.Get();
        if (bullet == null) return null;
        bullet.Init(pos, dir);
        return bullet;

    {

}
何时使用对象池
频繁创建和销毁的对象：子弹、粒子、伤害数字、敌人、拾取物。如果对象创建频率低于每秒一次，对象池可能是过度优化。

07 / 跨平台开发

一款游戏可能需要运行在 PC、主机、移动端、Web 等多个平台上。每个平台有不同的硬件能力、输入方式、分辨率、API 限制。跨平台开发的核心是抽象平台差异、共享核心逻辑。

平台差异概览

维度PC主机移动Web
输入 键鼠 + 手柄 手柄 触屏 + 陀螺仪 键鼠 + 触屏
分辨率 可变，多显示器 固定（4K/1080p） 极度碎片化 可变（浏览器窗口）
性能 差异巨大 固定硬件 受限（发热限制） 受限（WASM 开销）

分辨率与屏幕适配

屏幕适配策略取决于游戏类型：

2D 游戏适配策略

  • 固定宽度：保证水平视野一致，上下可能有黑边或额外可视区域
  • 固定高度：保证垂直视野一致，常用于竖屏游戏
  • 信封模式：设定最小和最大可视区域，在此范围内自适应
  • 像素完美：像素游戏必须保证整数倍缩放

3D 游戏适配策略

  • 固定 FOV：水平 FOV 固定，垂直 FOV 随宽高比变化
  • 动态分辨率：根据帧率动态调整渲染分辨率
  • UI 锚点系统：UI 元素锚定到屏幕边缘/角落

class GameConsole:
    var commands: Dictionary = {}
    var history: Array[String] = []
    var visible: bool = false

    func _ready() -> void:
        # 注册内置命令
        register("help", _cmd_help, "显示所有命令")
        register("god", _cmd_god, "切换无敌模式")
        register("noclip", _cmd_noclip, "切换穿墙模式")
        register("tp", _cmd_teleport, "传送到坐标: tp x y z")
        register("spawn", _cmd_spawn, "生成实体: spawn enemy_type")
        register("give", _cmd_give, "给予物品: give item_id count")
        register("speed", _cmd_speed, "设置游戏速度: speed 2.0")
        register("level", _cmd_level, "加载关卡: level level_name")
        register("stat", _cmd_stat, "显示性能统计")
        register("screenshot", _cmd_screenshot, "截图并保存")

                  desc: String) -> void:
        commands[name] = { "fn": callback, "desc": desc }

    func execute(input: String) -> String:
        var parts = input.strip_edges().split(" ")
        var cmd = parts[0].to_lower()
        var args = parts.slice(1)

        if not commands.has(cmd):
            return "Unknown command: %s" % cmd

        history.append(input)
        return commands[cmd]["fn"].call(args)

    func _cmd_speed(args: Array) -> String:
        if args.is_empty():
            return "当前速度: %.1f" % Engine.time_scale
        Engine.time_scale = args[0].to_float()
        return "游戏速度设为: %.1f" % Engine.time_scale

日志系统设计

一个好的日志系统是事后分析 Bug 的关键。特别是对于无法复现的、来自玩家报告的 Bug。

分级日志系统
enum LogLevel { TRACE, DEBUG, INFO, WARN, ERROR, FATAL }

class Logger:
    var _min_level: LogLevel = LogLevel.INFO
    var _outputs: Array[LogOutput] = []
    var _category: String

    func _init(category: String) -> void:
        _category = category

    func trace(msg: String) -> void: _log(LogLevel.TRACE, msg)
    func debug(msg: String) -> void: _log(LogLevel.DEBUG, msg)
    func info(msg: String) -> void:  _log(LogLevel.INFO, msg)
    func warn(msg: String) -> void:  _log(LogLevel.WARN, msg)
    func error(msg: String) -> void: _log(LogLevel.ERROR, msg)

    func _log(level: LogLevel, msg: String) -> void:
        if level < _min_level:
            return

        var entry = {
            "timestamp": Time.get_datetime_string_from_system(),
            "frame": Engine.get_frames_drawn(),
            "level": LogLevel.keys()[level],
            "category": _category,
            "message": msg,
        }

        for output in _outputs:
            output.write(entry)

# 使用方式
var log = Logger.new("Combat")
log.info("Player dealt 150 damage to Goblin_03")
log.warn("Boss HP below 10%, entering rage phase")
log.error("Invalid skill ID: %s" % skill_id)

日志输出到多个目标：

  • 控制台：开发时实时查看
  • 文件：滚动日志文件（按大小或日期分割）
  • 远程服务器：发布版本的崩溃日志和关键错误
  • 屏幕叠加：游戏画面上实时显示关键信息

重放系统（Replay）用于 Bug 复现

重放系统记录玩家的所有输入，可以精确重现游戏过程，是复现偶发 Bug 的终极武器。

重放系统原理

  1. 确定性：相同的输入序列 + 相同的初始状态 = 相同的结果
  2. 记录：每帧记录所有输入和随机数种子
  3. 回放：加载初始状态，按记录的输入重放
  4. 验证：回放时校验关键状态，确保确定性
重放系统基础实现
class ReplaySystem:
    var _recording: bool = false
    var _replaying: bool = false
    var _frames: Array[ReplayFrame] = []
    var _current_frame: int = 0
    var _initial_state: Dictionary
    var _rng_seed: int

    func start_recording() -> void:
        _frames.clear()
        _recording = true
        _rng_seed = randi()
        seed(_rng_seed)
        _initial_state = GameState.serialize()

    func record_frame(input: InputSnapshot) -> void:
        if not _recording: return
        _frames.append(ReplayFrame.new(
            Engine.get_frames_drawn(),
            input,
            # 记录校验用的状态哈希
            GameState.get_hash()
        ))

    func start_replay() -> void:
        GameState.deserialize(_initial_state)
        seed(_rng_seed)
        _current_frame = 0
        _replaying = true

    func get_replay_input() -> InputSnapshot:
        if not _replaying: return null
        if _current_frame >= _frames.size():
            _replaying = false
            return null
        var frame = _frames[_current_frame]
        # 验证确定性
        var current_hash = GameState.get_hash()
        if current_hash != frame.state_hash:
            push_error("Replay desync at frame %d!" % _current_frame)
        _current_frame += 1
        return frame.input

    func save_to_file(path: String) -> void:
        var data = {
            "version": "1.0",
            "seed": _rng_seed,
            "initial_state": _initial_state,
            "frame_count": _frames.size(),
            "frames": _frames.map(func(f): return f.serialize()),
        }
        var file = FileAccess.open(path, FileAccess.WRITE)
        file.store_string(JSON.stringify(data))

远程调试

发布后的游戏无法直接附加调试器。远程调试工具链：

  • 崩溃报告：Sentry、Crashlytics、Backtrace —— 自动收集 crash dump 和堆栈
  • 远程日志：将关键日志上传到服务器，按用户/设备过滤
  • 远程配置：通过服务端下发配置，无需更新客户端
  • 分析埋点：玩家行为数据、漏斗分析，发现隐性 Bug（如某关卡异常高的退出率）
  • 远程控制台：通过 WebSocket 连接到运行中的游戏，执行调试命令

09 / 项目管理

很多游戏项目死于管理失败而非技术问题。功能蔓延（Feature Creep）、不切实际的时间表、缺乏优先级 —— 这些是独立开发者和小团队面临的最大威胁。

游戏项目的范围管理（Scope）

Scope 管理是游戏项目成败的关键。核心原则：

Scope 三角形

功能 × 质量 × 时间 = 常数

你不能同时要更多功能、更高质量和更短的开发时间。独立开发者通常应该固定时间和质量，灵活调整功能范围。

控制 Scope 的方法：

  1. 定义 MVP（最小可玩产品）：如果游戏只能有 3 个功能，是哪 3 个？
  2. 分层需求：Must Have / Should Have / Nice to Have / Won't Have（MoSCoW）
  3. 时间盒：给每个功能设定时间上限，到时间没完成就砍掉或简化
  4. 原型验证：大功能先做最简原型验证是否有趣，不好玩就砍
  5. 定期重评估：每个里程碑重新审视剩余功能的必要性
Feature Creep 危险信号
  • "这个功能只需要多花一天" —— 累积效应会杀死项目
  • "别的游戏都有这个" —— 你不是别的游戏
  • "之后再砍也来得及" —— 沉没成本会让你不舍得砍
  • "玩家会期待这个" —— 先出 MVP 验证了再说

独立开发者的时间管理

实用策略

  • 固定工作时间：避免 crunch，持续稳定的产出远好于偶尔的爆发
  • 番茄工作法变体：45 分钟开发 + 15 分钟休息，保持专注
  • 开发日志：每天记录做了什么、遇到什么问题、明天做什么
  • 周回顾：每周末回顾进度，调整下周计划
  • 公开承诺：Devlog、社交媒体更新 —— 外部压力帮助维持动力
  • 最难的事先做：精力最好的时候做最有挑战的工作
  • 区分创造性工作和机械性工作：状态好时做设计/编程，疲劳时做美术调整/Bug 修复

代码审查

代码审查（Code Review）在游戏团队中容易被忽视，但对代码质量至关重要。

游戏项目代码审查重点

  • 性能：是否有每帧分配内存？是否有不必要的查找？
  • 线程安全：异步加载的资源访问是否安全？
  • 平台兼容：是否使用了特定平台的 API？
  • 硬编码：Magic Number 是否提取为常量/配置？
  • 内存泄漏：事件订阅是否有对应的取消订阅？
  • 边界条件：数组越界、空引用、除零
  • 可读性：命名是否清晰、注释是否解释了"为什么"

Git hooks — pre-commit 自动检查
#!/bin/bash
# .git/hooks/pre-commit

echo "Running pre-commit checks..."

# 1. 检查文件命名
bad_names=$(git diff --cached --name-only --diff-filter=ACR | \
  grep -E '[A-Z ]' | grep -E '^project/assets/')

if [ -n "$bad_names" ]; then
  echo "ERROR: 资源文件名不符合规范（需要小写+下划线）:"
  echo "$bad_names"
  exit 1
fi

# 2. 检查大文件（未通过 LFS）
large_files=$(git diff --cached --name-only --diff-filter=ACR | \
  while read f; do
    if [ -f "$f" ]; then
      size=$(wc -c < "$f")
      if [ "$size" -gt 5242880 ]; then  # 5MB
        echo "$f ($(($size / 1048576))MB)"
      fi
    fi
  done)

if [ -n "$large_files" ]; then
  echo "WARNING: 大文件未通过 LFS 管理:"
  echo "$large_files"
  echo "请使用 git lfs track 后再提交"
  exit 1
fi

# 3. 检查调试代码残留
if git diff --cached --name-only --diff-filter=ACM | \
  grep -E '\.gd$' | \
  xargs grep -l 'breakpoint()' 2>/dev/null; then
  echo "ERROR: 发现 breakpoint() 残留，请移除后再提交"
  exit 1
fi

echo "All checks passed!"
tools/asset_check.sh — 美术资源规格检查
#!/bin/bash
# 检查美术资源是否符合规格

ASSET_DIR="${1:-.}"
ERRORS=0

echo "Checking assets in: $ASSET_DIR"
echo "================================"

# 检查纹理尺寸是否为 2 的幂
echo "Checking texture dimensions..."
while IFS= read -r -d '' file; do
  if command -v identify &> /dev/null; then
    dims=$(identify -format "%wx%h" "$file" 2/dev/null)
    w=$(echo "$dims" | cut -d'x' -f1)
    h=$(echo "$dims" | cut -d'x' -f2)
    if [ -n "$w" ] && [ -n "$h" ]; then
      if ! (( (w & (w-1)) == 0 && (h & (h-1)) == 0 )); then
        echo "  WARN: Non-POT texture: $file (${w}x${h})"
        ((ERRORS++))
      fi
    fi
  fi
done < <(find "$ASSET_DIR/textures" -type f \
  \( -name "*.png" -o -name "*.jpg" \) -print0 2/dev/null)

# 检查文件大小
echo "Checking file sizes..."
while IFS= read -r -d '' file; do
  size=$(wc -c < "$file")
  if [ "$size" -gt 10485760 ]; then  # 10MB
    echo "  WARN: Large file: $file ($(($size/1048576))MB)"
    ((ERRORS++))
  fi
done < <(find "$ASSET_DIR" -type f -print0)

echo "================================"
echo "Found $ERRORS issue(s)."
exit $(( ERRORS > 0 ? 1 : 0 ))

全章总结

工程实践核心原则

  1. 结构先行：项目一开始就建立好目录结构和命名规范，后期成本呈指数增长
  2. 版本控制是生命线：不管团队大小，从第一天就使用 Git，频繁提交小改动
  3. 自动化一切可重复的事：构建、测试、检查 —— 机器做得比人好
  4. 测试不是浪费时间：纯逻辑单元测试 + 场景加载冒烟测试，投入产出比最高
  5. 先测量，再优化：Profiler 是唯一的真理来源，不要凭直觉优化
  6. 内存是资源，不是免费的：对象池、引用计数、流式加载是三大武器
  7. 跨平台先抽象：平台差异藏在接口后面，游戏逻辑只跟接口打交道
  8. Debug 工具是值得投资的：控制台、Debug Draw、日志系统 —— 前期投入，全程受益
  9. Scope 是独立开发者的命：砍功能 > 降质量 > 延期，能发布的游戏才有价值
  10. 流程服务于效率：协作流程适度就好，不要为了流程而流程
