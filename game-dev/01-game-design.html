01什么是游戏设计

游戏设计是关于创造有意义的选择的学科。它不是编写代码，不是绘制美术资源，不是制作音效——虽然这些都是游戏开发不可或缺的部分。游戏设计的核心任务是：定义规则、构建系统、塑造体验。

一个优秀的游戏设计师需要理解人类心理、系统思维、叙事结构、概率论、经济学、视觉传达等多个领域的知识。他们的最终目标不是创造一个"好看的产品"或"技术先进的作品"，而是创造一段让玩家投入、享受、并愿意反复体验的交互过程。

"游戏设计就是创造让玩家做出有意义选择的空间。" —— Sid Meier，《文明》系列创造者

游戏设计师的角色定位

游戏设计师在团队中的角色类似于建筑师——他们不亲自砌砖搬瓦，但他们决定了建筑的结构、空间布局、动线规划和居住体验。具体来说，游戏设计师需要承担以下职责：

  • 体验架构师 —— 定义玩家在游戏中会经历什么：什么时候感到紧张，什么时候感到成就感，什么时候感到好奇。整体的情感曲线是设计师的直接产出。
  • 规则制定者 —— 游戏的所有规则、限制、资源模型都由设计师制定。哪些行为被允许？哪些被惩罚？资源如何生成和消耗？这些决定了游戏的根本运作方式。
  • 平衡调控者 —— 一个数值差异可能让游戏从"有挑战"变成"令人挫败"。设计师负责调整难度曲线、经济系统、角色能力等一切影响游戏平衡的参数。
  • 沟通桥梁 —— 设计师是美术、程序、策划、音频之间的粘合剂。他们需要用清晰的文档和沟通确保每个人都理解游戏的愿景，并朝同一方向努力。
  • 玩家代言人 —— 在开发团队中，设计师始终站在玩家的角度思考。当技术团队想用某个方案但会影响体验时，设计师要替玩家发声。

游戏设计 vs 游戏开发 vs 游戏编程

初学者经常将这三个概念混为一谈，但它们代表了完全不同的职能领域。理解它们的区别对于规划你的学习路径至关重要。

维度 游戏设计 游戏开发 游戏编程
核心问题 "游戏应该是什么体验？" "如何把游戏做出来？" "代码如何实现这个功能？"
产出物 设计文档、规则原型、数值表、系统图 可运行的游戏产品 源代码、工具、引擎系统
核心技能 系统思维、心理学、叙事、概率论 项目管理、跨领域协调 编程语言、算法、引擎架构
关注重点 玩家体验、乐趣、意义 进度、质量、成本 性能、稳定性、可维护性
类比 建筑师 总承包商 结构工程师
独立开发者的现实：在独立游戏开发中，一个人往往需要同时身兼这三个角色。但即使如此，在工作时也要有意识地切换"设计帽子"和"技术帽子"。戴着技术帽子时解决实现问题，戴着设计帽子时关注体验问题——混在一起只会两边都做不好。

设计思维在游戏中的应用

设计思维（Design Thinking）是一种以用户为中心的问题解决方法论。在游戏设计中，这种方法论的应用非常自然且高效。其核心五步法可以直接映射到游戏设计流程中：

1. 共情（Empathize）

深入理解你的目标玩家群体。他们平时玩什么游戏？他们在游戏中追求什么？什么让他们感到挫败？通过观察玩家行为、阅读社区讨论、分析市场数据来建立对玩家的理解。

方法：玩家访谈、竞品分析、社区观察、数据分析

2. 定义（Define）

将你的发现提炼成一个清晰的设计目标。不是"做一个好游戏"，而是类似"为通勤族设计一个每局 3-5 分钟、能带来成就感的策略游戏"。越精确，越有方向。

方法：玩家画像、体验目标声明、设计约束清单

3. 构想（Ideate）

产生尽可能多的解决方案，不做判断。一个机制可以有二十种实现方式——先全部列出来，再评估。头脑风暴时最重要的原则是"不否定"。

方法：头脑风暴、类比法、随机输入法、限制变换法

4-5. 原型 & 测试（Prototype & Test）

用最低成本制作可玩的原型，然后让真实玩家测试。观察他们的行为——他们是否理解规则？是否感到有趣？在哪里放弃了？数据不会撒谎。

方法：纸面原型、灰盒原型、A/B 测试、有声思考法

核心理念

游戏设计不是一次性的蓝图制作，而是一个持续发现的过程。你的第一个想法几乎永远不是最好的想法。好的设计来自于反复的"制作→测试→学习→改进"循环。这也是为什么我们会在后面的章节中用大量篇幅讨论原型和迭代。

02MDA 框架

MDA 框架由 Robin Hunicke、Marc LeBlanc 和 Robert Zubek 在 2004 年提出，是游戏设计领域最有影响力的分析工具之一。它提供了一种结构化的方式来理解游戏体验的构成——从底层规则到最终的情感体验。

玩家视角（自右向左感知）

Mechanics（机制层）

机制是游戏的规则和系统。它们是可以被精确描述和编码的具体元素：移动规则、资源系统、战斗公式、建造条件等。机制是设计师直接控制的部分。

以国际象棋为例，机制包括：棋盘是 8×8 的方格，每种棋子有固定的移动方式（马走日、象走斜线），吃掉对方国王则获胜。这些规则是明确的、不可变的。

机制的关键特征：

  • 可以用代码或数学公式精确描述
  • 独立于玩家行为而存在
  • 是设计师最直接的调控手段
  • 本身不包含"乐趣"——乐趣来自机制之间的交互

Dynamics（动态层）

动态是机制在运行时产生的涌现行为（Emergent Behavior）。当玩家与机制互动时，会产生设计师可能预见（也可能没预见）的行为模式和策略。

继续以国际象棋为例：开局理论、中盘战术、残局技巧——这些都不是规则本身规定的，而是从简单规则中涌现出来的复杂策略系统。"牺牲换将"这种战术并没有写在规则书里，但它自然而然地从规则中产生。

动态的关键特征：

  • 不能被直接设计，只能通过调整机制来间接影响
  • 高度依赖玩家的参与方式
  • 涌现性——简单规则的组合可以产生极其复杂的行为
  • 多人游戏中的社交动态尤其难以预测

Aesthetics（美学体验层）

注意：这里的"美学"不是指画面好不好看，而是指游戏给玩家带来的情感体验。LeBlanc 定义了八种核心美学体验：

设计者视角 vs 玩家视角

MDA 框架最深刻的洞察在于：设计师和玩家从完全相反的方向看待游戏。

设计师视角：M → D → A

设计师从机制出发。他们定义规则、编码系统、设定参数。然后这些机制产生动态行为，最终（如果一切顺利的话）为玩家创造出预期的美学体验。

设计师的思考路径："如果我让子弹速度降低 20%，玩家就需要预判射击（动态），这会增加技巧感和成就感（美学）。"

玩家视角：A → D → M

玩家首先感受到的是情感体验："这个游戏好刺激/好治愈/好烧脑"。如果他们深入思考，会注意到产生这种体验的行为模式。只有最核心的玩家才会去研究底层机制。

玩家的感知路径："这游戏好有成就感（美学）→ 因为每次精确闪避都很爽（动态）→ 原来是因为判定框只有 3 帧（机制）。"

设计启示

这意味着：当玩家说"这游戏不好玩"时，问题出在美学层——但解决方案往往在机制层。你不能直接"设计乐趣"，你只能设计规则和系统，然后期望它们产生乐趣。这就是为什么游戏设计如此困难：因为你在间接操控体验。

案例分析

案例 1 《俄罗斯方块》—— 简洁之美

机制（M）：七种不同形状的方块从屏幕顶部随机下落。玩家可以左右移动方块和旋转方块。方块堆叠在底部。当一行被完全填满时，该行消除。消除多行可获得更多分数。下落速度随等级递增。

动态（D）：玩家发展出"留一列清四行"的T-spin策略。随着速度加快，玩家从深思熟虑的策略思考转变为依赖直觉的快速反应。堆积的方块制造持续的紧迫感。"差一行就满"的状态创造高度紧张。

美学（A）：挑战（不断提升的难度）、自愿服从（容易进入心流状态）、感官愉悦（消行时的满足感）。这三种体验的组合使得俄罗斯方块成为史上最令人上瘾的游戏之一。

设计启示：俄罗斯方块只有极少量的机制，但这些机制之间的交互产生了丰富的动态行为。这是"少即是多"的完美诠释——好的设计不是添加更多机制，而是让现有机制产生更多有趣的交互。
案例 2 《Among Us》—— 社交涌现

机制（M）：一名内鬼，多名船员。船员完成任务，内鬼暗中破坏和杀人。任何人可以召开会议进行讨论和投票。被投票最多的人被驱逐。船员完成所有任务或驱逐所有内鬼则获胜；内鬼杀掉足够多的船员则获胜。

动态（D）：信任与欺骗的复杂社交博弈。假不在场证明的编造与拆穿。"跟踪"策略——船员跟着可疑玩家以确认身份。"自证清白"——主动展示完成任务来证明自己是船员。联盟的形成与瓦解。诬陷策略——内鬼主动报告自己杀的人。

美学（A）：社交（与朋友的互动是核心乐趣）、叙事（每局都是一个独特的故事）、挑战（推理与说服）、表达（展示你的"演技"）。

设计启示：Among Us 的核心机制极其简单，但加上"自由交流"这一要素后，产生了无限丰富的社交动态。最好的多人游戏机制往往是那些能催生自然社交互动的机制。
案例 3 《文明 VI》—— 系统交织

机制（M）：回合制 4X 策略（探索、扩张、利用、消灭）。城市建设与人口管理。科技树与文化树。军事单位的生产、移动与战斗。外交系统（贸易、联盟、战争）。多种胜利条件（科技、文化、统治、宗教、分数）。

动态（D）："再来一回合"效应——每回合都有新进展，让人不忍停手。科技选择导致文明发展路径分化。与 AI 对手的长期博弈关系。资源争夺引发的地缘政治冲突。早期扩张 vs 内政发展的战略抉择。

美学（A）：幻想（引领文明从石器时代走向太空时代）、挑战（多维度的战略决策）、探索（地图迷雾的揭开）、表达（选择你的文明发展路线）、叙事（一局游戏就是一部文明史诗）。

设计启示：文明系列展示了如何通过多个系统的交织创造深度。但复杂不等于好——每个系统都必须与其他系统产生有意义的交互，否则就是无效的复杂性。Sid Meier 的设计哲学是"砍掉那些不有趣的部分"。

03核心游戏循环

如果说 MDA 框架告诉我们"游戏体验是如何构成的"，那么核心游戏循环（Core Game Loop）告诉我们的是"玩家在游戏中做什么"。每一个成功的游戏，无论多复杂，都可以被还原为一组嵌套的循环结构。

什么是游戏循环

游戏循环是玩家在游戏中反复执行的行为-反馈-奖励-行为的闭环。之所以叫"循环"，是因为玩家会不断地重复这个过程——而这正是游戏能持续吸引玩家的关键。

基础游戏循环模型

→

→
奖励 (Reward)

动机增强
→ ↺ 循环
核心原则

好的游戏循环满足三个条件：

  • 清晰——玩家清楚地知道自己可以做什么（行为选项明确）
  • 即时——每个行为都能得到快速且有意义的反馈
  • 递进——奖励能增强动机，推动玩家进入下一轮循环

如果循环中的任何一个环节断裂——比如反馈不清晰、奖励无意义、或者玩家不知道该做什么——玩家就会脱离游戏。

嵌套循环结构（微观/中观/宏观）

真实的游戏不会只有一个循环。它们由多层嵌套的循环组成，每一层对应不同的时间尺度和目标层级：

微观循环（Micro Loop）—— 秒级

玩家在几秒内反复执行的核心操作。这是游戏"手感"的来源。

  • 射击游戏：瞄准 → 开枪 → 命中反馈 → 弹药减少/敌人受伤 → 调整位置 → 再瞄准
  • 平台跳跃：跳跃 → 空中控制 → 着陆 → 判断下一个平台 → 再跳跃
  • 消除游戏：观察 → 发现匹配 → 交换 → 消除动画 → 新方块落下 → 再观察

设计要点：微观循环必须极其流畅、手感好、反馈即时。这是玩家停留在游戏中的最直接原因。如果每次按下跳跃键的感觉都不爽，再精彩的关卡设计也救不了。

中观循环（Meso Loop）—— 分钟到小时级

由多个微观循环组成的更大目标结构。提供方向感和成就感。

  • 射击游戏：完成一场对战 → 获得经验值/道具 → 升级/解锁 → 加入下一场对战
  • RPG：接受任务 → 探索区域 → 战斗/解谜 → 完成任务 → 获得奖励 → 接受新任务
  • Roguelike：选择路径 → 通过一系列房间 → 获取升级 → 遭遇 Boss → 死亡或通关

设计要点：中观循环给玩家一个"再玩一局/再做一个任务"的理由。它需要有明确的开始和结束，每次循环结束时玩家应该有所收获（不管是装备、技能还是游戏知识）。

宏观循环（Macro Loop）—— 小时到天/周级

跨越多个游戏会话的长期目标和进程系统。这是游戏"深度"和"长期留存"的来源。

  • 射击游戏：赛季 → 段位系统 → 解锁赛季奖励 → 新赛季重置
  • RPG：主线剧情推进 → 角色长期培养 → 解锁新区域 → 最终 Boss
  • 建造/经营：城市/基地持续扩展 → 解锁新建筑/科技 → 面对更大挑战 → 达成里程碑

设计要点：宏观循环是玩家"为什么还在玩这个游戏"的答案。它需要提供持续的目标感和进步感。没有宏观循环的游戏容易让人觉得"玩几局就腻了"。

经典游戏的循环拆解

循环拆解《哈迪斯》（Hades）

Supergiant 的 Roguelike 动作游戏是循环设计的典范：

循环层级结构时间尺度
微观 冲刺 → 攻击 → 闪避敌人攻击 → 使用技能 → 拾取生命/金币 秒级
中观 清理房间 → 选择奖励（神恩） → 进入下一房间 → 遭遇 Boss → 死亡或通关地区 分钟级
宏观 一轮逃离尝试 → 死亡回到大厅 → 永久升级 → 推进剧情对话 → 尝试新 Build → 再次逃离 30-60 分钟/轮
元宏观 解锁所有武器形态 → 完成所有 NPC 剧情 → 提高热度挑战 → 挑战极限难度 数十小时

精妙之处：Hades 天才般地将叙事融入了循环结构。在大多数 Roguelike 中，死亡只意味着"重来"，但在 Hades 中，死亡意味着回到家——你可以和 NPC 对话、推进关系、了解更多故事。这将"死亡"这个通常消极的事件转化为正向的叙事驱动力。玩家不再抗拒死亡，而是期待死后的对话。

循环拆解《Minecraft》
循环层级结构
微观 挖掘方块 → 获得资源 → 放置方块 → 建造结构
中观 收集材料 → 合成工具/装备 → 探索新区域 → 发现更稀有的资源

精妙之处：Minecraft 的循环是完全玩家驱动的。游戏不会告诉你"下一步应该做什么"——宏观目标完全由玩家自己设定。这是其深度自由度的根源，但也是许多新玩家感到迷茫的原因。后来的更新中添加的成就系统正是为了给缺乏自驱力的玩家提供引导。

拆解你最近玩的游戏的循环结构。

  1. 找出它的微观循环——你在几秒内反复做的事情是什么？
  2. 找出它的中观循环——是什么驱动你"再来一局/再完成一个任务"？
  3. 如果存在宏观循环，描述它是如何为你提供长期目标的
  4. 评估：各层循环之间的衔接是否顺畅？有没有某个层级的体验明显薄弱？

04玩家动机与心理学

理解玩家心理是游戏设计师最重要的能力之一。你不是在设计一个抽象的系统——你是在设计一段人类体验。如果你不理解人类的动机、情感和认知特点，你的设计就是在黑暗中射箭。

Bartle 玩家分类

Richard Bartle 在 1996 年提出了一个影响深远的玩家分类模型，基于他对 MUD（多用户地下城，网络游戏的前身）玩家行为的观察。他将玩家分为四种类型，基于两个维度：作用于玩家 vs 作用于世界，以及互动 vs 单向行为。

⚔ 成就者（Achiever）

座右铭："我要赢"

成就者追求游戏内的目标完成和资源积累。他们享受升级、收集、解锁成就的过程。排行榜和稀有成就是他们的核心驱动力。

他们喜欢：成就系统、排行榜、收集要素、可量化的进度、稀有物品

他们讨厌：没有明确目标、进度感不强、努力得不到可见回报

设计策略：提供清晰的目标层级、可追踪的进度条、有意义的成就奖励

☂ 探索者（Explorer）

座右铭："我想知道"

探索者享受发现的过程——无论是隐藏区域、游戏机制的深层原理，还是彩蛋和秘密。他们对未知充满好奇，乐于实验和试错。

他们喜欢：隐藏区域、秘密机制、开放世界、知识发现、彩蛋

他们讨厌：线性受限的路径、没有秘密可发现、所有东西都显而易见

设计策略：在地图中隐藏秘密、让系统有深度可挖掘、奖励好奇心

♡ 社交者（Socializer）

座右铭："我们一起"

社交者来游戏主要是为了与人互动。游戏本身只是他们社交的媒介。他们享受合作、交流、帮助新人、参与公会活动。

他们喜欢：聊天系统、公会/社团、合作任务、社区活动、角色自定义（用于社交展示）

他们讨厌：强制单人内容、无法与朋友互动、有毒社区

设计策略：创造需要协作的内容、提供社交空间、建设正向社区文化

☠ 杀手（Killer）

座右铭："我要赢你"

杀手享受对其他玩家施加影响——可以是竞技对战中的击败对手，也可以是通过策略操控其他玩家。他们追求的是相对优势，而非绝对成就。

他们喜欢：PvP 系统、竞技排名、对抗性玩法、展示实力的机会

他们讨厌：没有竞争对象、实力差异不被体现、无法展示优越性

设计策略：提供公平的竞技场、段位/天梯系统、竞技赛事

Bartle 模型的局限性：这个模型诞生于上世纪 90 年代的文字 MUD 环境，对现代游戏的适用性有所下降。一个玩家可能在不同游戏（甚至同一游戏的不同阶段）中表现出不同的类型倾向。更现代的分类模型（如 Quantic Foundry 的动机模型）提供了更细粒度的六维度分析：行动、社交、掌控、成就、创造、沉浸。但 Bartle 模型作为入门工具仍然非常有价值——它教会我们的核心认知是：不同玩家从同一个游戏中追求的东西完全不同。

心流理论（Flow Theory）

心理学家 Mihaly Csikszentmihalyi 提出的心流理论是理解"为什么游戏如此吸引人"的关键框架。心流是一种完全沉浸在活动中的状态——你忘记了时间的流逝，忘记了周围的环境，全部注意力都聚焦在当前的任务上。

心流的条件

要让玩家进入心流状态，游戏设计需要满足以下条件：

  • 技能与挑战匹配 —— 这是最核心的条件。如果挑战远超技能，玩家感到焦虑；如果技能远超挑战，玩家感到无聊。心流存在于两者之间的狭窄通道中。
  • 明确的目标 —— 玩家必须清楚地知道自己要做什么。模糊的目标会导致注意力分散。
  • 即时的反馈 —— 每一个行为都应该得到快速的反馈，让玩家知道自己做得对还是错。
  • 可控感 —— 玩家感到自己的行为对结果有影响，而不是被随机性或不公平的设计控制。
  • 注意力集中的环境 —— 减少干扰因素，让玩家能够专注。频繁的弹窗、广告、不必要的 UI 提示都是心流的杀手。

设计挑战

动态难度调整（DDA）是维持心流的技术手段之一。许多现代游戏会根据玩家的表现自动调整难度：如果玩家频繁死亡，敌人会变弱或提供更多提示；如果玩家轻松通关，难度会上升。《生化危机 4》是 DDA 的经典案例——它的"影响力系统"根据玩家死亡次数和通关速度来调整敌人数量、攻击频率和物品掉落。关键是做得不露痕迹——玩家不应该意识到难度在被调整，否则会损害成就感。

内在动机 vs 外在动机

这是自我决定理论（Self-Determination Theory, Deci & Ryan）在游戏设计中的核心应用。

内在动机（Intrinsic Motivation）

"因为有趣/有意义而做"

内在动机来自活动本身带来的满足感。人类有三种基本心理需求：

  • 自主性（Autonomy）—— 感觉是自己在做选择，而非被迫
  • 能力感（Competence）—— 感觉自己在进步、在掌握技能
  • 关联感（Relatedness）—— 与他人产生有意义的连接

游戏中的内在动机示例：

  • 因为战斗系统手感极好而反复战斗（能力感）
  • 因为想知道故事的走向而继续推进（好奇心）
  • 因为建造本身很有满足感而建造（自主性 + 能力感）

外在动机（Extrinsic Motivation）

"因为奖励/避免惩罚而做"

外在动机来自活动之外的回报，如积分、成就、排名、虚拟货币等。

游戏中的外在动机示例：

  • 每日登录奖励
  • 限时活动奖励
  • 排行榜排名
  • 成就/奖杯系统
  • 赛季 Pass 奖励

警告：过度依赖外在动机会产生"过度正当化效应"（Overjustification Effect）——当外在奖励被移除时，玩家反而比没有奖励时更不想玩。如果玩家只是为了日常任务奖励而登录，一旦奖励不够吸引了，他们会立刻流失。

黄金法则

最持久的游戏体验来自内在动机。外在动机可以引导和强化，但不能替代内在动机。设计时的优先级应该是：首先确保游戏的核心玩法本身有趣（内在动机），然后用外在奖励系统来强化和引导（外在动机）。如果你需要用大量外在奖励来"贿赂"玩家继续玩，说明你的核心玩法本身可能存在问题。

多巴胺与奖励机制

多巴胺是大脑中与期待和动机相关的神经递质。注意：多巴胺不是"快乐激素"——它更准确的功能是让你想要去做某件事。理解多巴胺机制对设计奖励系统至关重要。

可变比率奖励（Variable Ratio Schedule）

这是最强大的奖励模式，也是赌博和很多游戏奖励系统的核心。它的特点是：奖励出现的时机不可预测，但平均频率固定。

为什么有效：不确定性本身会触发多巴胺分泌。比起"每杀 10 个怪必掉一件装备"，"每杀一个怪都有 10% 的概率掉落"更能让玩家保持兴奋——因为每一次击杀都可能是那个幸运时刻。

在游戏中的应用：

  • 战利品掉落（Loot Drop）—— 怪物随机掉落装备
  • 抽卡/扭蛋（Gacha）—— 随机获得角色/物品
  • 程序化生成的关卡 —— 每次都不一样的体验
  • 随机事件 —— 意外遭遇、随机奖励

四种主要强化计划

类型描述效果游戏示例
固定比率 每完成 N 次行为给一次奖励 产生"接近奖励时加速"的行为模式 每 10 级解锁一个新技能
可变比率 平均每 N 次行为给一次奖励，但具体时机随机 持续高频行为，最难消退 怪物掉落、抽卡系统

损失厌恶（Loss Aversion）

心理学研究表明，人类对失去的痛苦感受约为获得同等价值快乐的 2 倍。游戏设计中大量利用了这一原理：

  • 限时活动 —— "如果你不现在登录，就会错过这个奖励"
  • Roguelike 的死亡惩罚 —— 你辛苦获得的一切在死亡时都可能失去
  • 耐久度系统 —— 不修理就会失去你的武器
  • FOMO（错失恐惧） —— 限时皮肤、赛季独占奖励

伦理警示：损失厌恶是一把双刃剑。适度使用可以增加紧张感和决策权重，但过度使用会让玩家感到被操控、焦虑，甚至产生不健康的游戏习惯。设计师有责任区分"有趣的紧张"和"有害的焦虑"。

审视一个你长期游玩的游戏，分析它的心理学机制。

  1. 你的主要动机是内在还是外在的？如果外在奖励全部移除，你还会玩吗？
  2. 游戏中有哪些可变比率奖励的例子？它们如何影响你的行为？
  3. 游戏是否使用了损失厌恶？如果是，你觉得这种使用是否合理？
  4. 你在什么时候容易进入心流状态？这与难度曲线的设计有什么关系？

05游戏类型与核心机制

游戏类型（Genre）是一种约定俗成的分类方式，帮助玩家快速理解一个游戏的基本玩法特征。但对于设计师来说，更重要的是理解每种类型背后的核心机制——是什么样的规则和系统定义了这种类型的本质特征。

主要游戏类型分类

动作类（Action）

核心机制：实时输入响应、碰撞检测、物理模拟

核心体验：技巧感、反应速度、手眼协调

动作游戏的核心在于实时操作。玩家的表现直接取决于他们的操作技巧和反应速度。所有动作游戏共享的底层设计问题是"如何让操作手感好"——这涉及输入延迟、动画取消、缓冲输入、打击反馈等极其细致的调校工作。

子类型：平台跳跃 格斗 动作冒险 Beat'em up 角色动作 (Character Action)

代表作：《蔚蓝》（精密平台跳跃）、《鬼泣 5》（极致连招系统）、《空洞骑士》（类银河恶魔城）

射击类（Shooter）

核心机制：射击手感（弹道、后坐力、命中判定）、武器系统、掩体/空间控制

核心体验：精准性、空间意识、紧张感

射击游戏的"手感"（Game Feel）可能是所有类型中最难做好的。一发子弹从按下扳机到命中敌人的整个过程——枪口火焰、后坐力动画、弹道轨迹、命中反馈（Hitmarker）、敌人受击反应——每一个细节都影响着射击是否"打起来爽"。

子类型：FPS TPS 战术射击 弹幕射击 (Bullet Hell) 大逃杀

代表作：《CS2》（竞技 FPS 标杆）、《Splatoon 3》（创新射击）、《东方 Project》（弹幕射击）

角色扮演（RPG）

核心机制：角色成长系统（经验值/等级/技能树）、数值系统（属性/装备/增益）、叙事分支

核心体验：代入感、成长感、叙事沉浸

RPG 的本质是角色的成长与变化。无论是日式 RPG 的线性叙事 + 回合制战斗，还是西方 RPG 的开放世界 + 自由选择，核心吸引力都是"看着你的角色从弱小变得强大，同时经历一个引人入胜的故事"。

子类型：JRPG WRPG ARPG CRPG MMORPG Roguelike RPG

代表作：《博德之门 3》（CRPG 巅峰）、《女神异闻录 5》（JRPG 典范）、《艾尔登法环》（ARPG）

策略类（Strategy）

核心机制：资源管理、决策树、战术/战略规划、信息博弈

核心体验：智力挑战、掌控感、长远规划

策略游戏考验的是决策质量而非操作速度。核心设计挑战是"如何让每个决策都有趣"——这意味着每个选择都必须有权衡（Trade-off）。如果存在一个明显的最优策略，游戏就会变得无聊。

子类型：RTS 回合制策略 (TBS) 4X 塔防 自走棋 卡牌策略

代表作：《文明 VI》（4X）、《杀戮尖塔》（Roguelike 卡牌）、《火焰纹章》（SRPG）

模拟/建造/经营（Simulation / Building / Management）

核心机制：系统模拟、资源流动、状态管理、空间规划

核心体验：创造感、优化感、表达感

这类游戏让玩家扮演"上帝"或"管理者"的角色，通过系统层面的决策来影响一个虚拟世界的运作。核心乐趣来自于看到你的决策产生连锁反应，以及不断优化系统效率的满足感。

子类型：城市建造 农场经营 生存建造 工厂自动化 生活模拟

代表作：《城市天际线 2》（城建）、《缺氧》（生存模拟）、《异星工厂》（自动化）、《星露谷物语》（农场）

解谜类（Puzzle）

核心机制：模式识别、逻辑推理、空间思维、规则发现

核心体验：顿悟时刻（Aha moment）、智力成就感

解谜游戏的设计核心是制造"卡住→突破"的体验。玩家面对一个看似无解的问题，通过观察、思考和尝试，突然发现解决方法——这个"顿悟时刻"是解谜游戏最宝贵的体验。难度设计的关键是让玩家感到"差一点就想到了"，而非"完全没有头绪"。

子类型：纯解谜 解谜冒险 编程解谜 物理解谜 叙事解谜

代表作：《The Witness》（纯解谜）、《传送门 2》（物理解谜）、《Baba is You》（规则解谜）

每种类型的核心机制特征

类型混合与创新

现代游戏设计中最令人兴奋的方向之一就是类型融合。几乎所有近年来的突破性作品都是类型混合的产物：

成功的类型混合案例

  • 《杀戮尖塔》= Roguelike + 卡牌构筑 —— 将 Roguelike 的随机性与卡牌游戏的策略深度结合，开创了一个全新的子类型。现在"Roguelike Deckbuilder"已经成为独立的品类。
  • 《吸血鬼幸存者》= Roguelike + 弹幕射击 + 自动化 —— 极简操作（只需移动）+ 大量涌现交互 + Roguelike 选择，用最低成本创造了高度上瘾的体验。
  • 《塞尔达：旷野之息》= 开放世界 + 物理模拟 + 解谜 —— 用物理和化学系统替代传统的"道具解谜"，让解题方式从固定变为无限开放。
  • 《极乐迪斯科》= RPG + 视觉小说 + 模拟 —— 去掉了 RPG 的战斗系统，全部的"战斗"都发生在对话和内心独白中。技能检定决定的不是你的攻击力，而是你的洞察力、共情力、说服力。
  • 《缺氧》= 模拟经营 + 生存 + 物理模拟 —— 将气体流动、热力学、管道系统等物理模拟融入经营游戏，创造了极深的优化空间。
混合设计原则

类型混合不是随意的机制叠加。成功的混合有一个共同特点：被混合的机制之间产生了协同效应（Synergy），1 + 1 > 2。失败的混合则是在主菜上堆配菜——每加一个新系统都增加了认知负担，但没有让核心体验变得更好。

判断标准：如果移除被混合的元素，核心体验是否会塌陷？如果会，说明融合是成功的；如果不会，说明那只是一个附加功能，不是真正的融合。

尝试一次类型混合实验。

  1. 随机选择两个游戏类型（比如"解谜 + 格斗"，"农场经营 + 弹幕射击"）
  2. 思考：这两种类型的核心机制如何产生有趣的交互？
  3. 写出 3 个可能的游戏概念
  4. 评估：哪个概念中两种类型之间有最强的协同效应？

06原型设计与迭代

"不要在纸上设计游戏。"这是每一个资深游戏设计师都会告诉新人的第一条建议。游戏是交互体验——只有当人与系统互动时，设计才能被真正验证。原型就是最低成本的验证方式。

"你的第一个想法是你最差的想法。最好的想法来自迭代。" —— Will Wright，《模拟城市》《模拟人生》创造者

快速原型的重要性

为什么原型如此重要？因为游戏设计中存在一个残酷的事实：你无法通过想象来判断一个机制是否有趣。

一个在脑海中觉得"肯定好玩"的机制，在实际玩到时可能毫无乐趣。反过来，一个看起来"很无聊"的机制，可能在玩的时候出奇地吸引人。在你亲手构建一个可玩原型之前，你对"好不好玩"的所有判断都只是猜测。

原型的目的不是做出一个完整的游戏，而是回答一个具体的设计问题：

  • "这个核心机制有趣吗？" → 核心机制原型
  • "这个关卡的难度曲线合理吗？" → 关卡原型
  • "这种操控方式舒服吗？" → 操控手感原型
  • "玩家能理解这些规则吗？" → 可用性原型

每个原型只需要回答一个问题。尝试用一个原型回答所有问题，结果就是你做了一个小型的完整游戏——这不叫原型，叫提前开发。

纸面原型 vs 数字原型

纸面原型（Paper Prototype）

制作时间：30 分钟 - 数小时

适用场景：

  • 回合制或低实时性的游戏
  • 规则和经济系统验证
  • 关卡布局和空间规划
  • 卡牌游戏、桌游机制
  • 对话/分支叙事流程

材料：卡片、骰子、棋子、白纸、记号笔、便利贴

优势：成本极低、迭代极快（修改规则只需要几秒钟）、强迫你聚焦于核心机制而非表现效果

局限：无法验证实时操作手感、无法模拟复杂的物理交互

数字原型（Digital Prototype）

制作时间：数小时 - 数天

适用场景：

  • 实时操作/动作类游戏
  • 物理模拟验证
  • 操控手感调校
  • 视觉效果对体验的影响
  • 网络多人交互

工具：游戏引擎（Unity、Godot）、编程框架、Twine（叙事原型）、Figma（UI 流程原型）

优势：能验证实时交互、可以精确测量参数、更接近最终产品体验

局限：成本较高、容易陷入"打磨细节"而偏离原型初衷

"灰盒原型"（Greybox Prototype） 是数字原型的一种重要形式。所有美术资源都用简单的几何体（灰色方块、胶囊体）代替，只关注空间尺度、操控手感和关卡节奏。几乎所有 3A 级别的游戏都会经历灰盒阶段——在确认关卡设计有效之前，不会投入美术资源。这是一种极其高效的开发方法，独立开发者更应该掌握。

迭代设计流程

迭代不是"修修补补"，而是一个有纪律的循环过程。每一轮迭代都应该有明确的目标、可度量的结果和清晰的决策标准。

迭代循环

  1. 提出假设 —— "如果我让角色跳跃高度增加 30%，玩家会觉得移动更自由。"假设必须是可验证的、具体的。
  2. 构建原型 —— 用最少的工作量实现假设。只改动与假设直接相关的部分。抵制"顺便改一下这个"的冲动。
  3. 进行测试 —— 让真实玩家测试原型。观察他们的行为，记录数据。不要在测试过程中解释规则——如果你需要解释，说明设计不够直观。
  4. 分析结果 —— 将观察到的行为与假设进行对比。假设被验证了吗？如果没有，是什么出了差错？是假设本身有问题，还是实现有偏差？
  5. 做出决策 —— 基于分析结果做出设计决策：保留、修改还是放弃当前方案。然后进入下一轮迭代。

Playtest 方法论

Playtest（游戏测试）是迭代设计中最关键的环节。但很多新手设计师不知道如何有效地进行 playtest。以下是经过实践验证的方法论：

Playtest 的黄金规则

  • 闭嘴 —— 测试开始后，不要解释、不要提示、不要辩护。你不会在玩家身边告诉他们怎么玩你发布的游戏。让玩家自己摸索，观察他们在哪里困惑。
  • 观察行为，忽略意见 —— 玩家会告诉你"这里不好玩"，但很少能准确说出为什么。关注他们做了什么（行为数据）比关注他们说了什么（主观评价）更有价值。
  • 提问不引导 —— 不要问"你觉得战斗系统好玩吗？"（暗示了你想听到的答案）。要问"你在战斗中印象最深的是什么？"（开放性问题）。
  • 记录一切 —— 录像、录音、笔记。你以为你会记住的东西，你不会。尤其要记录"出乎意料的玩家行为"——这些往往是最有价值的设计线索。

三种 Playtest 类型

类型测试者目的适用阶段
自我测试 设计师自己 快速验证基本功能和初步手感 原型初期
内部测试 团队成员或朋友 发现明显的设计问题和 Bug 原型中期
外部测试 目标玩家群体 验证可用性、乐趣度、留存度 原型后期到 Beta

注意：自我测试的价值有限，因为你太了解自己的设计。你知道所有的隐藏机制、最优策略和设计意图——而你的玩家不知道。自我测试只能验证"这东西能跑起来"，不能验证"这东西好不好玩"。

有声思考法（Think Aloud Protocol）

这是可用性测试中的经典方法，在游戏测试中同样有效。让测试者在玩的过程中不断说出他们在想什么：

  • "我看到了一个门，我觉得应该能打开……但是按了按钮没反应？"
  • "这个怪物看起来很强，我不确定应不应该打它……"
  • "哦！原来这里可以跳上去！好酷！"

这种方法能让你直接"听到"玩家的思考过程，快速定位设计中的困惑点和惊喜点。

制作一个纸面原型。

  1. 选择一个简单的游戏概念（比如一个 2 人竞速到终点的棋盘游戏）
  2. 用卡纸、便利贴和骰子，在 30 分钟内制作一个可玩原型
  3. 找一个朋友试玩，你在旁边观察（记住：不要解释规则之外的内容）
  4. 根据观察修改一个规则，然后再测试一次
  5. 记录两次测试的差异：哪个版本更好？为什么？

07游戏设计文档（GDD）

游戏设计文档（Game Design Document，简称 GDD）是将设计想法转化为团队可执行方案的桥梁。它不是文学作品，不是要写得"好看"——它是一份技术性的沟通工具，目的是让团队的每一个成员都清楚地理解游戏应该是什么样。

GDD 的结构与内容

一份完整 GDD 通常包含以下模块：

模块内容主要读者
概览 游戏概念、核心体验目标、目标受众、平台、参考作品 所有人
核心机制 玩家可以执行的所有行为、规则详述、系统交互 设计师、程序员
游戏循环 核心循环图示、进程系统、session 结构 设计师、程序员
关卡/世界设计 世界结构、关卡列表、难度曲线、环境叙事 关卡设计师、美术
数值系统 属性定义、公式、成长曲线、经济模型 程序员、QA
叙事设计 故事大纲、角色设定、对话系统、分支结构 叙事设计师、美术
美术方向 风格参考、色彩规范、角色设定概念、UI 风格 美术团队
音频设计 音乐风格、音效需求列表、环境音设计 音频设计师
技术需求 平台需求、性能目标、存档系统、网络架构 程序员
变现策略 商业模式、DLC 计划、内购设计（如适用） 制作人、市场

一页 GDD vs 完整 GDD

一页 GDD（One-Page GDD）

适用场景：项目早期概念验证、游戏 Jam、向投资人/团队快速传达概念

内容：

  • 游戏名称和一句话描述
  • 核心机制（最多 3 个）
  • 核心循环描述
  • 目标体验（1-2 种 MDA 美学类型）
  • 参考作品（2-3 个，说明参考了什么）
  • 独特卖点（USP）
  • 目标平台和受众

原则：如果你不能在一页纸上清晰描述你的游戏，你可能还没想清楚。一页 GDD 不是完整 GDD 的缩写版——它是一个聚焦工具，强迫你提炼出游戏的本质。

完整 GDD

适用场景：正式开发阶段、团队协作、外包沟通

注意事项：

  • 活文档 —— GDD 不是一次性写完的。它随着开发进程不断更新。标记版本号和修改日期。
  • 可搜索 —— 使用目录、标签、超链接。团队成员需要能快速找到他们关心的部分。
  • 具体而非模糊 —— "战斗要紧张刺激"是无效描述。"敌人攻击前有 0.8 秒的预警动画，玩家需要在 0.3 秒的窗口内闪避"才是有效描述。
  • 包含"为什么" —— 不只写"是什么"，还要写"为什么这样设计"。几个月后你自己都会忘记当初的设计理由。

## 3.1 移动系统

概述：角色使用八方向移动，支持冲刺和闪避。移动手感追求
轻快响应，参考《蔚蓝》的地面操控感。

基础移动：
• 移动速度：8 单位/秒
• 加速时间：0.08 秒（几乎即时，接近原生响应）
• 减速时间：0.12 秒（轻微滑行感）
• 空中水平速度：地面速度的 90%

冲刺：
• 触发：按住 Shift/R2
• 速度倍率：1.6x
• 体力消耗：3 点/秒
• 冲刺中可以取消进入闪避

闪避：
• 触发：Space/Circle（可在地面和空中使用）
• 闪避距离：3.5 单位
• 闪避持续时间：0.25 秒
• 无敌帧：0.15 秒（从触发开始计算）
• 冷却时间：0.4 秒
• 空中闪避次数：1 次（落地重置）

设计意图：移动系统追求"即时响应 + 轻微惯性"的手感。
闪避的无敌帧窗口（0.15 秒）经过测试，对核心玩家群体
来说属于"有挑战但可掌握"的难度水平。空中闪避限制为 1 次
是为了防止空中无限闪避导致的关卡设计限制失效。
现代 GDD 趋势：越来越多的团队开始用 Notion、Confluence 或 Google Docs 等协作工具替代传统的 Word 文档。关键优势是版本历史、实时协作、交叉引用和嵌入媒体。一些团队甚至用 Wiki 形式来组织 GDD，让每个系统有自己的页面。形式不重要，重要的是它能有效沟通设计意图。

08实践：设计你的第一个游戏概念

理论学得再多，不动手设计就永远只是旁观者。这一章是一个完整的实践项目——从一个模糊的灵感出发，一步步走完游戏概念设计的全流程。

从灵感到概念的步骤

  1. 收集灵感种子
    灵感可以来自任何地方：一个有趣的物理现象、一段个人经历、一种你想表达的情感、一个"如果……会怎样"的问题。不要一开始就想"做什么类型的游戏"——先找到你想表达的核心体验或有趣的核心互动。

    练习：花 10 分钟写下 10 个"如果……会怎样？"的问题。比如：
    • "如果重力方向会随着玩家的情绪变化？"
    • "如果你是一棵树，需要在城市中生存一百年？"
    • "如果两个玩家看到的是完全不同的世界，但必须合作？"
  2. 选择并聚焦
    从你的灵感列表中选出最让你兴奋的一个。然后用一句话描述它可能是什么样的游戏体验。不需要精确——你只需要一个方向。

    判断标准：选择那个你在解释时眼睛会发亮的想法。如果你自己都不兴奋，你不可能做出让别人兴奋的东西。
  3. 定义核心体验目标
    回到 MDA 框架，明确你想让玩家体验到什么。选择 1-2 种核心美学体验，并用一段话描述你想象中玩家在游戏过程中的感受。

    模板："我希望玩家在玩这个游戏时，主要感受到 [美学类型]。具体来说，当他们 [核心行为] 时，会感到 [情感]，因为 [原因]。"
  4. 设计核心机制
    提出 2-3 个支撑核心体验的机制。每个机制用一句话描述。然后问自己：这些机制之间会产生什么动态交互？是否有涌现行为的可能？

    检验：尝试用一副扑克牌或骰子模拟你的核心机制。如果你无法用实体道具模拟，可能说明你的机制还不够清晰。
  5. 画出核心循环
    用简单的箭头图画出你的微观循环和中观循环。标注每个环节的行为、反馈和奖励。检查循环是否闭合——有没有驱动力推动玩家从一个环节进入下一个？
  6. 识别参考作品
    找到 2-3 个与你的概念有相似元素的游戏。不是为了模仿它们，而是为了学习它们的设计智慧和避免它们的设计失误。对每个参考作品，明确你想学习的具体元素是什么。
  7. 撰写一页 GDD
    用上一章的模板，把你所有的想法整理成一份一页 GDD。这份文档应该让一个陌生人也能理解你的游戏概念。

概念验证清单

在投入开发之前，用以下清单验证你的概念是否有足够的可行性和吸引力：

核心乐趣验证

  • ☐ 你能用一句话说清楚"这个游戏最有趣的部分是什么"吗？
  • ☐ 核心机制在纸面原型上是否已经展现了乐趣？
  • ☐ 核心循环的每个环节是否都有清晰的驱动力？
  • ☐ 你的设计是否在至少一个维度上与现有游戏有明显差异？

范围与可行性验证

  • ☐ 以你当前的技能和资源，你能在合理时间内完成这个项目吗？
  • ☐ 你是否已经明确了项目的最小可玩版本（MVP）包含什么？
  • ☐ 你的 MVP 是否已经包含了核心乐趣？（如果 MVP 不有趣，加更多内容也不会有趣）
  • ☐ 有没有你不确定能否实现的技术风险？如果有，优先验证它们。

受众与市场验证

  • ☐ 你能描述出你的目标玩家是谁吗？（不是"所有人"）
  • ☐ 这些玩家目前在哪里找游戏？（Steam、App Store、itch.io？）
  • ☐ 同类游戏在市场上的表现如何？是过于拥挤还是有空间？
  • ☐ 你能用一段 30 秒的描述让你的目标玩家产生兴趣吗？

常见新手设计陷阱

陷阱 1：范围过大（Scope Creep）

这是杀死独立游戏项目的头号原因。新手设计师总是想做"一个拥有 100 小时内容的开放世界 RPG"作为第一个项目。

解药：你的第一个游戏应该能在 1-3 个月内完成。强迫自己在核心机制确定后做减法而不是加法。问自己："如果我只能保留两个机制，我会保留哪两个？"

实际案例：Celeste（蔚蓝）的前身是一个 Game Jam 作品，只花了 4 天制作。核心机制只有跳跃和冲刺。但因为这两个机制被打磨到了极致，所以即使内容极少也足够吸引人。后续的完整版就是在这个坚实的核心上扩展的。

陷阱 2：想法珍贵症（Idea Hoarding）

"这个想法太好了，我要等到技术/资源/时机都完美的时候再做。"

现实：想法本身几乎没有价值，执行才有价值。同一个想法，10 个人做出来会是 10 个完全不同的游戏。更重要的是，你以为的"完美想法"在实际测试后往往需要大幅修改。越早开始执行，越早发现问题。

解药：今天就开始做。用纸笔或最简单的工具。如果你还在"准备"和"计划"，你就还没有开始做游戏。

陷阱 3：先美术后设计

花了三个月设计角色形象、绘制精美的场景图，却还没有一个可玩的原型。

现实：美术是游戏最昂贵的部分之一。在你确定核心玩法有趣之前投入美术资源，就像在确定菜谱之前就把盘子镀了金——方向错了，所有投入都白费。

解药：先灰盒，后美术。用方块和圆圈验证玩法。当你的灰盒原型已经让人觉得"好玩"的时候，再考虑美术方向。

陷阱 4：模仿而非理解

"我喜欢塞尔达，所以我要做一个和塞尔达一样的游戏。"

问题：你复制了塞尔达的表层（开放世界、解谜、战斗），但不理解它的深层设计逻辑（引导式自由探索、化学引擎的涌现性、由密度和视线引导的动线设计）。结果就是一个"看起来像塞尔达但玩起来不像"的东西。

解药：研究你喜欢的游戏的设计决策，而不只是外在特征。问"为什么这么设计"比"设计了什么"更重要。然后把你学到的设计原理应用到你自己独特的概念中。

陷阱 5：忽视测试

"等游戏做完了再测试。"

现实：到那时修改成本已经高得离谱。一个需要在游戏完成后才发现的核心机制问题，可能需要推翻数月的工作。而如果在第一周就做了纸面原型测试，可能只需要改一条规则。

解药：从第一天起就测试。每做出一个新机制，立刻找人玩。不要害怕"太早了"或"太粗糙"——原型的全部意义就在于它可以很粗糙。

最后的话

游戏设计不是天赋，而是技能。和所有技能一样，它可以通过学习、练习和反思来提升。读完这篇教程，你已经具备了游戏设计的基础理论框架。但理论只是起点——真正的学习发生在你动手做的过程中。

你的第一个游戏不需要是杰作。它只需要是完成的。一个完成的平庸作品比一百个未完成的"完美"设想更有价值——因为只有完成才能带来你在理论中学不到的那些经验。

现在就开始做吧。
