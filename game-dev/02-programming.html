01 — 游戏引擎概览

什么是游戏引擎？

游戏引擎是一个可复用的软件框架，它将游戏开发中通用的底层系统抽象出来，让开发者可以专注于游戏逻辑和内容创作，而不必每次都从零搭建渲染管线、物理模拟、音频播放等基础设施。

核心概念

游戏引擎 ≠ 游戏。引擎是工具，游戏是产品。一个好的引擎应该让你更快地做出好游戏，而不是让你花更多时间"学引擎"。

引擎提供的核心系统

渲染系统

2D/3D 图形渲染管线、着色器管理、光照计算、后处理效果、粒子系统。现代引擎通常同时支持 OpenGL/Vulkan/DirectX/Metal。

物理引擎

碰撞检测与响应、刚体/软体模拟、射线检测、关节约束。常见集成：Box2D（2D）、Bullet / PhysX / Jolt（3D）。

音频系统

音效播放、3D 空间音频、音频总线与混音、实时效果（混响、EQ）。部分引擎集成 FMOD / Wwise。

输入系统

键盘、鼠标、手柄、触屏的统一抽象。输入映射、死区处理、振动反馈、多设备热插拔。

资源管理

纹理、模型、音频、场景的加载/卸载/缓存。资源引用、异步加载、打包与压缩。

脚本 / 编辑器

可视化场景编辑器、属性检查器、调试工具、热重载。脚本系统让非程序员也能调整游戏逻辑。

此外，成熟引擎还提供：动画系统（骨骼动画、状态机、IK）、UI 系统、网络层、导航/寻路、本地化、版本控制集成、多平台导出等。

Godot：独立开发者的利器

优势：

  • 完全开源免费，MIT 协议——你的游戏赚多少钱都不用给引擎分一分
  • GDScript 语法类 Python，上手极快；节点系统直觉化
  • 引擎本体仅 ~40MB，启动秒开，资源占用低
  • 2D 拥有独立的渲染管线，不是"3D 引擎的 2D 模式"
  • Godot 4 引入 Vulkan 渲染器，3D 能力大幅提升
  • 社区驱动开发，不受单一公司商业决策影响

劣势：

  • 3D 在大型开放世界场景下仍不及 Unreal
  • 主机导出需要付费第三方方案（W4 Games 提供）
  • 生态相对年轻，高质量插件/资产数量有限
  • 大型团队的工作流支持不如 Unity/Unreal 成熟

推荐场景：独立开发者、2D 游戏、Game Jam、学习游戏开发、对自由度有极高要求的项目。

Unity：最大公约数

优势：

  • 最大的社区和生态系统，几乎任何问题都能搜到答案
  • Asset Store 插件和美术资产数量碾压其他引擎
  • C# 是优秀的工业级语言，IDE 支持完善
  • 移动平台优化做得好，市场占有率极高
  • DOTS/ECS 提供了面向数据设计的高性能路径

劣势：

  • 2023 年 Runtime Fee 事件严重伤害了开发者信任
  • 引擎代码闭源，遇到引擎 Bug 只能等官方修
  • 版本升级经常引入 Breaking Changes
  • 渲染管线碎片化（Built-in / URP / HDRP），选择困难
  • 编辑器启动慢，内存占用大

推荐场景：移动端游戏、中等规模 3D 游戏、需要大量现成资产的项目、团队有 C# 经验。

Unreal Engine 5：工业级重炮

优势：

  • Nanite 虚拟几何体——可以直接导入电影级模型，无需手动减面
  • Lumen 全局光照——实时的、动态的全局光照和反射
  • Blueprints 可视化编程——策划和美术也能写逻辑
  • MetaHuman / Chaos Physics / World Partition 等前沿技术
  • 主机平台支持最好，AAA 工作室首选

劣势：

  • 学习曲线陡峭，C++ 入门门槛高
  • 项目体积大，编译时间长
  • 不适合小型 2D 游戏——用 UE 做小游戏是"杀鸡用牛刀"
  • 收入超 $1M 后 5% 的分成对独立开发者是较大负担
  • 硬件要求高，开发机至少 32GB RAM + 好显卡

推荐场景：3A 或写实风格的 3D 游戏、需要最前沿渲染技术、团队有 C++ 经验、电影级过场动画。

自研引擎：极客之路

什么时候考虑自研：

  • 你的游戏有极其特殊的渲染或架构需求，现有引擎无法满足
  • 你对极致性能有要求（如 Noita 的物理像素模拟）
  • 你的核心目标就是学习底层技术
  • 你在做一个需要长期维护 10+ 年的项目

自研的代价：

  • 80% 的时间在做"引擎"而不是"游戏"
  • 所有 Bug 都是你的 Bug，没有社区帮你
  • 需要深厚的图形学、操作系统、编译器知识
  • 多平台支持需要自己写每个平台的适配层

知名自研案例：id Tech（Doom）、RE Engine（Capcom）、Noita（自研 Falling Sand 引擎）、Celeste（FNA/MonoGame 框架）。

GDScript 深入

GDScript 是 Godot 专门设计的脚本语言。它的语法受 Python 启发，但做了大量游戏开发特化。

GDScript
# GDScript 基本语法示例
extends CharacterBody2D

# 导出变量——在编辑器中可直接调整
@export var speed: float = 200.0
@export var jump_force: float = -400.0

var gravity: float = ProjectSettings.get_setting("physics/2d/default_gravity")

func _physics_process(delta: float) -> void:
    # 应用重力
    if not is_on_floor():
        velocity.y += gravity * delta

    # 跳跃
    if Input.is_action_just_pressed("jump") and is_on_floor():
        velocity.y = jump_force

    # 水平移动
    var direction = Input.get_axis("move_left", "move_right")
    velocity.x = direction * speed

    move_and_slide()

GDScript 的独特优势：

  • 一等公民集成——节点路径、信号、导出变量都是语言级特性，不是"绑定"
  • 热重载——修改脚本后游戏无需重启，实时生效
  • 紧凑——同样逻辑比 C# 少写 40-60% 的代码
  • 类型提示可选——快速原型用动态类型，正式项目用静态类型

C# 在游戏开发中

C# (Unity)
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    [SerializeField] private float speed = 5f;
    [SerializeField] private float jumpForce = 10f;

    private Rigidbody2D rb;
    private bool isGrounded;

    void Start()
    {
        rb = GetComponent<Rigidbody2D>();
    }

    void Update()
    {
        float moveX = Input.GetAxisRaw("Horizontal");
        rb.velocity = new Vector2(moveX * speed, rb.velocity.y);

        if (Input.GetButtonDown("Jump") && isGrounded)
        {
            rb.AddForce(Vector2.up * jumpForce, ForceMode2D.Impulse);
        }
    }
}

C++ 在游戏开发中的核心地位无可替代——几乎所有引擎的底层都是 C/C++ 写的。但直接用 C++ 写游戏逻辑的开发效率远低于 GDScript 或 C#。Unreal 的 Blueprint 就是为了缓解这个问题而生的。

Rust：新兴势力

Rust 在游戏开发中的存在感正在快速上升。Bevy 引擎是最有代表性的 Rust 游戏引擎，采用 ECS 架构。

Rust (Bevy)
use bevy::prelude::*;

// 组件定义
#[derive(Component)]
struct Player;

#[derive(Component)]
struct Speed(f32);

// 系统
fn player_movement(
    keyboard: Res<ButtonInput<KeyCode>>,
    mut query: Query<(&mut Transform, &Speed), With<Player>>,
    time: Res<Time>,
) {
    for (mut transform, speed) in &mut query {
        if keyboard.pressed(KeyCode::ArrowRight) {
            transform.translation.x += speed.0 * time.delta_secs();
        }
    }
}

Rust 的游戏开发优势：内存安全保证、零成本抽象、无 GC 停顿、强大的类型系统、Cargo 包管理。
劣势：学习曲线极陡、编译时间长、生态不成熟、热重载支持弱。

03 — 游戏架构模式

好的架构让代码易于修改。游戏开发的核心挑战是：需求变化极快——今天设计的敌人明天可能要加新行为、换新武器、变成 Boss。好的架构模式能让这些变化的成本最小化。

Entity-Component-System (ECS)

核心思想

组合优于继承。ECS 将游戏对象分解为三个概念：
Entity（实体）= 一个 ID，什么都不是
Component（组件）= 纯数据，没有行为
System（系统）= 纯行为，操作特定组件组合的数据

为什么需要 ECS？传统继承的问题

传统继承的灾难
# 经典的"菱形继承"问题
# 你的游戏有这些东西：
#   - 玩家：能移动、有血量、能攻击
#   - 敌人：能移动、有血量、能攻击
#   - 炮塔：有血量、能攻击、不能移动
#   - 宝箱：有血量、不能攻击、不能移动
#   - 子弹：能移动、能攻击、没血量
#
# 用继承怎么组织？
# GameObject → MovableObject → AttackableMovable → Player?
# 那炮塔放哪？子弹放哪？
# 答案是：继承树无法优雅地表达"特性的任意组合"

ECS 的解法

GDScript 风格伪代码
# ===== 组件：纯数据 =====
class PositionComponent:
    var x: float
    var y: float

class VelocityComponent:
    var vx: float
    var vy: float

class HealthComponent:
    var current: float
    var max_hp: float

class AttackComponent:
    var damage: float
    var cooldown: float

# ===== 实体：组件的组合 =====
# 玩家 = Position + Velocity + Health + Attack
# 炮塔 = Position + Health + Attack (没有 Velocity！)
# 子弹 = Position + Velocity + Attack (没有 Health！)
# 宝箱 = Position + Health (只有这两个！)

# ===== 系统：操作特定组件的逻辑 =====
class MovementSystem:
    # 只处理同时拥有 Position 和 Velocity 的实体
    func update(delta):
        for entity in query(PositionComponent, VelocityComponent):
            entity.position.x += entity.velocity.vx * delta
            entity.position.y += entity.velocity.vy * delta

class DamageSystem:
    # 处理碰撞时的伤害计算
    func on_collision(attacker, target):
        if attacker.has(AttackComponent) and target.has(HealthComponent):
            target.health.current -= attacker.attack.damage

ECS 的关键优势：

  • 缓存友好——同类组件在内存中连续排列，CPU 缓存命中率极高
  • 并行化——不同 System 如果操作不同组件，可以并行执行
  • 灵活组合——新增实体类型只需要组合已有组件，无需改代码
  • 解耦彻底——MovementSystem 完全不知道 DamageSystem 的存在
Godot 中的 ECS

Godot 用的是节点+场景架构，不是传统 ECS，但它的节点系统本身就体现了组合思想。你可以把 Node 看作 Entity，子节点看作 Component。纯 ECS 架构可以看 Bevy（Rust）或 Unity DOTS。

状态机（FSM）与层级状态机（HFSM）

状态机是游戏中最常用的模式之一，特别适合管理角色的行为状态。

GDScript
# ===== 简单有限状态机 =====
extends Node
class_name StateMachine

var current_state: State
var states: Dictionary = {}

func _ready():
    # 将所有子节点注册为状态
    for child in get_children():
        if child is State:
            states[child.name] = child
            child.state_machine = self
    # 初始化第一个状态
    current_state = get_child(0)
    current_state.enter()

func transition_to(state_name: String):
    if not states.has(state_name):
        return
    current_state.exit()
    current_state = states[state_name]
    current_state.enter()

func _process(delta):
    current_state.update(delta)

func _physics_process(delta):
    current_state.physics_update(delta)

GDScript
# ===== 基础状态类 =====
extends Node
class_name State

var state_machine: StateMachine

func enter() -> void:
    pass

func exit() -> void:
    pass

func update(_delta: float) -> void:
    pass

func physics_update(_delta: float) -> void:
    pass

# ===== 具体状态示例：Idle =====
class_name IdleState
extends State

func enter():
    owner.animation_player.play("idle")

func physics_update(delta):
    # 检测输入，决定是否切换状态
    if Input.get_axis("left", "right") != 0:
        state_machine.transition_to("Run")
    if Input.is_action_just_pressed("jump"):
        state_machine.transition_to("Jump")

层级状态机（HFSM）

当状态数量增多时，简单 FSM 会出现"状态爆炸"。HFSM 通过嵌套状态来解决。

概念
# 层级状态机示例结构：
#
# Alive（存活状态）
#   ├── OnGround（在地面）
#   │   ├── Idle（待机）
#   │   ├── Run（跑动）
#   │   └── Attack（攻击）
#   ├── InAir（在空中）
#   │   ├── Jump（跳跃上升）
#   │   ├── Fall（下落）
#   │   └── AirAttack（空中攻击）
#   └── OnWall（在墙上）
#       ├── WallSlide（滑墙）
#       └── WallJump（蹬墙跳）
# Dead（死亡状态）
#
# 好处：Alive 状态的公共逻辑（受伤判定）只写一次
# 而不是在每个叶子状态里都写

观察者模式 / 事件系统

游戏中大量逻辑是"当 X 发生时，做 Y"。观察者模式（在 Godot 中就是信号系统）是解耦这种关系的标准方案。

GDScript
# ===== 事件总线（全局信号）=====
# event_bus.gd — 注册为 AutoLoad
extends Node

# 定义全局信号
signal player_died
signal score_changed(new_score: int)
signal enemy_spawned(enemy: Node2D)
signal item_collected(item_type: String, value: int)
signal level_completed(level_index: int)

# --- 在 Player 脚本中发射信号 ---
func die():
    EventBus.player_died.emit()

func collect_coin():
    score += 10
    EventBus.score_changed.emit(score)

# --- 在 UI 脚本中监听信号 ---
func _ready():
    EventBus.score_changed.connect(_on_score_changed)
    EventBus.player_died.connect(_on_player_died)

func _on_score_changed(new_score: int):
    score_label.text = "Score: %d" % new_score

关键好处：Player 不需要知道 UI 的存在，UI 也不需要直接引用 Player。它们通过事件总线间接通信。

命令模式（Command Pattern）

命令模式将"动作"封装为对象。这在撤销/重做、回放系统、输入缓冲中极其有用。

GDScript
# ===== 命令模式——用于关卡编辑器的撤销/重做 =====
class_name Command

func execute() -> void:
    pass

func undo() -> void:
    pass

# 具体命令：放置方块
class PlaceBlockCommand extends Command:
    var position: Vector2i
    var block_type: int
    var previous_block: int  # 记住之前的值，用于 undo
    var tilemap: TileMap

    func _init(_pos, _type, _tilemap):
        position = _pos
        block_type = _type
        tilemap = _tilemap
        previous_block = tilemap.get_cell_source_id(0, position)

    func execute():
        tilemap.set_cell(0, position, block_type, Vector2i.ZERO)

    func undo():
        if previous_block == -1:
            tilemap.erase_cell(0, position)
        else:
            tilemap.set_cell(0, position, previous_block, Vector2i.ZERO)

# 命令历史管理器
class CommandHistory:
    var history: Array[Command] = []
    var redo_stack: Array[Command] = []

    func execute_command(cmd: Command):
        cmd.execute()
        history.append(cmd)
        redo_stack.clear()  # 新操作清空重做栈

    func undo():
        if history.is_empty():
            return
        var cmd = history.pop_back()
        cmd.undo()
        redo_stack.append(cmd)

    func redo():
        if redo_stack.is_empty():
            return
        var cmd = redo_stack.pop_back()
        cmd.execute()
        history.append(cmd)

对象池模式（Object Pool）

频繁创建和销毁对象（子弹、粒子、敌人）会导致 GC 压力和性能抖动。对象池预先创建一批对象，用完放回池中而不是销毁。

GDScript
# ===== 通用对象池 =====
class_name ObjectPool
extends Node

@export var scene: PackedScene
@export var pool_size: int = 20

var available: Array[Node] = []
var active: Array[Node] = []

func _ready():
    # 预创建对象
    for i in pool_size:
        var obj = scene.instantiate()
        obj.set_process(false)
        obj.visible = false
        add_child(obj)
        available.append(obj)

func acquire() -> Node:
    if available.is_empty():
        print("Pool exhausted! Consider increasing size.")
        return null
    var obj = available.pop_back()
    obj.set_process(true)
    obj.visible = true
    active.append(obj)
    return obj

func release(obj: Node):
    obj.set_process(false)
    obj.visible = false
    active.erase(obj)
    available.append(obj)

# 使用示例
# var bullet = bullet_pool.acquire()
# bullet.global_position = gun.global_position
# bullet.direction = aim_direction
# ...
# 子弹飞出屏幕后: bullet_pool.release(bullet)

04 — 游戏循环的技术实现

游戏循环是所有游戏的心跳。它是一个不断重复的过程：处理输入 → 更新状态 → 渲染画面。听起来简单，但细节决定了游戏运行是否流畅。

处理输入
→
更新逻辑
→
物理模拟
→
渲染画面
→
循环

Delta Time：帧率独立的基石

核心概念

delta（或 dt）是上一帧到当前帧之间经过的秒数。60fps 时约为 0.0167 秒，30fps 时约为 0.0333 秒。乘以 delta 的运动在任何帧率下都会以相同速度运行。

GDScript
# ===== 错误做法：帧率依赖的运动 =====
func _process(_delta):
    position.x += 5  # 每帧移动 5 像素
    # 60fps → 每秒移动 300 像素
    # 30fps → 每秒移动 150 像素
    # 144fps → 每秒移动 720 像素  ← 不同机器表现不同！

# ===== 正确做法：帧率独立的运动 =====
@export var speed = 300.0  # 每秒 300 像素

func _process(delta):
    position.x += speed * delta
    # 60fps → 300 * 0.0167 = 5.0 像素/帧
    # 30fps → 300 * 0.0333 = 10.0 像素/帧
    # 结果：不论帧率，每秒都是 300 像素 ✓

固定时间步 vs 可变时间步

特性 可变时间步 (_process) 固定时间步 (_physics_process)
调用频率 尽可能快，取决于帧率 固定间隔（Godot 默认 60次/秒）
delta 值 每帧不同 始终相同（默认 0.01667）
适合 渲染、动画、UI、视觉效果 物理、碰撞检测、网络同步
确定性 不确定——不同帧率结果不同 确定——可精确回放/同步
卡顿行为 大 delta → 可能穿墙 累积 → 追帧，但不会穿墙 
GDScript
# ===== Godot 的双循环设计 =====

func _process(delta):
    # 可变时间步——用于视觉/非物理逻辑
    sprite.rotation += rotation_speed * delta
    camera.position = camera.position.lerp(target_pos, 0.1)
    update_ui()
    update_particles()

func _physics_process(delta):
    # 固定时间步——用于物理/碰撞/移动
    velocity.y += gravity * delta
    move_and_slide()
    check_attack_collision()

物理帧 vs 渲染帧的脱耦

在现代引擎中，物理更新和渲染更新是解耦的。这意味着：

时间线示意
# 假设物理 60Hz，渲染 144Hz
#
# 时间 →
# 物理:  |---P---|---P---|---P---|---P---|
# 渲染:  |-R-|-R-|-R-|-R-|-R-|-R-|-R-|-R-|-R-|
#
# 渲染帧之间，物理状态可能没有更新
# 解决方案：物理插值（Physics Interpolation）
#
# Godot 4 中启用：
# Project Settings → Physics → Common → Physics Interpolation = true
# 或代码中：
# ProjectSettings.set_setting(
#     "physics/common/physics_interpolation", true
# )
最佳实践
  • 移动/物理放 _physics_process——确保碰撞检测正确
  • 动画/视觉放 _process——尽可能流畅
  • 始终使用 delta 乘以速度值
  • 开启物理插值避免低物理帧率下的视觉抖动
  • 不要在 _process 里直接修改物理体的 velocity

05 — 物理引擎原理

碰撞检测算法

碰撞检测是物理引擎最核心的职责。它分为两个阶段：

宽相检测
Broad Phase
→
窄相检测
Narrow Phase
→
碰撞响应
Response

宽相检测用简单的包围盒快速排除不可能碰撞的对象对，把 O(n^2) 的问题缩小到一个小子集。窄相检测对候选对进行精确的几何碰撞测试。

AABB（轴对齐包围盒）

最简单的碰撞检测。只需要比较矩形的边界——不涉及旋转。

伪代码
# AABB 碰撞检测——两个矩形是否重叠
func aabb_overlap(a: Rect2, b: Rect2) -> bool:
    return (
        a.position.x < b.position.x + b.size.x and
        a.position.x + a.size.x > b.position.x and
        a.position.y < b.position.y + b.size.y and
        a.position.y + a.size.y > b.position.y
    )

# 原理：如果两个矩形在 X 轴和 Y 轴上都有重叠，则碰撞
# 时间复杂度: O(1) 每对
# 适用: 宽相检测、不旋转的矩形碰撞体

SAT（分离轴定理）

SAT 可以检测任意凸多边形的碰撞。核心思想：如果两个凸形状不碰撞，那么一定存在一条轴，将它们的投影分开。

伪代码
# SAT 碰撞检测核心逻辑
func sat_overlap(poly_a, poly_b) -> bool:
    # 获取所有需要检查的轴（两个多边形的所有边的法线）
    var axes = get_normals(poly_a) + get_normals(poly_b)

    for axis in axes:
        # 将两个多边形投影到这条轴上
        var proj_a = project_polygon(poly_a, axis)
        var proj_b = project_polygon(poly_b, axis)

        # 如果投影不重叠——找到了分离轴，不碰撞
        if not projections_overlap(proj_a, proj_b):
            return false

    # 所有轴上都重叠——碰撞！
    return true

func get_normals(polygon) -> Array:
    var normals = []
    for i in polygon.size():
        var edge = polygon[(i+1) % polygon.size()] - polygon[i]
        # 法线 = 将边向量旋转 90 度
        normals.append(Vector2(-edge.y, edge.x).normalized())
    return normals

GJK（Gilbert-Johnson-Keerthi）

GJK 是更高级的碰撞检测算法，基于闵可夫斯基差（Minkowski Difference）。如果两个形状的闵可夫斯基差包含原点，则碰撞。

GJK vs SAT 对比

特性SATGJK
适用形状凸多边形任意凸形状（含圆、胶囊体）
性能多边形顶点少时快迭代算法，收敛快
碰撞信息直接得到穿透深度和法线需要 EPA 算法补充
实现难度中等较高
在引擎中Box2D 使用Bullet / PhysX 使用

刚体物理

刚体物理模拟物体在力的作用下的运动。核心是牛顿运动定律的离散化。

伪代码
# 刚体物理的基本更新步骤
func physics_step(body, dt):
    # 1. 累加所有外力（重力、推力、摩擦...）
    var total_force = Vector2.ZERO
    total_force += Vector2(0, GRAVITY) * body.mass   # 重力
    total_force += body.applied_force                # 外部施加的力
    total_force += -body.velocity * body.drag        # 阻力

    # 2. F = ma → a = F/m
    var acceleration = total_force / body.mass

    # 3. 半隐式欧拉积分（Semi-implicit Euler）
    body.velocity += acceleration * dt        # 先更新速度
    body.position += body.velocity * dt       # 再用新速度更新位置

    # 4. 清除本帧施加的力
    body.applied_force = Vector2.ZERO

触发器 vs 碰撞体

这是初学者常常混淆的两个概念。

碰撞体（Collider）

  • 产生物理碰撞响应——物体会弹开
  • 用于墙壁、地面、实体障碍物
  • Godot: StaticBody2D, CharacterBody2D, RigidBody2D
  • 有质量、摩擦力、弹性系数

触发器 / 区域（Trigger / Area）

  • 检测重叠但不阻挡物体通过
  • 用于拾取物、伤害区域、触发剧情
  • Godot: Area2D
  • 通过信号通知进入/退出
GDScript
# ===== 使用 Area2D 作为拾取触发器 =====
extends Area2D

@export var heal_amount: int = 25

func _ready():
    # 当有物体进入此区域时触发
    body_entered.connect(_on_body_entered)

func _on_body_entered(body: Node2D):
    if body.is_in_group("player"):
        body.heal(heal_amount)
        queue_free()  # 拾取后销毁自己

物理引擎最佳实践

实践指南
  1. 永远不要在 _process 中移动物理体——用 _physics_process
  2. 使用碰撞层和掩码——让子弹只和敌人碰撞，不和其他子弹碰撞
  3. 简化碰撞形状——用简单的矩形/圆形代替复杂的多边形轮廓
  4. 避免缩放碰撞形状——会产生意外的物理行为，直接改形状大小
  5. RigidBody 不要直接设 position——用 apply_force() 或 apply_impulse()
  6. 高速物体用 CCD——连续碰撞检测防止"子弹穿墙"
  7. 射线检测代替碰撞——命中扫描武器用 Raycast，不用实体子弹

06 — 游戏 AI 编程

游戏 AI 不追求"真正的智能"，而是追求让玩家觉得好玩的行为。一个"太聪明"的 AI 可能让游戏无趣，一个"看起来聪明但有弱点"的 AI 才是好设计。

有限状态机 AI

最经典的 AI 实现方式。每个敌人在有限个状态之间切换。

GDScript
# ===== 敌人 AI 状态机 =====
extends CharacterBody2D

enum AIState { IDLE, PATROL, CHASE, ATTACK, FLEE }

var state: AIState = AIState.PATROL
var player: Node2D

@export var detection_range = 200.0
@export var attack_range = 50.0
@export var flee_hp_threshold = 20.0
@export var patrol_speed = 60.0
@export var chase_speed = 120.0

func _physics_process(delta):
    match state:
        AIState.PATROL:
            do_patrol(delta)
            if can_see_player():
                state = AIState.CHASE

        AIState.CHASE:
            do_chase(delta)
            if distance_to_player() < attack_range:
                state = AIState.ATTACK
            elif distance_to_player() > detection_range * 1.5:
                state = AIState.PATROL
            if health < flee_hp_threshold:
                state = AIState.FLEE

        AIState.ATTACK:
            do_attack()
            if distance_to_player() > attack_range * 1.2:
                state = AIState.CHASE

        AIState.FLEE:
            do_flee(delta)

func can_see_player() -> bool:
    if not player:
        return false
    var dist = global_position.distance_to(player.global_position)
    if dist > detection_range:
        return false
    # 射线检测——确保没有墙挡住
    var space = get_world_2d().direct_space_state
    var query = PhysicsRayQueryParameters2D.create(
        global_position, player.global_position
    )
    var result = space.intersect_ray(query)
    return result and result.collider == player

行为树（Behavior Tree）

行为树比状态机更灵活、更可扩展。它是 AAA 游戏 AI 的标准方案。

行为树核心节点类型

Sequence（顺序）：依次执行子节点，全部成功则成功，任一失败则失败（类似 AND）
Selector（选择）：依次尝试子节点，任一成功则成功，全部失败则失败（类似 OR）
Decorator（装饰器）：修改子节点的行为（取反、重复、条件守卫）
Leaf（叶子）：执行具体动作或检查条件

行为树结构示例
# 一个巡逻兵的行为树
#
# Selector (根)
# ├── Sequence [战斗]
# │   ├── Condition: 看见玩家？
# │   ├── Selector [战斗策略]
# │   │   ├── Sequence [近战]
# │   │   │   ├── Condition: 距离 < 攻击范围？
# │   │   │   └── Action: 攻击
# │   │   └── Action: 追击玩家
# │   └── Decorator [血量低时]
# │       ├── Condition: HP < 20%？
# │       └── Action: 逃跑
# └── Sequence [巡逻]
#     ├── Action: 移动到巡逻点
#     ├── Action: 等待 2 秒
#     └── Action: 切换到下一个巡逻点
GDScript
# ===== 简化的行为树实现 =====
class_name BTNode

enum Status { SUCCESS, FAILURE, RUNNING }

func tick(actor, delta) -> Status:
    return Status.FAILURE

# Sequence 节点
class BTSequence extends BTNode:
    var children: Array[BTNode] = []

    func tick(actor, delta) -> Status:
        for child in children:
            var result = child.tick(actor, delta)
            if result != Status.SUCCESS:
                return result  # FAILURE 或 RUNNING
        return Status.SUCCESS

# Selector 节点
class BTSelector extends BTNode:
    var children: Array[BTNode] = []

    func tick(actor, delta) -> Status:
        for child in children:
            var result = child.tick(actor, delta)
            if result != Status.FAILURE:
                return result  # SUCCESS 或 RUNNING
        return Status.FAILURE

# 条件叶子节点示例
class CanSeePlayer extends BTNode:
    func tick(actor, delta) -> Status:
        if actor.can_see_player():
            return Status.SUCCESS
        return Status.FAILURE

# 动作叶子节点示例
class ChasePlayer extends BTNode:
    func tick(actor, delta) -> Status:
        var dir = (actor.player.position - actor.position).normalized()
        actor.velocity = dir * actor.chase_speed
        actor.move_and_slide()
        return Status.RUNNING

A* 寻路算法

A* 是游戏中最常用的寻路算法。它在 Dijkstra 的基础上加入了启发式估计，大幅减少搜索范围。

伪代码
# ===== A* 寻路核心算法 =====
func astar(start, goal, grid):
    # open_set: 待探索的节点（优先队列，按 f_score 排序）
    var open_set = PriorityQueue()
    open_set.push(start, 0)

    # g_score: 从起点到该节点的实际代价
    var g_score = { start: 0 }

    # came_from: 路径回溯用
    var came_from = {}

    while not open_set.is_empty():
        var current = open_set.pop()

        # 到达目标！
        if current == goal:
            return reconstruct_path(came_from, current)

        for neighbor in get_neighbors(current, grid):
            # 经过 current 到达 neighbor 的代价
            var tentative_g = g_score[current] + cost(current, neighbor)

            if tentative_g < g_score.get(neighbor, INF):
                came_from[neighbor] = current
                g_score[neighbor] = tentative_g

                # f_score = g_score + h(启发式估计)
                var f_score = tentative_g + heuristic(neighbor, goal)
                open_set.push(neighbor, f_score)

    return []  # 无路径

# 启发式函数——曼哈顿距离（网格寻路）
func heuristic(a, b) -> float:
    return abs(a.x - b.x) + abs(a.y - b.y)

A* 的关键要素

  • f(n) = g(n) + h(n) —— 总估计代价 = 实际代价 + 启发式估计
  • 启发式要可容许（admissible）——不能高估实际距离，否则可能找到非最优路径
  • 常见启发式：曼哈顿距离（4方向网格）、欧几里得距离（任意方向）、切比雪夫距离（8方向网格）
  • Godot 内置 AStarGrid2D——直接用，不需要自己实现

导航网格（NavMesh）

对于非网格的、连续空间的 3D/2D 场景，NavMesh 比 A* 网格更高效。它将可行走区域划分为凸多边形，然后在多边形之间寻路。

GDScript
# ===== 在 Godot 中使用 NavigationAgent2D =====

@export var speed = 150.0

        return

    # 获取下一个路径点
    var next_pos = nav_agent.get_next_path_position()
    var direction = (next_pos - global_position).normalized()

    velocity = direction * speed
    move_and_slide()

07 — 网络编程基础

网络多人游戏是游戏开发中最复杂的领域之一。它不仅涉及编程技术，还涉及分布式系统、网络工程、安全性。

架构模式

客户端-服务器（Client-Server）

一个权威服务器运行游戏逻辑，客户端发送输入、接收状态更新。

  • 服务器是"真相的唯一来源"
  • 反作弊天然优势——客户端无法篡改游戏状态
  • 需要运维服务器成本
  • 延迟 = 客户端到服务器的往返时间

代表：几乎所有竞技类多人游戏（CS2、Valorant、LOL）

点对点（Peer-to-Peer）

所有玩家直接互联，每个客户端既是服务器也是客户端。

  • 无需专用服务器——降低成本
  • 延迟取决于玩家之间的网络距离
  • 同步问题更复杂
  • 反作弊困难——每个客户端都有完整游戏状态

代表：格斗游戏（GGPO回滚式）、部分 RTS

状态同步 vs 帧同步

维度 状态同步 帧同步（Lockstep）
传输内容 游戏对象的状态（位置、血量...） 玩家的输入指令
带宽需求 随实体数量增长 仅与玩家数量相关，极低
确定性要求 不需要——服务器是权威 必须完全确定性（同输入 = 同结果）
断线重连 容易——发送当前状态快照 困难——需要从头快进所有历史输入
适合类型 FPS、MMO、大部分动作游戏 RTS（星际争霸）、格斗游戏、MOBA
实现难度 中等 高（需要确定性物理、固定点数等）

延迟补偿技术

网络延迟是多人游戏的头号敌人。以下是对抗延迟的核心技术。

客户端预测（Client-side Prediction）

伪代码
# ===== 客户端预测 =====
# 问题：玩家按下"前进"，要等服务器确认才能动？太卡了！
# 解决：客户端立即执行移动，同时发送给服务器

var pending_inputs = []  # 已发送但未被服务器确认的输入
var input_sequence = 0

func client_update(delta):
    var input = get_player_input()
    input.sequence = input_sequence
    input_sequence += 1

    # 1. 立即在本地执行（预测）
    apply_input(input, delta)

    # 2. 发送给服务器
    send_to_server(input)

    # 3. 保存到待确认队列
    pending_inputs.append(input)

func on_server_state_received(server_state):
    # 服务器发来了权威状态
    position = server_state.position

    # 丢弃已被服务器处理的输入
    pending_inputs = pending_inputs.filter(
        func(input): return input.sequence > server_state.last_processed_input
    )

    # 重新应用尚未确认的输入（和解/Reconciliation）
    for input in pending_inputs:
        apply_input(input, FIXED_DELTA)

实体插值（Entity Interpolation）

伪代码
# ===== 实体插值 =====
# 问题：其他玩家的位置更新是离散的，每 50ms 收到一次
# 直接设置位置 → 瞬移/抖动
# 解决：在两个已知状态之间平滑插值

var state_buffer = []  # 存储最近收到的服务器状态
var interpolation_delay = 0.1  # 100ms 延迟换取平滑

func render_other_player():
    # 渲染时间 = 当前时间 - 插值延迟
    var render_time = current_time - interpolation_delay

    # 找到 render_time 前后的两个状态
    var before = find_state_before(render_time)
    var after = find_state_after(render_time)

    if before and after:
        # 在两个状态之间线性插值
        var t = (render_time - before.timestamp) / (after.timestamp - before.timestamp)
        display_position = before.position.lerp(after.position, t)

回滚式网络代码（Rollback Netcode）

GGPO / 回滚系统

格斗游戏和快节奏 P2P 游戏的黄金标准。核心思想：

  1. 预测——假设对方重复上一帧的输入
  2. 当收到实际输入时——如果预测正确，什么都不用做
  3. 如果预测错误——回滚到上一个确认状态，用正确输入重新模拟到当前帧

关键要求：游戏状态必须可序列化（保存/加载）、物理必须确定性、回滚+重模拟必须在一帧内完成。

网络架构选择指南

推荐方案
  • 合作 PvE（2-4人）→ 主机制 P2P（一个玩家当 host），状态同步
  • 竞技 FPS→ 专用服务器 + 客户端预测 + 服务器和解 + 实体插值
  • 格斗游戏→ P2P + 回滚式帧同步（GGPO）
  • RTS→ 帧同步（所有客户端运行确定性模拟）
  • MMO→ 分区服务器集群 + 状态同步 + 兴趣管理
  • 独立开发者首选→ Godot 内置的 MultiplayerPeer + ENet，从简单的 P2P 开始

空间分区数据结构

当场景中有大量物体时，"每个物体和其他所有物体检测碰撞"是 O(n^2) 的灾难。空间分区将空间划分为区域，只检测同一区域或相邻区域内的物体。

四叉树（Quadtree）——2D 空间分区

伪代码
# ===== 四叉树——2D 空间分区 =====
class QuadTree:
    var boundary: Rect2        # 本节点覆盖的区域
    var capacity: int = 4      # 分裂前最多容纳的对象数
    var objects: Array = []
    var divided: bool = false
    var nw: QuadTree            # 四个子节点
    var ne: QuadTree
    var sw: QuadTree
    var se: QuadTree

    func insert(obj):
        if not boundary.has_point(obj.position):
            return false

        if objects.size() < capacity:
            objects.append(obj)
            return true

        if not divided:
            subdivide()

        return (nw.insert(obj) or ne.insert(obj) or
                sw.insert(obj) or se.insert(obj))

    func query(range: Rect2) -> Array:
        var found = []

        if not boundary.intersects(range):
            return found

        for obj in objects:
            if range.has_point(obj.position):
                found.append(obj)

        if divided:
            found += nw.query(range)
            found += ne.query(range)
            found += sw.query(range)
            found += se.query(range)

        return found

# 使用：
# 每帧重建四叉树，然后对每个对象只查询周围区域
# 从 O(n²) 降为 O(n log n)

八叉树（Octree）与 BVH

八叉树（3D 版四叉树）

将 3D 空间递归切分为 8 个子立方体。适合均匀分布的物体。Minecraft 的区块系统本质上是一种空间分区。

BVH（层级包围盒）

以物体为中心构建树，而不是以空间为中心。对不均匀分布的物体更高效。现代物理引擎（Box2D、Bullet）的宽相检测首选。

渲染优化技术

LOD（Level of Detail）

概念
# LOD 策略：物体离摄像机越远，使用越简单的模型
#
# LOD 0 (近): 10000 三角面，完整纹理，法线贴图
# LOD 1 (中): 3000 三角面，简化纹理
# LOD 2 (远): 500 三角面，低分辨率纹理
# LOD 3 (极远): 100 三角面 或 Billboard（面片）
# 超出距离: 不渲染
#
# UE5 的 Nanite 自动处理 LOD——你导入电影级模型，
# 引擎实时流式加载合适精度的几何体

遮挡剔除（Occlusion Culling）

被其他物体完全挡住的物体不需要渲染。引擎可以通过预计算或实时检测来跳过这些物体。

  • 视锥剔除（Frustum Culling）——不在摄像机视野内的物体不渲染（所有引擎自动做）
  • 遮挡剔除——被墙/大建筑挡住的物体不渲染（需要配置或手动设置遮挡体）
  • 距离剔除——超出一定距离的物体不渲染

批处理（Batching）

Draw Call 优化

每次 CPU 告诉 GPU "画这个东西"就是一个 Draw Call。Draw Call 本身的开销 > 实际绘制的开销。所以 1000 个小 Draw Call 远比 1 个大 Draw Call 慢。

批处理将多个使用相同材质的物体合并为一次 Draw Call。在 Godot 中，使用相同材质的 CanvasItem 会自动批处理。在 3D 中，可以用 MultiMeshInstance3D 来渲染大量相同模型（草、树、石头）。

性能优化 Checklist

优化检查清单
  • 先让游戏跑起来，再优化。过早优化是最大的坑
  • 用 Profiler 找真正的瓶颈。不要凭直觉猜
  • 使用对象池处理频繁创建/销毁的对象（子弹、粒子、敌人）
  • 减少 _process 中的工作——不是每帧都需要更新所有逻辑
  • 使用信号/事件代替轮询（polling）
  • 空间分区——四叉树/八叉树用于大量实体的碰撞检测
  • 合理使用 LOD——远处的物体不需要高精度
  • 合批渲染——相同材质的物体尽量合并
  • 避免在 GDScript 热路径中做复杂计算——性能关键代码用 C++ GDExtension
  • 纹理使用合适的压缩格式和尺寸
  • 使用 MultiMeshInstance 渲染大量相同模型
  • 物理——简化碰撞形状、合理设置碰撞层

代码原则

KISS（Keep It Simple）

不要过度设计。一个简单的 if-else 可能比精心设计的设计模式更好。先让它工作，再让它优雅。

单一职责

一个脚本做一件事。Player.gd 不应该同时处理移动、攻击、UI、存档。把它们拆成组件。

数据驱动

武器伤害、敌人血量、关卡配置用 Resource 文件存储，不要硬编码在脚本里。策划可以直接在编辑器里调。

防御性编程

检查 null、验证数组边界、用 assert() 标注前提条件。你未来的自己会感谢你。

你的第一个项目该怎么选？

新手最常犯的错误

第一个项目是"开放世界 RPG"或"多人在线 Battle Royale"。这些项目的工作量是你预估的 10-50 倍。你会在"80% 完成"的状态下放弃，因为最后 20% 花了 80% 的时间。从能在 2-4 周内完成的项目开始。

第一个项目推荐路径

  1. Pong（1-2 天）——学会游戏循环、输入处理、碰撞检测
  2. 打砖块 / Breakout（3-5 天）——关卡设计、更复杂的碰撞、得分系统
  3. 2D 平台跳跃（1-2 周）——角色控制器、地图编辑、敌人 AI、动画
  4. Top-down Shooter / Roguelike（2-4 周）——程序生成、多种武器、存档
  5. 你自己的创意项目——但范围要控制在 1-3 个月

关键：每个项目都要完成到"可发布"的程度——有主菜单、有结算画面、有音效。完成一个小游戏比做一半大游戏有价值一万倍。
