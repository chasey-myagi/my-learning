Part 01

系统设计思维

什么是游戏系统

游戏系统是一组相互关联的规则和机制，它们共同作用产生游戏体验。系统不是单个功能，而是功能之间的关系网络。一个孤立的"攻击"动作不是系统；攻击 + 防御 + 生命值 + 伤害计算 + 属性克制组合在一起，才构成了一个战斗系统。

理解游戏系统的关键在于：整体大于部分之和。系统中各元素的交互会产生单个元素无法产生的行为和体验。

涌现性（Emergence）与复杂性

涌现性是系统设计的圣杯——简单规则交互产生复杂、不可预测但有意义的行为。涌现性使游戏超越设计者的预想，创造出无限可能的玩法空间。

🧬
简单规则 → 复杂行为

《我的世界》只有放置/破坏方块 + 合成 + 物理模拟几条基本规则，却涌现出红石计算机、自动化农场、像素艺术等设计者从未预想的玩法。

🔥
元素交互涌现

《塞尔达：旷野之息》的物理/化学引擎让火 + 风 = 火旋风，雷 + 金属 = 导电，水 + 寒冷 = 冰面。每种组合创造新战术。

🎭
AI 行为涌现

《RimWorld》的殖民者有个性、社交需求、技能偏好，这些简单属性组合后产生了戏剧性的故事——背叛、恋爱、精神崩溃。

设计心法

涌现性的关键不是规则的数量，而是规则之间的正交性（orthogonality）。如果 A 规则和 B 规则可以独立运作也可以组合，那就有涌现的空间。如果 A 依赖 B 才能工作，那只是一条更复杂的规则。

复杂性（Complexity）vs 深度（Depth）

这是游戏设计中最关键的区分之一：

维度复杂性（Complexity）深度（Depth）
定义玩家需要理解的规则数量有意义决策的数量和质量
玩家感受学习成本高、信息过载容易上手但不断发现新策略
典型表现大量独立系统、特殊情况多少量规则产生丰富交互
好例子《矮人要塞》（高复杂+高深度）围棋（低复杂+高深度）
坏例子许多手游（高复杂+低深度）井字棋（低复杂+低深度）

设计目标：最大化深度/复杂性比率。用最少的规则创造最多有意义的决策。

系统耦合与解耦

系统之间的连接方式决定了游戏的可维护性、可扩展性和涌现性。

紧耦合 谨慎使用

系统 A 直接调用/依赖系统 B 的内部状态。修改 B 必然影响 A。

// 紧耦合：战斗直接读取装备
damage = player.weapon.baseDamage
       * player.strength
       + player.weapon.enchant.bonus

问题：武器系统改动 → 战斗系统崩溃

松耦合 推荐

系统通过统一接口/事件通信，不关心对方内部实现。

// 松耦合：通过统计值接口
damage = stats.get("attack_power")
// attack_power 由装备/buff/技能等
// 各系统独立贡献，战斗系统不关心来源

优势：各系统独立开发、测试、修改

正反馈与负反馈循环

反馈循环是系统自我调节的核心机制，决定了游戏是趋向稳定还是趋向极端。

正反馈循环（Positive Feedback）

输出增强输入，"强者更强，弱者更弱"。制造加速和雪球效应。

→

↻

正反馈的作用：缩短游戏时间、制造高潮、奖赏领先者。但不受控的正反馈会让落后者绝望。

负反馈循环（Negative Feedback）

输出抑制输入，"领先者受阻，落后者获助"。制造平衡和拉锯。

领先者

→
对手追赶

↻

负反馈的作用：保持竞争悬念、延长游戏时间、给落后者希望。但过强的负反馈会让努力无意义。

设计要点

好的游戏混合使用正反馈和负反馈。通常在短期使用正反馈（奖赏好的操作），在长期/全局使用负反馈（保持竞争平衡）。例如：《马里奥赛车》的道具系统——落后的玩家获得更强道具（负反馈），但领先者的操作技术优势仍然有效（正反馈）。

系统思维案例分析

🏛
《文明》系列 —— 多维系统交织

《文明》是系统设计的教科书。它展示了如何让多个看似独立的系统深度交织：

  • 科技树 → 军事单位 → 战争 → 领土 → 资源 → 科技研究速度：完美的循环依赖
  • 文化系统作为负反馈：文化弱的文明城市会被"文化翻转"
  • 幸福度系统限制扩张：城市越多 → 人民越不满 → 生产力下降（负反馈）
  • 多胜利条件创造路径多样性：科技胜利、文化胜利、征服胜利、外交胜利互相制约

核心设计智慧：每个系统都是其他系统的输入或约束，没有孤立存在的系统。

🏭
《Factorio》 —— 自动化与涌现

《Factorio》展示了纯系统涌现的魅力：

  • 核心规则极简：采矿 → 运输 → 加工 → 组装。只有四个动词
  • 复杂性由玩家创造：合成配方树的深度逼迫玩家构建越来越复杂的工厂
  • 正反馈驱动：自动化 → 产量提升 → 可以自动化更多流程 → 指数增长
  • 系统约束制造挑战：污染 → 虫群进攻 → 需要防御 → 消耗资源 → 需要更多自动化
  • 物流问题涌现：游戏没有"设计物流"的规则，但传送带的物理限制自然产生了物流瓶颈
🎪
《RimWorld》 —— AI 故事生成器

《RimWorld》将系统设计推向了叙事涌现的极致：

  • AI 故事讲述者：不是设计关卡，而是设计"节奏曲线"——卡桑德拉稳步升级，费比随机波动，兰迪完全混乱
  • 殖民者系统：每个角色有 50+ 属性/特质，组合产生独特个性和冲突
  • 需求系统：食物、休息、娱乐、社交、美观度等需求交织，不满积累可能导致精神崩溃
  • 社交系统涌现：好感度 + 共事 + 性格相容性 → 自然发展出恋爱、仇恨、心理创伤等"剧情"
  • Mod 证明系统设计的成功：松耦合架构让社区创造了数千个 Mod，不断扩展系统的涌现空间
Part 02

经济系统设计

资源类型分类

游戏经济的基础是资源设计。资源可以按多个维度分类：

分类维度类型特征示例
持久性可消耗使用后消失药水、弹药、食物
永久获取后永久持有解锁的技能、成就
时间型随时间产生/衰减体力值、耐久度、士气
获取方式可交易玩家间可流通金币、材料、装备
绑定不可转移荣誉点、灵魂绑定装备
产出系统自动产生每日收益、被动收入
稀缺性软上限获取变难但无天花板经验值、金币
硬上限有明确数量限制背包格数、技能点

水龙头与水槽模型（Faucet-Drain）

这是经济系统设计中最核心的概念框架。把经济想象成一个水箱：

水龙头
（资源来源）
任务奖励/击杀掉落
日常收益/出售物品
→
玩家资产池
当前持有的所有资源
→
水槽
（资源消耗）
升级消耗/购买商品
修理费用/税收
核心平衡公式
资产变化率 = Σ(水龙头流入) - Σ(水槽流出)
当变化率 > 0 → 通胀（资源贬值）；当变化率 < 0 → 通缩（资源升值）

常见水龙头设计

  • 击杀掉落：最基础的来源，需要根据怪物难度精确调控掉落量
  • 任务奖励：一次性大额注入，用于关键进度节点
  • 日常/周常：稳定流入，构成玩家"收入"的基础
  • 交易利润：在有玩家交易的游戏中，低买高卖产生的财富（零和，不增加总量）
  • 系统产出：基地收入、投资回报等自动化收入

常见水槽设计

  • 装备强化/升级：最大的资源消耗池，通常伴随概率机制
  • 消耗品购买：药水、弹药等持续性消耗
  • 修理/维护费：对装备使用征收"使用税"
  • 交易税：每次交易抽取一定比例，持续从系统中移除货币
  • 死亡惩罚：掉落金币/经验，直接从系统中移除资源
  • 外观/装饰：纯消费性支出，最健康的水槽类型

通货膨胀控制

通胀是在线游戏经济的头号敌人。随着玩家群体积累财富，物价飙升，新玩家面临巨大经济壁垒。

通胀控制策略

  1. 绑定机制：装备/材料获取后不可交易，阻断财富向老玩家集中
  2. 强力水槽：设计吸引力极高的消费内容（限量外观、稀有坐骑），大量回收货币
  3. 通货衰减：持有大量货币时征收存储费/税（模拟现实中的通胀税）
  4. 赛季重置：定期清档或开新服，从根本上解决经济积累问题
  5. 动态定价：NPC 商品价格随服务器总货币量动态调整
  6. 差异化货币：不同内容使用不同货币，分散通胀压力

交易系统

玩家间交易是最复杂的经济子系统，设计不当会摧毁整个经济。

拍卖行模式

特点：集中式、透明价格、高效匹配

优势：价格发现准确，交易效率高

风险：容易被脚本/工作室操控，价格波动大

自由交易模式

特点：面对面交易、社交驱动

优势：增强社交、允许讨价还价

风险：诈骗频发、RMT 泛滥、不透明

限制交易模式

特点：只能交易特定物品/货币

优势：保护核心进度不被交易破坏

风险：地下交易（如代打、账号交易）

货币设计

货币是经济系统的血液。大多数成功的游戏采用多货币体系：

货币类型获取方式用途设计目的
基础货币（金币）所有活动广泛产出日常消耗、基础交易通用润滑剂
高级货币（钻石）充值、稀有活动加速、外观、便利商业变现
竞技货币（荣誉）PvP 活动竞技专属奖励驱动特定行为
赛季货币限时活动赛季专属内容限时紧迫感
社交货币（友情点）好友互动社交专属奖励促进社交
货币设计陷阱

货币过多会导致认知负担——玩家不理解每种货币的价值和用途。经验法则：核心货币不超过 3 种，辅助货币不超过 5 种。如果你发现自己在设计第 10 种货币，说明系统需要简化。

经济模拟与测试

经济系统上线前必须经过严格的模拟测试：

  1. 蒙特卡洛模拟：模拟数千个不同行为模式的玩家，运行数月游戏时间，观察经济走势
  2. 极端行为测试：模拟"只打怪不消费"的储蓄型玩家和"能买就买"的消费型玩家
  3. 多人博弈测试：模拟玩家间交易行为，检测套利空间
  4. 时间加速测试：观察经济在 1 年、2 年后的健康度
# 简单经济模拟伪代码
class EconomySim:
    def simulate(self, days=365, players=1000):
        for day in range(days):
            for player in self.players:
                # 每日收入（水龙头）
                income = self.calc_daily_income(player)
                player.gold += income

                # 每日支出（水槽）
                expense = self.calc_daily_expense(player)
                player.gold -= expense

                # 记录经济指标
                self.metrics.record(day, player)

            # 全局统计
            avg_gold = mean(p.gold for p in self.players)
            gini = self.calc_gini_coefficient()
            price_index = self.calc_price_index()

            if gini > 0.6:  # 贫富差距警告
                print(f"Day {day}: 基尼系数过高 {gini:.2f}")
            if price_index > 200:  # 物价翻倍警告
                print(f"Day {day}: 通胀严重, CPI={price_index}")
Part 03

进度系统

经验值与等级

等级系统是最古老也最常见的进度框架。核心设计问题：每升一级需要多少经验？

常用升级经验公式
EXP(n) = base × nexponent
n = 当前等级, base = 基础经验, exponent = 增长因子（通常 1.5 ~ 2.5）

升级经验计算器

等级系统设计要点

  • 前期快速升级：给新玩家正反馈，让他们在前 30 分钟至少升 3-5 级
  • 中期稳定节奏：每级需要的时间相对稳定（30分钟 ~ 2小时）
  • 后期显著放缓：最后 10% 的等级可能需要总时间的 50%
  • 等级上限：明确的终点给玩家目标感，但要有满级后的内容
Lv.1 → Lv.10（初期）~2 小时
Lv.10 → Lv.30（中期）~20 小时
Lv.30 → Lv.50（后期）~80 小时

进度条表示升级速度（越长越快）。后期虽然时间更长但内容也更丰富。

技能树设计模式

线性树 简单

A → B → C → D。每个技能只有一个前置。简单明了，但缺乏选择深度。适合动作游戏。

分支树 经典

A → B 或 C → D 或 E。关键节点提供互斥选择。WoW 天赋树的经典模式。

网状树 复杂

任意节点可连接多个前置。《流放之路》的星座盘。构建空间巨大但难以平衡。

模块树 现代

独立模块可自由组合。《杀戮尖塔》的卡牌构建。灵活性最高，涌现性最强。

解锁系统

解锁系统控制玩家何时获得新能力/内容。好的解锁节奏让玩家始终有新东西可期待。

解锁节奏黄金法则

玩家在任何时刻都应该能看到"下一个解锁"在不远处。解锁间隔不应超过 15-20 分钟（移动端）或 30-60 分钟（主机/PC）。每次解锁应该改变玩法而不只是数值提升。

成就系统

成就系统是对玩家行为的外部认可，同时引导玩家探索他们可能不会自发尝试的内容。

成就设计分类

类型描述示例设计目的
进度型完成主要内容击败第一个Boss标记里程碑
探索型发现隐藏内容找到所有秘密区域鼓励探索
挑战型完成高难度任务无伤通关为硬核玩家提供目标
社交型与其他玩家互动组队完成副本促进社交
创意型用非常规方式完成用炸鱼击败Boss鼓励实验
收集型收集全部某类物品集齐所有坐骑延长游戏寿命

Meta-progression（永久进度）

Meta-progression 是 Roguelike 游戏的核心创新——死亡不是终点，而是另一种形式的进步。

开始一局
→
游戏过程
获取临时能力
→
死亡
→
获得 Meta 货币
→
永久升级
解锁新能力/角色
↻

经典案例：

  • 《Hades》：暗黑之镜提供永久属性提升，钥匙解锁新武器，对话推进剧情——死亡 = 更多故事
  • 《杀戮尖塔》：每局解锁新卡牌加入卡池，20 级攀升解锁全部卡牌后才是真正的开始
  • 《以撒的结合》：击败 Boss 解锁新道具进入道具池，数百次通关才能看到全部内容
设计心法

Meta-progression 的精髓是"不是变强，而是变多"。永久升级应该主要提供选项多样性（新角色、新技能、新起始道具），而不是纯数值提升。如果永久升级太强，游戏会变成"刷 meta 货币"，局内决策变得不重要。

进度节奏控制

进度节奏是玩家"感觉"的核心。太快则廉价，太慢则挫败。

进度节奏的三段论

  1. 蜜月期（0-20%）：快速解锁、频繁奖励、新机制不断出现。目标：让玩家"上钩"
  2. 探索期（20-70%）：节奏平稳、策略深化、有足够内容变化保持兴趣。目标：建立习惯
  3. 精通期（70-100%）：挑战显著提升、奖励变得稀有而珍贵、出现需要高度技巧的内容。目标：留住核心玩家

伤害计算公式

伤害公式是战斗系统的数学基础。选择哪种公式决定了整个战斗的"手感"。

减法模型（线性）
伤害 = 攻击力 - 防御力
简单直观，但防御力等于攻击力时伤害为 0（"防御穿透"问题）
百分比减伤模型
伤害 = 攻击力 × (1 - 防御力 / (防御力 + K))
K 为常数。防御力提供百分比减伤，永远不会 100% 免伤。《英雄联盟》使用此模型
乘法模型
伤害 = 攻击力 × 攻击力 / (攻击力 + 防御力)
攻防博弈更平滑，双倍攻击力 ≈ 双倍伤害，双倍防御力 ≈ 半伤

属性设计原则

  • 每个属性必须有用：如果某属性对某类角色毫无价值（"陷阱属性"），要么重新设计要么删除

战斗平衡方法论

DPS 基准法

所有伤害型角色/技能都换算成 DPS（每秒伤害）进行比较：

DPS = (单次伤害 × 命中率 × 暴击系数) / 攻击间隔

如果角色 A 的 DPS 远高于角色 B，且没有其他方面的劣势来补偿，说明不平衡。

注意：DPS 只是一个维度。生存能力、控制能力、团队增益都要纳入综合评估。可以设计一个"战斗力综合公式"：综合评分 = DPS × 0.4 + 生存 × 0.3 + 功能 × 0.3

石头-剪刀-布 平衡

让每种策略都有克制关系，确保没有"最优解"：

难度曲线理论

理想的难度曲线不是线性增长，而是锯齿形波浪——持续上升的整体趋势中穿插有规律的放松区间。

心流理论（Flow Theory）

心理学家 Csikszentmihalyi 提出的心流理论是难度设计的理论基础：

😰

挑战远超能力

🎯

😴

设计目标：让难度始终在心流区的边缘波动。稍微高于玩家能力以保持紧张，但不超过能力太多以避免挫败。

动态难度调整（DDA）

DDA 是游戏根据玩家表现自动调整难度的机制。做得好，玩家浑然不觉；做得差，玩家觉得被"操纵"。

DDA 实现策略

class DynamicDifficulty:
    def __init__(self):
        self.performance_window = []  # 最近 N 次战斗表现
        self.difficulty_level = 1.0   # 1.0 = 标准

    def record_combat(self, result):
        """记录战斗表现"""
        score = 0
        score += result.health_remaining / result.max_health  # 剩余血量比
        score += 1.0 if result.victory else 0
        score -= result.deaths * 0.3
        score += result.combo_accuracy * 0.2

        self.performance_window.append(score)
        if len(self.performance_window) > 10:
            self.performance_window.pop(0)

        self.adjust()

    def adjust(self):
        """根据近期表现调整难度"""
        avg = mean(self.performance_window)

        if avg > 0.8:   # 玩家碾压
            self.difficulty_level += 0.05
        elif avg < 0.3:  # 玩家苦战
            self.difficulty_level -= 0.08  # 降低更快（减少挫败）
        # 否则保持不变

        # 限制范围
        self.difficulty_level = clamp(self.difficulty_level, 0.5, 2.0)

    def apply(self, enemy):
        """应用到敌人属性"""
        enemy.hp *= self.difficulty_level
        enemy.damage *= (0.7 + 0.3 * self.difficulty_level)
        enemy.ai_aggression *= self.difficulty_level
DDA 的暗面

《生化危机4》的 DDA 是成功典范——玩家不知不觉中获得更多弹药和回血道具。《最后的生还者 第二章》也在玩家反复死亡后悄悄降低敌人攻击性。但《孢子》的 DDA 被玩家发现后遭到强烈反感——"你在作弊！"。关键原则：DDA 必须不可见，或者让玩家自己选择。

玩家能力成长曲线

玩家的"真实能力"由两部分组成：角色数值能力（装备/等级）和玩家操作能力（技巧/知识）。

有效战力 = 数值能力 × 操作系数
操作系数范围通常 0.3（新手）~ 1.5（高手）
好的难度设计让"平均水平"的操作系数 = 1.0

难度设置的设计

难度选择不只是"简单/普通/困难"三个按钮。现代游戏的难度设计越来越精细。

经典模式

简单/普通/困难/噩梦。简单粗暴但有效。问题是玩家开局选"普通"，后面嫌太简单或太难。

细粒度调节

《Celeste》可以单独调节游戏速度、无限冲刺、无敌等。《最后的生还者2》有 60+ 个可单独调节的辅助选项。

叙事模式

几乎不可能失败，让不擅长游戏的玩家也能体验故事。如《底特律：成为人类》。

无难度选择

宫崎英高的选择——"难度就是设计的一部分"。通过灵活的 build 和求助系统间接调难度。

橡皮筋效应（Rubber Banding）

橡皮筋效应是负反馈循环的一种特定表现：落后者获得额外助力，领先者受到额外阻力。

橡皮筋设计策略

  • 赛车游戏：AI 车手在落后时跑得更快（《马里奥赛车》）或获得更好道具
  • 体育游戏：落后队伍的射门准度微调提高（《FIFA》系列的争议"脚本"）
  • 策略游戏：弱势方获得额外资源或科技加速（《文明》的追赶机制）
  • FPS：落后队伍重生时间缩短或获得更多弹药

关键：橡皮筋效应必须有度。太强会让领先毫无意义（"我为什么要努力？"），太弱则起不到保持悬念的作用。最佳设计是让橡皮筋给落后者机会但不保证翻盘。

关卡设计

关卡设计原则

🎯
目标清晰

玩家在任何时刻都应该知道"我要去哪"和"我要做什么"。可以通过视觉引导（灯光、色彩）、空间引导（道路方向）、叙事引导（NPC提示）实现。

🗺
可读性（Readability）

玩家需要快速理解空间结构。可交互物体有统一视觉标识，危险区域有明确警告，安全区域让人放松。

🔄
循环与捷径

魂系列的精髓——探索后解锁回到起点的捷径，形成"环形"关卡结构。玩家的探索努力获得物质回报（更短的路径）。

🎭
意外与奖励

偏离主路径的探索应该有奖励。隐藏区域、秘密通道、彩蛋让好奇心得到回报，训练玩家主动探索。

引导设计（无教程的教学）

最好的教程是没有教程。通过关卡设计本身教会玩家，比弹出文字框高效百倍。

任天堂的「4步教学法」

Step 1 · 安全引入

在安全环境中首次展示机制。例如：一个不会掉落的平台上放一个可推动的箱子，旁边有一个刚好需要箱子才能到达的高台。

Step 2 · 简单应用

添加微小风险要求玩家主动使用机制。例如：推箱子跨过一个小坑到达对面。失败代价很低（重来很快）。

Step 3 · 复杂变化

加入时间压力或组合元素。例如：需要在计时平台消失前推箱子到位，或者需要推两个箱子到不同位置。

Step 4 · 终极考验

将该机制与其他已学机制组合，创造终极挑战。例如：在敌人攻击下推箱子解谜，同时避开陷阱。

设计心法

《超级马里奥》1-1 是关卡引导设计的教科书。前 10 秒内，玩家会自然地学到：移动、跳跃、问号块可以打、蘑菇会移动、吃了蘑菇会变大、管道可以踩、栗子头是敌人。所有这些没有一行文字说明。

任务/对象布局

敌人布局原则

  • 视线控制：让玩家先看到敌人，准备策略，再接触。偷袭应是例外而非常态
  • 组合搭配：远程 + 近战 + 盾兵的组合比 5 个相同敌人更有趣
  • 地形利用：敌人应占据有利地形（高处、掩体后），迫使玩家思考进攻路线
  • 渐进展示：新敌人类型第一次出现时应单独出场，让玩家了解其行为模式

物品布局原则

  • 危险前补给：大战前放置回复道具，暗示即将到来的挑战
  • 风险-回报：最好的奖励放在最危险或最隐蔽的地方
  • 面包屑引导：沿正确路径放置小奖励，引导玩家前进方向
  • 探索回报：分支路径末端必须有奖励，否则玩家会觉得浪费时间

2D vs 3D 关卡设计差异

2D 关卡特点
  • 信息展示更直观（一屏展示全局）
  • 空间理解门槛低
  • 以"平面构成"为核心
  • 依赖重力和平台跳跃
  • 迷路风险低
  • 开发成本更低
3D 关卡特点
  • 空间感和沉浸感更强
  • 垂直空间利用（多层结构）
  • 以"视线引导"为核心
  • 相机管理是核心挑战
  • 需要更多导航辅助
  • 开发成本显著更高

关卡设计工作流

Phase 1 · 概念设计

确定关卡的核心体验目标（"恐惧感"、"成就感"、"探索的喜悦"）。绘制流程草图，标注关键节拍点。这个阶段用纸笔就够了。

Phase 2 · 灰盒（Greybox）

用基础几何体搭建关卡骨架。不做任何美术，纯粹测试空间尺度、节奏和游戏性。这是最重要的阶段——大量迭代。

Phase 3 · 游戏性验证

在灰盒上放入敌人、道具、触发器。反复测试难度和节奏。请团队成员或测试者试玩，观察他们的行为。

Phase 4 · 美术处理

替换灰盒为正式资产。添加灯光、粒子、环境叙事细节。美术不应改变已验证的游戏性。

Phase 5 · 打磨（Polish）

音效、视觉反馈、过场动画、性能优化。处理边界情况和可能的序列破坏（sequence break）。

程序化生成

噪声算法

噪声算法是程序化生成的数学基础，用于创造自然、有机的随机内容。

Perlin 噪声

Ken Perlin 于 1983 年发明的梯度噪声算法，广泛用于地形生成、纹理生成、云雾效果。

// Perlin 噪声核心思想（简化）
// 1. 在网格每个整数点放置随机梯度向量
// 2. 对于任意点 (x, y)，找到其所在网格的四个角点
// 3. 计算点到四个角点的距离向量与梯度向量的点积
// 4. 用平滑插值函数（通常是 6t^5 - 15t^4 + 10t^3）混合

function perlin2D(x, y):
    // 获取网格坐标
    xi = floor(x), yi = floor(y)
    // 获取局部坐标 (0-1)
    xf = x - xi, yf = y - yi

    // 获取四个角的梯度点积
    n00 = dot(gradient[xi][yi],     (xf,   yf))
    n10 = dot(gradient[xi+1][yi],   (xf-1, yf))
    n01 = dot(gradient[xi][yi+1],   (xf,   yf-1))
    n11 = dot(gradient[xi+1][yi+1], (xf-1, yf-1))

    // 平滑插值
    u = smoothstep(xf)
    v = smoothstep(yf)
    return lerp(lerp(n00, n10, u), lerp(n01, n11, u), v)

多octave噪声（分形噪声）

单层噪声过于平滑。叠加多层不同频率和振幅的噪声创造更自然的效果：

fbm(p) = Σi=0..n amplitudei × noise(frequencyi × p)
常用参数：amplitude = 0.5（每层振幅减半）, frequency = 2.0（每层频率翻倍）
// 分形布朗运动（FBM）
function fbm(x, y, octaves=6):
    value = 0
    amplitude = 1.0
    frequency = 1.0
    total_amplitude = 0

    for i in range(octaves):
        value += amplitude * perlin2D(x * frequency, y * frequency)
        total_amplitude += amplitude
        amplitude *= 0.5     // 每层振幅减半（persistence）
        frequency *= 2.0     // 每层频率翻倍（lacunarity）

    return value / total_amplitude  // 归一化到 [-1, 1]

// 地形生成应用
for x, y in map:
    height = fbm(x * 0.01, y * 0.01)
    if height < -0.2:   tile = DEEP_WATER
    elif height < 0.0:  tile = SHALLOW_WATER
    elif height < 0.3:  tile = GRASS
    elif height < 0.6:  tile = HILL
    else:               tile = MOUNTAIN

// WFC 核心算法（简化）
class WFC:
    def __init__(self, width, height, tiles, rules):
        self.grid = [[set(tiles) for _ in range(width)]
                     for _ in range(height)]  # 每格初始为所有可能
        self.rules = rules  # {tile: {方向: [可相邻tiles]}}

    def solve(self):
        while not self.is_collapsed():
            # 1. 找到熵最低的格子（可能性最少）
            x, y = self.find_min_entropy()

            # 2. 坍缩：随机选一个可能的 tile
            tile = random.choice(list(self.grid[y][x]))
            self.grid[y][x] = {tile}

            # 3. 传播约束
            self.propagate(x, y)

    def propagate(self, start_x, start_y):
        """约束传播：移除不满足邻接规则的可能性"""
        stack = [(start_x, start_y)]
        while stack:
            x, y = stack.pop()
            for dx, dy, direction in neighbors:
                nx, ny = x + dx, y + dy
                if not in_bounds(nx, ny): continue

                # 计算该邻居位置还允许哪些 tile
                allowed = set()
                for tile in self.grid[y][x]:
                    allowed |= self.rules[tile][direction]

                # 移除不被允许的可能性
                before = len(self.grid[ny][nx])
                self.grid[ny][nx] &= allowed

                if len(self.grid[ny][nx]) < before:
                    stack.append((nx, ny))  # 继续传播

WFC 的优势在于它保证局部一致性——每个 tile 和它的邻居都满足预定义的规则。这使得生成的结果在视觉和逻辑上都是合理的。

BSP 地牢生成

Binary Space Partitioning（BSP）是 Roguelike 地牢生成的经典算法。

// BSP 地牢生成
function generateDungeon(width, height, minRoomSize):
    // 1. 递归分割空间
    root = BSPNode(0, 0, width, height)
    split(root, minRoomSize)

    // 2. 在叶子节点生成房间
    for leaf in root.getLeaves():
        leaf.room = randomRoom(
            x = leaf.x + rand(1, leaf.w/4),
            y = leaf.y + rand(1, leaf.h/4),
            w = rand(minRoomSize, leaf.w - 2),
            h = rand(minRoomSize, leaf.h - 2)
        )

    // 3. 生成走廊连接相邻房间
    connectRooms(root)

function split(node, minSize):
    if node.w < minSize*2 and node.h < minSize*2:
        return  // 太小了，停止分割

    // 选择分割方向（较长的轴）
    if node.w > node.h:
        splitX = rand(node.x + minSize, node.x + node.w - minSize)
        node.left  = BSPNode(node.x, node.y, splitX - node.x, node.h)
        node.right = BSPNode(splitX, node.y, node.x + node.w - splitX, node.h)
    else:
        splitY = rand(node.y + minSize, node.y + node.h - minSize)
        node.left  = BSPNode(node.x, node.y, node.w, splitY - node.y)
        node.right = BSPNode(node.x, splitY, node.w, node.y + node.h - splitY)

    split(node.left, minSize)
    split(node.right, minSize)

程序化 vs 手工设计的权衡

程序化生成 优势
  • 无限可重玩性
  • 开发效率高（一次编写算法）
  • 海量内容（不需要逐个手工制作）
  • 适合 Roguelike/沙盒
  • 每次体验独特
程序化生成 劣势
  • 缺乏手工打磨的精致感
  • 难以控制叙事节奏
  • 可能生成不合理内容
  • 调试困难（问题难以复现）
  • 缺乏设计者意图的"惊喜"
最佳实践

最好的方案通常是混合方法：用程序化生成创建基础结构，再用手工设计的模块（预制件/模板）填充关键位置。《Dead Cells》用手工设计的房间模块 + 程序化的模块排列。《Spelunky》用程序化的关卡结构 + 手工设计的房间模板。这样既有随机性带来的新鲜感，又有设计者精心安排的体验。

Roguelike 生成策略

层级递进

每层使用不同的生成规则。浅层房间大、敌少；深层房间小、敌密、陷阱多。逐层增加新的地形元素和敌人类型。

关键路径保证

先生成从入口到出口的关键路径，确保可通关性。然后在路径外添加分支、死胡同和秘密区域作为探索奖励。

主题约束

地下城 → 石头墙+火把+骷髅；森林 → 树木+蘑菇+精灵。主题不仅是美术，还约束敌人类型和物品分布。

种子系统

允许玩家分享种子（seed）来重现相同地图。竞速和挑战模式依赖固定种子确保公平性。

Part 08

数值策划基础

数值设计的核心概念

数值策划（Game Balancing / Economy Design）是将设计意图转化为具体数字的艺术与科学。核心原则：数值服务于体验，不是体验服务于数值。

数值设计三要素

📐
量级（Scale）

数值的绝对大小。100 点伤害 vs 10000 点伤害本质上可以相同，但大数字给人更强的正反馈。许多手游故意使用大数字（百万伤害）来营造爽感。

📊
比率（Ratio）

数值之间的相对关系。攻击力是防御力的 2 倍，升级后变成 2.5 倍——这个变化比绝对值更重要。平衡的本质是控制比率。

📈
增长（Growth）

数值随时间/等级的变化速率。线性增长让人感觉"稳定"，指数增长制造"爆发感"，对数增长产生"天花板感"。

常用数学模型

线性模型

f(x) = a × x + b
等差增长。每级固定提升同样的值。简单可预测。

适用场景：基础属性成长（每级 +5 力量）、固定奖励递增。

问题：后期增幅感知下降——从 100 到 105 的 5% 提升远不如从 10 到 15 的 50% 提升有感觉。

指数模型

f(x) = a × bx
等比增长。每级按比例提升。前期平缓后期爆炸。

适用场景：升级所需经验值（越来越多）、装备战力（越来越强）。

问题：数值膨胀（Power Creep）——后期数字大到失去意义。需要定期进行"数值压缩"（stat squish）。

对数模型

f(x) = a × log(b × x + 1)
前期快速增长，后期趋于平缓。天然的收益递减。

适用场景：暴击率（堆到一定值后几乎不增长）、移动速度上限。

S 曲线（Logistic）

f(x) = L / (1 + e-k(x - x₀))
L = 上限, k = 增长速率, x₀ = 中点。前期慢、中期快、后期趋平。

适用场景：角色能力上限、命中率计算、难度曲线。

数值验证与平衡测试

边界测试

测试最低和最高属性值。全力量角色 vs 全敏捷角色的差异是否在可接受范围？是否存在"无敌"或"无法通关"的组合？

蒙特卡洛模拟

运行 10000 次模拟战斗，统计胜率、平均 TTK、伤害分布。如果战士 vs 法师胜率是 90%，说明严重不平衡。

敏感度分析

改变单个参数 ±10%，观察结果变化多大。如果暴击率从 30% 变为 33% 导致 DPS 提升 25%，说明这个参数过于敏感。

玩家行为模拟

模拟不同类型玩家（速推/肝帝/休闲/氪金），检查每类玩家的游戏体验是否都在可接受范围内。

经济模拟

经济模拟是验证整个游戏经济健康度的终极手段：

// 经济健康度检查指标
class EconomyHealthCheck:
    def check(self, sim_data):
        results = {}

        # 1. 基尼系数（贫富差距）
        # 0 = 完全平等, 1 = 一人独占
        # 健康范围: 0.3 - 0.5
        results['gini'] = self.calc_gini(sim_data.player_wealth)

        # 2. 物价稳定性
        # CPI 涨幅应 < 5% / 月
        results['cpi_change'] = self.calc_cpi_change(sim_data.prices)

        # 3. 新玩家购买力
        # 新玩家一周内应能买到基础装备
        results['newbie_affordability'] = (
            sim_data.avg_weekly_income_new /
            sim_data.basic_gear_price
        )

        # 4. 货币流速
        # 每日交易总量 / 货币总存量
        # 健康范围: 0.05 - 0.15
        results['velocity'] = (
            sim_data.daily_transaction_volume /
            sim_data.total_money_supply
        )

        # 5. 沉淀率
        # 被"闲置"的货币占比，越高说明缺乏消费动力
        results['idle_ratio'] = (
            sim_data.idle_money /
            sim_data.total_money_supply
        )

        return results

// 伤害计算流程
function calculateDamage(attacker, defender, skill):
    // 1. 基础伤害
    if skill.type == "physical":
        raw = attacker.P_ATK × skill.power / 100
        defense = defender.P_DEF
    else:
        raw = attacker.M_ATK × skill.power / 100
        defense = defender.M_DEF

    // 2. 防御减伤（百分比模型）
    reduction = defense / (defense + 100)
    base_damage = raw × (1 - reduction)

    // 3. 属性克制
    element_multi = getElementMultiplier(skill.element, defender.weakness)
    // 克制: ×2.0  |  抵抗: ×0.5  |  中性: ×1.0  |  吸收: ×-0.5

    // 4. 暴击判定
    if random() < attacker.CRIT_RATE / 100:
        crit_multi = 1.5 + (attacker.STR or attacker.INT) × 0.01
    else:
        crit_multi = 1.0

    // 5. 随机波动（±10%）
    variance = 0.9 + random() × 0.2

    // 6. 最终伤害
    final = floor(base_damage × element_multi × crit_multi × variance)
    return max(final, 1)  // 最低伤害 1

// CTB（Conditional Turn-Based）行动系统
// 类似 FFX 的系统，速度决定行动频率

class TurnManager:
    def __init__(self):
        self.timeline = []  // 行动时间线

    def calculate_turns(self, units):
        """计算未来 10 个行动的顺序"""
        for unit in units:
            # 每个单位按固定间隔插入时间线
            interval = 1000 / unit.SPEED  // 速度越高间隔越短
            t = interval
            while t < max_timeline:
                self.timeline.append((t, unit))
                t += interval

        self.timeline.sort(key=lambda x: x[0])

    def apply_skill_delay(self, unit, skill):
        """技能使用后的延迟"""
        # 重型技能有更长的恢复时间
        delay = skill.recovery × (100 / unit.SPEED)
        unit.next_turn += delay
        # 这意味着快速角色用重型技能也需要等待
        # 创造了"速度 vs 威力"的有意义选择

常见系统设计陷阱

1
数值膨胀（Power Creep）

为了"新内容必须更强"而持续提高数值上限。最终导致旧内容完全失去意义，新玩家面临巨大的追赶压力。
解法：横向扩展（新的侧重/风格）而不是纵向膨胀（更大的数字）。

2
陷阱选项

提供看起来是选择、实际永远不应该选的选项。浪费玩家的时间和资源，降低信任感。
解法：每个选项都应该有至少一个场景是最优的。如果做不到，就删掉它。

3
过度复杂

设计了 20 种互相影响的状态效果，连策划自己都记不清交互规则。
解法："如果你无法用一句话向新手解释这个机制，它就太复杂了。"

4
最优策略固化

存在一个远优于其他策略的"最优解"，所有玩家都使用相同的构建/策略。
解法：石头剪刀布设计、定期平衡更新、元游戏（meta）轮换。

5
乘法陷阱

多个独立系统的 buff 叠乘后产生设计者未预料的极端数值。+50% × +50% × +50% ≈ +237.5%，而不是 +150%。
解法：限制同类 buff 的叠加数量，或使用加法叠加后再应用。

6
忽视边际效用

从 0% 暴击到 10% 暴击的感知提升远大于从 80% 到 90%。如果暴击率线性叠加，后期暴击装备会感觉"没用"。
解法：用对数或收益递减模型，或者让高暴击率解锁额外效果（超暴击）。

最后的忠告

系统设计的最高境界是"少即是多"。不要因为可以添加复杂性就添加。每个新系统都有维护成本、平衡成本、认知成本。

开始时用最简单的规则，让玩家试玩，观察什么有趣什么无聊，然后只添加解决真实问题的系统。

记住任天堂的宫本茂说过："一个好的点子就是解决了多个问题的点子。"
