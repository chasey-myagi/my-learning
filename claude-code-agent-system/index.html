01架构总览与核心概念

Claude Code 的多代理系统围绕一个核心思想构建：主代理通过 Agent 工具派生子进程，子进程通过独立的 query 循环完成任务后，将结果以 <task-notification> XML 格式回传。

02Agent 类型分类体系

03SubAgent 生成与执行流程

AgentTool.call() 是所有子代理生成的入口点。它处理三条主要路径：Teammate 生成、Fork 子代理、标准 SubAgent。核心执行引擎是 runAgent() 生成器函数。

AgentTool.call() 路由决策树 ============================== 输入: { prompt, subagent_type, description, name, team_name, isolation, ... } | v team_name && name ? ----YES----> spawnTeammate() | 返回 teammate_spawned NO | v subagent_type 为空 ? | +----+----+ | | YES NO | | v v isForkSubagentEnabled() ? selectedAgent = agents.find(subagent_type) | | YES NO | | v v FORK_AGENT GENERAL_PURPOSE_AGENT | | +------+-------------+ | v isolation === 'remote' ? ---YES---> teleportToRemote() | 返回 remote_launched NO | v isolation === 'worktree' ? ---YES---> createAgentWorktree() | v run_in_background / forceAsync ? | | YES NO | | v v 异步路径: 同步路径: registerAsyncAgent() for await (msg of runAgent()) runAsyncAgentLifecycle() 收集消息直到完成 返回 async_launched 返回 completed + result

runAgent() 核心引擎

位于 src/tools/AgentTool/runAgent.ts，这是一个 AsyncGenerator 函数，驱动子代理的完整生命周期：

export async function* runAgent({

  model,               // 模型覆盖
  maxTurns,            // 最大轮次限制
  availableTools,      // 预组装的工具池
  useExactTools,       // Fork: 使用精确工具集（缓存命中）
  worktreePath,        // Worktree 隔离路径
  ...
}): AsyncGenerator<Message, void> {
  // 1. 初始化 Agent MCP 服务器
  // 2. 创建子代理上下文 (createSubagentContext)
  // 3. 组装系统提示词
  // 4. 进入 query() 循环
  // 5. yield 每条消息给调用方
  // 6. 清理资源
}

关键机制：MCP 服务器初始化

每个 Agent 可以通过 frontmatter 声明自己的 MCP 服务器。initializeAgentMcpServers() 函数：

  • 字符串引用：引用父级已连接的 MCP 服务器（共享连接，不做清理）
  • 内联定义：{ name: config } 对象创建新连接（Agent 结束时清理）
  • plugin-only 策略：当 strictPluginOnlyCustomization 锁定 MCP 时，跳过用户自定义 Agent 的 frontmatter MCP

上下文创建：createSubagentContext()

位于 src/utils/forkedAgent.ts，这是保证子代理状态隔离的核心函数。它创建一组与父级隔离的参数集（CacheSafeParams），确保：

  • 系统提示词、工具定义、模型在 fork 场景下字节相同（最大化 prompt cache 命中）
  • 文件状态缓存独立（fork 克隆父级的，新 SubAgent 从空开始）
  • content replacement state 隔离

04Fork 子代理机制

Fork 是 Claude Code 最精巧的代理派生方式。与普通 SubAgent 从零上下文开始不同，Fork 子代理继承父级的完整对话历史和系统提示词，实现 prompt cache 共享。

Fork 的设计哲学

Fork yourself (omit subagent_type) when the intermediate tool output isn't worth keeping in your context. The criterion is qualitative -- "will I need this output again" -- not task size.

Fork 的核心思想是：当你需要执行一项工作但不需要在主对话中保留其过程输出时，fork 一个自己的副本去完成。

FORK_AGENT 定义

// src/tools/AgentTool/forkSubagent.ts
export const FORK_AGENT = {
  agentType: FORK_SUBAGENT_TYPE,     // 'fork'
  tools: ['*'],                       // 继承父级全部工具
  maxTurns: 200,
  model: 'inherit',                   // 继承父级模型（缓存命中关键）
  permissionMode: 'bubble',           // 权限提示冒泡到父级终端
  source: 'built-in',
  getSystemPrompt: () => '',          // 不使用，直接传入父级渲染后的字节
} satisfies BuiltInAgentDefinition

Fork 消息构建流程

buildForkedMessages() 函数是 prompt cache 共享的关键。它确保所有 fork 子进程产生字节相同的 API 请求前缀：

Cache 共享策略

1. 保留父级 assistant 消息的完整内容（所有 tool_use、thinking、text block）
2. 为每个 tool_use 生成相同占位符的 tool_result："Fork started -- processing in background"
3. 在最后追加一个 per-child 唯一的 directive text block
4. 结果：[...history, assistant(all_tool_uses), user(placeholder_results..., directive)]
5. 只有最后一个 text block 不同 -- 最大化 cache hit

Fork 子进程的指令格式

// buildChildMessage() 输出的消息结构
<fork-boilerplate>
STOP. READ THIS FIRST.
You are a forked worker process. You are NOT the main agent.

RULES (non-negotiable):
1. Do NOT spawn sub-agents; execute directly.
2. Do NOT converse, ask questions, or suggest next steps
3. USE your tools directly: Bash, Read, Write, etc.
4. If you modify files, commit your changes before reporting.
5. Keep your report under 500 words.
6. Your response MUST begin with "Scope:".

Output format:
  Scope: <assigned scope>
  Result: <findings>
  Key files: <paths>
  Files changed: <list with commit hash>
  Issues: <if any>
</fork-boilerplate>

<fork-directive>{具体指令}</fork-directive>

递归 Fork 防护

// isInForkChild() 通过检测消息中的 boilerplate 标签防止递归
export function isInForkChild(messages: MessageType[]): boolean {
  return messages.some(m => {
    if (m.type !== 'user') return false
    return content.some(
      block => block.type === 'text' &&
        block.text.includes('<fork-boilerplate>'),
    )
  })
}

双重防护：先检查 querySource（compaction-resistant），再做消息扫描 fallback。

特性Fork 子代理标准 SubAgent

06消息传递与代理间通信

SendMessageTool 是代理间通信的唯一通道。它支持点对点消息、广播、结构化协议消息，以及跨会话通信。

SendMessage 输入格式

// src/tools/SendMessageTool/SendMessageTool.ts
const inputSchema = z.object({
  to: z.string(),      // 接收方：name / "*" / "uds:path" / "bridge:session"
  summary: z.string(), // 5-10 词 UI 预览

    z.string(),        // 纯文本消息

消息路由机制

SendMessage 路由决策 ===================== to 参数 | +---> "*" ---> handleBroadcast() | 遍历 teamFile.members | 为每人调用 writeToMailbox() | +---> "name" ---> handleMessage() | writeToMailbox(name, ...) | +---> "uds:path" ---> UDS 跨会话 (本地 socket) | +---> "bridge:id" ---> Remote Control 跨机器 | +---> {type: ---> 结构化协议处理: | "shutdown_ - handleShutdownRequest() | request"} - handleShutdownApproval() | - handleShutdownRejection() | - handlePlanApproval() +---> agentId ---> queuePendingMessage() 续接已存在的 SubAgent

Mailbox 文件系统通信

Teammate 之间的消息通过 writeToMailbox() 写入文件系统，接收方在下一轮工具调用时自动读取。消息格式：

{
  from: "researcher",
  text: "Found the auth bug in validate.ts:42",
  summary: "Auth bug found",
  timestamp: "2026-03-31T10:00:00Z",
  color: "#fb923c"
}

SubAgent 续接 (SendMessage to agentId)

当 to 字段指向一个已有的 agent ID 时，消息被排入 pendingMessages 队列，在 agent 的下一个 tool-round 边界被消费。这是 Coordinator 模式中续用 worker 的核心机制。

// 从 prompt.ts 中的使用示例
SendMessage({ to: "agent-a1b", message: "Fix the null pointer..." })

结构化协议：Shutdown 流程

  1. Leader 发送 { type: "shutdown_request" }
  2. Teammate 收到后决定是否批准
  3. 批准：发回 { type: "shutdown_response", approve: true }
  4. In-process teammate 直接 abort controller
  5. tmux/iTerm2 teammate 调用 gracefulShutdown()

07上下文隔离与工具权限

Claude Code 通过多层机制确保子代理的安全隔离：工具过滤、权限模式、Worktree 文件系统隔离、以及 CLAUDE.md 优化剥离。

工具过滤链

filterToolsForAgent() 在 agentToolUtils.ts 中实现了多级工具过滤：

// 1. MCP 工具 - 始终允许
if (tool.name.startsWith('mcp__')) return true

// 2. ExitPlanMode - plan 模式下允许
if (toolMatchesName(tool, EXIT_PLAN_MODE_V2_TOOL_NAME)
    && permissionMode === 'plan') return true

// 3. ALL_AGENT_DISALLOWED_TOOLS - 全局禁止集合
if (ALL_AGENT_DISALLOWED_TOOLS.has(tool.name)) return false

// 4. CUSTOM_AGENT_DISALLOWED_TOOLS - 非内置 Agent 额外禁止
if (!isBuiltIn && CUSTOM_AGENT_DISALLOWED_TOOLS.has(tool.name)) return false

// 5. ASYNC_AGENT_ALLOWED_TOOLS - 异步 Agent 白名单
if (isAsync && !ASYNC_AGENT_ALLOWED_TOOLS.has(tool.name)) return false

resolveAgentTools() 工具解析

将 Agent 定义的工具列表（白名单/黑名单/通配符）解析为最终工具集：

  • ['*'] 或 undefined：使用全部可用工具（过滤后）
  • 显式列表：只保留列出的工具
  • disallowedTools：从可用集合中排除
  • Agent(worker, researcher) 语法：限制子 Agent 可派生的类型

权限模式（PermissionMode）

模式行为使用场景
bubble权限提示冒泡到父级终端Fork 子代理（共享父级终端）
dontAsk不弹出权限提示claude-code-guide（只读操作）

Worktree 文件系统隔离

当 isolation: "worktree" 时，createAgentWorktree() 创建一个 Git worktree 副本：

// src/tools/AgentTool/forkSubagent.ts - buildWorktreeNotice()
export function buildWorktreeNotice(
  parentCwd: string, worktreeCwd: string
): string {
  return `You've inherited the conversation context above from a parent
  agent working in ${parentCwd}. You are operating in an isolated git
  worktree at ${worktreeCwd} -- same repository, same relative file
  structure, separate working copy. Paths in the inherited context
  refer to the parent's working directory; translate them to your
  worktree root.`
}

Worktree 完成后，如果有变更则保留分支信息返回给父级；无变更则自动清理。

CLAUDE.md 优化剥离

只读 Agent（Explore、Plan）设置 omitClaudeMd: true，从上下文中移除 CLAUDE.md 内容：

// src/tools/AgentTool/runAgent.ts
const shouldOmitClaudeMd =
  agentDefinition.omitClaudeMd &&
  !override?.userContext &&
  getFeatureValue_CACHED_MAY_BE_STALE('tengu_slim_subagent_claudemd', true)

const { claudeMd: _omittedClaudeMd, ...userContextNoClaudeMd } = baseUserContext
const resolvedUserContext = shouldOmitClaudeMd
  ? userContextNoClaudeMd

这为每周 3400 万+ 次 Explore 调用节省了约 5-15 Gtok 的开销。

08Coordinator 模式

Coordinator 模式将主 Agent 转变为一个纯调度者：它不直接使用 Bash/Edit/Write 等工具，而是通过 Agent 和 SendMessage 指挥 Worker 完成所有实际工作。

启用方式

// src/coordinator/coordinatorMode.ts
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false
}

通过 CLAUDE_CODE_COORDINATOR_MODE=1 环境变量启用。与 Fork 机制互斥。

Coordinator 的专属工具集

Coordinator 只拥有以下工具：

  • Agent -- 生成新 Worker
  • SendMessage -- 续接已有 Worker
  • TaskStop -- 停止正在运行的 Worker

Coordinator 系统提示词关键原则

核心设计原则（来自源码）
  • Never delegate understanding -- 不要写 "based on your findings, fix the bug"，要自己综合理解后写出具体 spec
  • Parallelism is your superpower -- 独立的 Worker 应并行启动
  • Continue vs Spawn -- 高上下文重叠用 SendMessage 续接，低重叠用 Agent 新建
  • Task notifications are user-role messages -- Worker 结果以 <task-notification> XML 到达

异步生命周期：runAsyncAgentLifecycle()

这是所有后台 Agent 共用的生命周期管理器，位于 agentToolUtils.ts：

  1. 创建 ProgressTracker
  2. 可选启动 Agent 总结器（startAgentSummarization()）
  3. 遍历 runAgent() 的消息流
  4. 每条消息：追加到 AppState、更新进度、发送 SDK 事件
  5. 完成时：completeAsyncAgent() -> classifyHandoffIfNeeded() -> enqueueAgentNotification()
  6. 中止时：killAsyncAgent() -> 发送部分结果通知
  7. 失败时：failAsyncAgent() -> 发送错误通知
关键设计：状态转换优先于通知装饰

源码注释 gh-20236 记录了一个重要教训：completeAsyncAgent() 必须在 classifyHandoffIfNeeded()（API 调用）和 getWorktreeResult()（git 操作）之前执行。这样 TaskOutput(block=true) 可以立即解除阻塞，不会因通知装饰步骤的延迟而卡住。
