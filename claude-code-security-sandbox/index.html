🛡 安全架构总览 -- 纵深防御模型

Claude Code 的安全不是单点防护，而是六层纵深防御（Defense-in-Depth）。每一层独立工作，即使某层被绕过，后续层仍然能阻止攻击。

L1 OS 级沙箱隔离 (Seatbelt / bubblewrap) sandbox-adapter.ts
↓
L2 AST 语法解析 + 语义提取 (tree-sitter) ast.ts, parser.ts
↓
L3 静态安全验证器链 (20+ 独立验证器) bashSecurity.ts
↓
L4 权限规则匹配 (allow / deny / ask) bashPermissions.ts
↓
L5 路径验证 + 文件系统防护 pathValidation.ts, filesystem.ts
↓
L6 用户审批 (人类参与决策 / Human-in-the-Loop) permissions.ts

核心设计原则是 Fail-Closed：任何无法理解或解析的命令，都会回退到"需要用户批准"的状态。从源码注释可以直接看到这一哲学：

// The key design property is FAIL-CLOSED: we never interpret structure we
// don't understand. If tree-sitter produces a node we haven't explicitly
// allowlisted, we refuse to extract argv and the caller must ask the user.
💡
关键架构决策

沙箱是独立于权限系统的 OS 级屏障。即使所有静态分析器都被绕过、用户误批准了危险命令，沙箱仍然能阻止文件系统写入和网络访问。这就是"纵深防御"的意义。

📦 Bash 沙箱机制 -- OS 级隔离

沙箱是整个安全体系的最外层。它在操作系统层面限制进程的文件系统访问、网络连接和系统调用，由 @anthropic-ai/sandbox-runtime 包提供。

双平台沙箱实现

🍏

macOS: Seatbelt

使用 macOS 原生的 sandbox-exec 机制（Seatbelt 框架），通过 SBPL（Sandbox Profile Language）定义允许的文件路径、网络域名和系统调用。支持 glob 模式匹配路径。

🐧

Linux/WSL: bubblewrap

使用 bwrap（bubblewrap）进行命名空间隔离。通过 bind-mount 精确控制可读/可写目录，配合 socat 进行网络过滤。不支持 glob 模式（已知限制）。

沙箱配置转换

sandbox-adapter.ts 是桥接层，将 Claude Code 的设置格式转换为沙箱运行时配置。核心转换逻辑覆盖三个维度：

维度配置来源安全策略
文件系统 permissions.allow/deny + sandbox.filesystem.* 当前目录默认可写，settings.json / .claude/skills 始终 deny-write
网络 WebFetch 域名规则 + sandbox.network.* 企业可启用 allowManagedDomainsOnly 锁定域名白名单
进程 sandbox.enableWeakerNestedSandbox 嵌套沙箱（Docker-in-sandbox）可选放宽

Bare Git Repo 攻击防御

一个精巧的安全细节：攻击者可能在工作目录植入 HEAD、objects/、refs/ 等文件，让 Git 将工作目录误判为 bare repo。如果 config 中含有 core.fsmonitor，就能在非沙箱 git 命令中实现代码执行。

// SECURITY: Git's is_git_directory() treats cwd as a bare repo if it has
// HEAD + objects/ + refs/. An attacker planting these (plus a config with
// core.fsmonitor) escapes the sandbox when Claude's unsandboxed git runs.
const bareGitRepoFiles = ['HEAD', 'objects', 'refs', 'hooks', 'config']
// 已存在的文件：deny-write（只读绑定）
// 不存在的文件：命令结束后 scrub（清除植入文件）

沙箱启用判断

shouldUseSandbox.ts 的决策流程：

  • 1. 检查 SandboxManager.isSandboxingEnabled() -- 平台是否支持、依赖是否就绪、设置是否开启
  • 2. 检查 dangerouslyDisableSandbox 参数 -- 需要策略允许才能生效
  • 3. 检查 excludedCommands -- 用户配置的排除命令（注意：这是便利功能，不是安全边界）
⚠️
源码注释原文

"NOTE: excludedCommands is a user-facing convenience feature, not a security boundary. It is not a security bug to be able to bypass excludedCommands." 真正的安全控制是沙箱权限系统本身。

🌳 AST 安全解析 -- tree-sitter 驱动

ast.ts 使用 tree-sitter-bash 解析命令，替代了早期基于 shell-quote + 手写字符遍历的方案。核心思路：用允许列表（allowlist）而非阻止列表。

解析架构

tree-sitter 将 bash 命令解析为抽象语法树，然后 ast.ts 遍历语法树，对每个节点进行分类：

type ParseForSecurityResult =
  | { kind: 'simple'; commands: SimpleCommand[] }   // 可以安全提取 argv[]
  | { kind: 'too-complex'; reason: string }          // 无法分析，需要用户审批
  | { kind: 'parse-unavailable' }                    // tree-sitter 不可用

节点类型允许列表

关键设计：只允许已知安全的节点类型通过。未知节点类型一律触发 too-complex：

✅

结构节点 (递归遍历)

program, list, pipeline, redirected_statement -- 递归进入找叶子节点

🔴

危险节点 (立即拒绝)

command_substitution, process_substitution, subshell, for_statement, function_definition 等 14 种

解析差异攻击防护

bash 和 tree-sitter 在某些边界情况下对命令的解析不一致（Parser Differential），这是一类重要的攻击面。ast.ts 内置了多种预检：

检测项对应正则/逻辑防御目标
控制字符/[\x00-\x08\x0B-\x1F\x7F]/bash 静默丢弃但 tree-sitter 保留的字符
Unicode 空白/[\u00A0\u1680\u2000-\u200B...]/不可见空白让用户看不到隐藏内容
反斜杠空格/\\[ \t]/bash 视为转义空格，tree-sitter 保留原文
Zsh 动态目录/~\[/~[name] 在 zsh 中可执行任意代码
Zsh EQUALS/(?:^|[\s;&|])=[a-zA-Z_]/=cmd 展开为 which cmd 的完整路径
花括号展开/\{[^{}\s]*(,|\.\.)[^{}\s]*\}/tree-sitter 不展开花括号，bash 会

变量追踪与安全展开

ast.ts 还实现了轻量级变量追踪（varScope），当命令中出现 VAR=value && cmd $VAR 时，能追踪 VAR 的赋值并安全替换。但对于危险场景（如 $IFS 注入、bare 变量的 word-splitting），会保守拒绝：

// Unquoted $VAR in bash undergoes word-splitting (on $IFS) and pathname
// expansion (glob matching on * ? [). A value containing these metacharacters
// cannot be trusted as a bare arg: `VAR="-rf /" && rm $VAR` -> bash runs
// `rm -rf /` (two args) but our argv would have `['rm', '-rf /']` (one arg)

🔗 静态安全验证器链

Zsh 特殊危险命令

由于 BashTool 通过用户默认 shell 执行（通常是 zsh），需要额外检测 zsh 特有的危险命令：

const ZSH_DANGEROUS_COMMANDS = new Set([
  'zmodload',    // 加载 zsh 模块（mapfile, net/tcp, zpty...）
  'emulate',     // -c 参数是 eval 等价物
  'sysopen',     // 精细文件操作 (zsh/system)
  'sysread', 'syswrite', 'sysseek',  // 直接 fd 读写
  'zpty',        // 伪终端命令执行
  'ztcp',        // TCP 连接（数据窃取）
  'zsocket',     // Unix/TCP 套接字
  'zf_rm', 'zf_mv', 'zf_ln', 'zf_chmod',  // 绕过二进制检查的内置命令
  // ... 更多
])
☠️
为什么 zmodload 最危险？

zmodload zsh/system 可以加载 sysopen/syswrite（文件 I/O），zsh/zpty 可以执行命令，zsh/net/tcp 可以建立网络连接窃取数据。一条 zmodload 就能打开多个攻击面。

💣 危险命令检测与拦截

三层检测机制独立运作：静态模式匹配、权限规则系统、和用户提示信息生成。

破坏性命令警告

destructiveCommandWarning.ts 对已知的高风险命令模式生成人类可读的警告信息，在用户审批界面显示。这是 纯信息性的，不影响权限判断逻辑：

git commit --amendNote: may rewrite the last commitGit
rm -rfNote: may recursively force-remove filesFile
DROP TABLENote: may drop database objectsDB
kubectl deleteNote: may delete Kubernetes resourcesInfra
terraform destroyNote: may destroy infrastructureInfra

危险权限模式检测

dangerousPatterns.ts 定义了在 auto-mode 下必须清除的危险权限前缀：

export const DANGEROUS_BASH_PATTERNS: readonly string[] = [
  // 解释器 -- 可执行任意代码
  'python', 'python3', 'node', 'deno', 'ruby', 'perl', 'php',
  // 包管理器 -- 可运行任意脚本
  'npx', 'bunx', 'npm run', 'yarn run',
  // Shell
  'bash', 'sh', 'zsh', 'fish',
  // 元命令
  'eval', 'exec', 'env', 'xargs', 'sudo',
  // 远程执行
  'ssh',
]
💡
设计理念

允许规则如 Bash(python:*) 意味着模型可以执行任意 Python 代码。在 auto-mode 下，这些规则会被自动剥离（permissionSetup.ts），强制每次执行都需要用户确认。

命令语义解释

commandSemantics.ts 处理的不是安全检测，而是命令退出码的语义理解 -- grep 返回 1 不是错误（而是"未找到匹配"），diff 返回 1 表示"文件不同"。这防止 Claude 误将正常行为判定为错误。

📁 文件系统防护体系

多层文件系统保护覆盖路径验证、危险路径拦截、大小写混淆防御和符号链接安全。

路径验证

pathValidation.ts 为每个文件操作命令定义了路径提取器和操作类型（read/write/create），然后对提取的路径进行多重检查：

  • ▶ 路径遍历检查：检测 .. 路径遍历尝试（containsPathTraversal）
  • ▶ 符号链接解析：safeResolvePath 先解析符号链接再验证，防止符号链接指向受保护路径
  • ▶ 工作目录限制：写操作默认只允许在项目工作目录内
  • ▶ UNC 路径防护：containsVulnerableUncPath 检测 Windows UNC 路径攻击

危险路径绝对拦截

isDangerousRemovalPath 对高危删除目标实行零容忍，即使用户配置了 allow 规则也必须手动审批：

// 危险路径包括：
// - '*' (删除所有文件)
// - /*  (任何以 /* 结尾的路径)
// - /   (根目录)
// - ~   (家目录)
// - /usr, /tmp, /etc 等根目录直接子目录
// - C:\, D:\ (Windows 驱动器根)
// - C:\Windows, C:\Users 等

敏感文件保护

filesystem.ts 定义了两组受保护目标：

📄

DANGEROUS_FILES

.gitconfig, .bashrc, .zshrc, .profile, .mcp.json, .ripgreprc -- 可被用于代码执行或数据窃取的配置文件

📂

DANGEROUS_DIRECTORIES

.git, .vscode, .idea, .claude -- 包含敏感配置或可执行文件的目录

大小写混淆防御

macOS 和 Windows 的文件系统是大小写不敏感的，攻击者可以用 .cLauDe/Settings.locaL.json 绕过保护。normalizeCaseForComparison 统一转换为小写进行比较：

// This prevents bypassing security checks using mixed-case paths on
// case-insensitive filesystems (macOS/Windows)
export function normalizeCaseForComparison(path: string): string {
  return path.toLowerCase()
}

沙箱写入白名单

当沙箱启用时，写入路径不仅要满足应用层权限，还要在沙箱白名单中。但即使在白名单中，denyWithinAllow 列表（如 .claude/settings.json）仍然被禁止写入 -- 防止沙箱逃逸。

🔧 sed 注入防御 -- 深度验证

sedValidation.ts 专门处理 sed 命令的安全验证。sed 表面是文本替换工具，但包含 w（写文件）、e（执行命令）等危险操作。

白名单模式

只有两种 sed 模式被允许自动执行：

👁

Pattern 1: 行打印

sed -n '1,5p' -- 必须有 -n 标志，表达式只允许 Np、N,Mp 形式的打印命令

🔄

Pattern 2: 替换

sed 's/old/new/g' -- 只允许 / 作为分隔符，flag 限制为 g p i I m M 1-9，禁止 w e W E

多层防御策略

即使命令匹配了白名单模式，仍然经过 containsDangerousOperations 黑名单二次检查：

  • ❌ 非 ASCII 字符：拒绝 Unicode 同形字（fullwidth w = ｗ）
  • ❌ 花括号：拒绝 sed 块语法（过于复杂无法分析）
  • ❌ 换行符：拒绝多行命令
  • ❌ 反斜杠技巧：s\ 反斜杠分隔符、\| \# 替代分隔符
  • ❌ 写命令：任何位置的 w file、W file
  • ❌ 执行命令：任何位置的 e、e cmd
  • ❌ 替换 flag：s/old/new/w 或 s/old/new/e
  • ❌ 偏执检查：以 s 开头且以 w/W/e/E 结尾的未知模式

Flag 组合攻击防御

// 拒绝危险的 flag 组合：-ew, -eW, -ee, -we
// 这些可以将 sed 的 -e 表达式和 -w 写入文件合并在一个标志中
if (/-e[wWe]/.test(withoutSed) || /-w[eE]/.test(withoutSed)) {
  throw new Error('Dangerous flag combination detected')
}
✅
设计哲学

sed 验证遵循"保守拒绝"原则。containsDangerousOperations 的注释写道："CONSERVATIVE REJECTIONS: Broadly reject patterns that could be dangerous. When in doubt, treat as unsafe."

🔐 密钥与认证安全

API 密钥和 OAuth 令牌通过平台原生安全存储管理，配合性能优化的预取机制。

存储架构

🍏

macOS: Keychain

通过 security CLI 工具操作 macOS Keychain。服务名为 Claude Code（旧版 API Key）和 Claude Code-credentials（OAuth 令牌）。

🐧

其他平台: 纯文本 + Fallback

plainTextStorage.ts 作为回退。macOS 上使用 createFallbackStorage(keychain, plainText) 级联策略。Linux 暂无 libsecret 集成（TODO）。

Keychain 预取优化

keychainPrefetch.ts 是一个精巧的性能优化：在 main.tsx 初始化阶段并行启动两个 keychain 读取子进程，与模块加载并行执行：

// 两个 keychain 读取各约 32ms，串行需要 ~65ms
// 预取后在 main.tsx import 阶段并行完成，几乎零成本
export function startKeychainPrefetch(): void {
  const oauthSpawn = spawnSecurity(credentialsServiceName)
  const legacySpawn = spawnSecurity(legacyServiceName)
  prefetchPromise = Promise.all([oauthSpawn, legacySpawn])
}

安全细节

  • 🔒 stdin 长度限制：security -i 命令使用 4096 字节的 fgets 缓冲区。超长输入会被截断导致数据丢失。设定 SECURITY_STDIN_LINE_LIMIT = 4096 - 64 作为安全上限。
  • 🔒 缓存 TTL：Keychain 读取结果有 TTL 缓存，避免每次 API 请求都启动子进程。
  • 🔒 Stale-While-Error：如果 keychain 读取失败，保留之前的缓存值（而非缓存 null），防止一次性故障导致"未登录"错误。
  • 🔒 超时处理：预取有 10 秒超时。超时的结果不会写入缓存（因为可能确实有 key），让同步路径重试。

🌐 企业策略与网络安全

企业部署通过 MDM 配置、Policy Limits API 和网络白名单实现集中安全管理。

MDM 企业管理

通过操作系统级别的 MDM（Mobile Device Management）读取企业策略：

平台配置路径说明
macOScom.anthropic.claudecode pref domain只读取 /Library/Managed Preferences/（管理员写入），不读取用户 ~/Library
WindowsHKLM\SOFTWARE\Policies\ClaudeCodeHKLM 需要管理员权限，HKCU 为最低优先级
Linux/etc/claude-code/managed-settings.json无 MDM 等价物，使用文件替代

优先级遵循 "first source wins"：remote > HKLM/plist > managed-settings.json > HKCU

Policy Limits 服务

policyLimits/index.ts 从 Anthropic API 获取组织级策略限制，用于禁用特定 CLI 功能。设计特点：

  • ▶ Fail-Open：API 失败时不阻塞（非 fail-closed，这是特例）-- 用户体验优先
  • ▶ ETag 缓存：304 Not Modified 减少带宽
  • ▶ 1 小时后台轮询：保持策略最新
  • ▶ OAuth 资格检查：只有 Team/Enterprise 订阅用户才获取策略

网络安全边界

WebFetch 工具（WebFetchTool.ts）有独立的域名白名单和安全限制：

⚠️
WebFetch 白名单 != 沙箱网络白名单

源码注释特别强调：预批准域名列表（130+ 个开发文档站点）只用于 WebFetch GET 请求。沙箱网络限制不会继承这个列表，因为部分域名（如 huggingface.co、kaggle.com）允许文件上传，不限制的网络访问会导致数据泄露风险。

企业可通过 sandbox.network.allowManagedDomainsOnly: true 将网络访问完全限制在策略配置的域名白名单内，覆盖用户自行添加的域名。

沙箱 settings.json 写入防护

沙箱配置特别将 settings 文件加入写入黑名单，防止沙箱内进程通过修改配置文件来逃逸沙箱：

// Always deny writes to settings.json files to prevent sandbox escape
const settingsPaths = SETTING_SOURCES.map(source =>
  getSettingsFilePathForSource(source)
).filter(Boolean)
denyWrite.push(...settingsPaths)
denyWrite.push(getManagedSettingsDropInDir())
// 同时保护 .claude/skills 目录
denyWrite.push(resolve(originalCwd, '.claude', 'skills'))

📚 设计哲学与总结

从源码中提炼出的安全设计原则，以及这套体系的整体评价。

核心设计原则

🚫

Fail-Closed

任何未理解的命令结构都回退到用户审批。宁可多问，不可放过。
