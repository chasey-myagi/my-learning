启动流水线：并行预取三重奏

main.tsx 的前 20 行是整个 CLI 最关键的启动路径 —— 三个 side-effect import 必须在所有其他模块之前执行， 把 ~135ms 的模块求值时间转化为免费的并行窗口。

Claude Code 的启动流水线是一个经典的 "在等待的时候把活干了" 模式。 main.tsx 文件的前三条 import 带有刻意的副作用：

// main.tsx: 前 20 行 — 顺序敏感的副作用导入
import { profileCheckpoint } from './utils/startupProfiler.js';
profileCheckpoint('main_tsx_entry');           // 1. 标记入口时间

import { startMdmRawRead } from './utils/settings/mdm/rawRead.js';
startMdmRawRead();                              // 2. 启动 MDM plist 子进程

import { startKeychainPrefetch } from './utils/secureStorage/keychainPrefetch.js';
startKeychainPrefetch();                        // 3. 启动 Keychain 并行读取

// ↓ 接下来 ~135ms 的 import 求值…… 子进程已经在跑了
import { feature } from 'bun:bundle';
import { Command } from '@commander-js/extra-typings';
import chalk from 'chalk';
// ... 100+ more imports ...

Keychain 预取的精巧设计

keychainPrefetch.ts 有几个值得注意的工程细节：

  • 超时保护：10 秒超时。若子进程超时（err.killed）， 不 prime 缓存，让后续同步 spawn 重试 —— 避免把"真有 key 但没取到"误判为"没有 key"
  • 最小导入链：只导入 child_process + macOsKeychainHelpers.ts， 刻意避免 macOsKeychainStorage.ts（它会拉入 execa → human-signals → cross-spawn， 额外 ~58ms 的同步模块初始化）
  • 状态判断：legacyApiKeyPrefetch 区分 null（未完成） 和 { stdout: null }（完成但无 key），让同步读取路径只信任已完成的结果

MDM 原始读取

rawRead.ts 在 macOS 上并行发射多个 plutil 子进程读取 plist 配置， 在 Windows 上并行查询 HKLM / HKCU 注册表。一个快速优化： 用同步的 existsSync 先检查 plist 文件是否存在 —— 非 MDM 机器根本没有这些文件， 跳过 ~5ms 的无用 plutil 进程启动。

StartupProfiler：亚毫秒级检查点系统

三个分层采样的性能剖析器覆盖启动、查询、无头三种模式，零开销旁路设计确保非采样用户完全无感。

性能优化的前提是能测量。Claude Code 构建了一个基于 perf_hooks 的三级 profiler 体系：

Profiler覆盖范围采样率 (内部)采样率 (外部)触发方式

核心设计原则："决策在模块加载时做一次，非采样路径零开销"

// startupProfiler.ts: 模块级决策
const SHOULD_PROFILE = DETAILED_PROFILING || STATSIG_LOGGING_SAMPLED;

  if (!SHOULD_PROFILE) return;  // 非采样用户：一个 if 的成本
  const perf = getPerformance();
  perf.mark(name);

}

Bun 死代码消除 (DCE)

feature('FLAG') 不是运行时检查 —— 它是编译期的布尔常量替换， 让 Bun bundler 在 tree-shaking 时整块删除未启用的特性代码。

import { feature } from 'bun:bundle' 是 Bun bundler 提供的特殊模块。 每个 feature('FLAG') 调用在构建时被替换为 true 或 false 字面量， 配合 bundler 的 DCE pass，整个条件分支在输出中被彻底删除。

// 源码中的写法
const coordinatorModeModule = feature('COORDINATOR_MODE')
  ? require('./coordinator/coordinatorMode.js')
  : null;

const assistantModule = feature('KAIROS')
  ? require('./assistant/index.js')
  : null;

// 编译后（若 KAIROS=false）：
// const assistantModule = null;
// assistant/index.js 整个模块树不会被打入 bundle

性能影响：每个被关闭的 feature flag 不仅省去了运行时的条件判断， 更重要的是整个相关模块子树（coordinator、assistant、transcriptClassifier 等） 不会出现在最终 bundle 中。这直接减少了启动时需要解析和求值的 JavaScript 体积。

这种模式贯穿了多个文件：claude.ts、setup.ts、compact.ts、 toolExecution.ts 等都使用 feature() 来条件加载模块。 值得注意的是，所有被 feature gate 保护的导入都使用 require() 而非 import() —— 这是因为只有同步 require 才能被 Bun 的静态分析识别并在 DCE 时处理。

延迟加载策略：按需付费

从 OpenTelemetry (~1.2MB) 到 gRPC (~700KB)，从 AWS SDK (~279KB) 到原生音频模块， 每一个重量级依赖都有精心设计的懒加载路径。

Claude Code 的延迟加载遵循一个原则："你不用的东西，不该在启动时为它买单"。 以下是几个典型案例：

OpenTelemetry (init.ts)

注释原文："initializeTelemetry is loaded lazily via import() to defer ~400KB of OpenTelemetry + protobuf modules. gRPC exporters (~700KB via @grpc/grpc-js) are further lazy-loaded within instrumentation.ts."

// init.ts: 1P event logging 延迟初始化
void Promise.all([
  import('../services/analytics/firstPartyEventLogger.js'),
  import('../services/analytics/growthbook.js'),
]).then(([fp, gb]) => {
  fp.initialize1PEventLogging();
});
gRPC 按协议懒加载 (instrumentation.ts)

OTLP 导出器有 grpc / http-json / http-protobuf 三种协议，但一个进程只用一种。 静态导入会加载全部六个包 (~1.2MB)；动态导入只加载实际协议的那一个：

switch (protocol) {
  case 'grpc': {
    // Lazy-import to keep @grpc/grpc-js (~700KB) out of
    // the telemetry chunk when protocol is http/protobuf
    const { OTLPMetricExporter } = await import(
      '@opentelemetry/exporter-metrics-otlp-grpc'
    );
    break;
  }
  case 'http/protobuf': { /* 只加载 proto 包 */ }
  case 'http/json':     { /* 只加载 http 包 */ }
}

API 预连接与 TCP+TLS 握手

一个 fire-and-forget 的 HEAD 请求，把 ~100-200ms 的 TCP+TLS 握手从关键路径上移走。

apiPreconnect.ts 实现了一个简洁的预连接策略：在 init() 中 CA 证书和代理配置完成后，立即发射一个 HEAD 请求。 Bun 的 fetch 共享全局 keep-alive 连接池，所以后续的真实 API 请求可以复用已经建好的连接。

export function preconnectAnthropicApi(): void {
  if (fired) return;
  fired = true;

  // 跳过：Bedrock / Vertex / Foundry (不同端点)
  // 跳过：proxy / mTLS / unix socket (SDK 用独立 dispatcher)

  const baseUrl = process.env.ANTHROPIC_BASE_URL
    || getOauthConfig().BASE_API_URL;

  // HEAD = 无响应体 → 连接立即可复用
  // 10s 超时 → 慢网络不阻塞进程
  void fetch(baseUrl, {
    method: 'HEAD',
    signal: AbortSignal.timeout(10_000),
  }).catch(() => {});
}
调用时机的演进：早期版本在 cli.tsx 中调用，但那时 settings.json 还没加载完， ANTHROPIC_BASE_URL、代理配置、mTLS 配置都不可见 —— 预连接会 warm 错误的连接池。 现在移到 init.ts 中 applyExtraCACertsFromConfig() + configureGlobalAgents() 之后，确保预连接使用正确的传输配置。

Prompt Caching：多级缓存体系

三级缓存作用域（global / org / none）、精确到 5 分钟 / 1 小时的 TTL 控制、 缓存中断检测与归因系统 —— 这是目前 CLI 中最精密的性能优化之一。

Prompt Caching 是 Claude API 的一项核心能力 —— 对于每次请求中不变的 system prompt 和 tool schema 部分， 服务端可以缓存其 token 计算结果，后续请求命中时省去重复计算。 Claude Code 在客户端做了大量工作来最大化命中率。

缓存作用域策略

api.ts 中的 splitSysPromptPrefix() 函数将 system prompt 拆分为不同缓存级别的块：

三种缓存布局
场景块1块2块3块4
有 MCP 工具 Attribution (null) Prefix (org) 其余内容 (org) --
Global cache + boundary Attribution (null) Prefix (null) 静态内容 (global) 动态内容 (null)
默认 / 3P Attribution (null) Prefix (org) 其余内容 (org) --

每个块上的 cache_control 字段指导服务端：

cache_control: {
  type: 'ephemeral',
  scope: 'global' | 'org',  // global = 跨组织共享
  ttl: '5m' | '1h'          // 1h 仅限白名单用户
}

缓存中断检测 (promptCacheBreakDetection.ts)

这是一个 ~700 行的精密系统，分两阶段运行：

  1. Phase 1 (pre-call)：在 API 请求前对 system prompt、tool schema、model、betas 等做哈希快照， 检测哪些字段发生了变化
  2. Phase 2 (post-call)：API 响应后比较 cache_read_input_tokens， 若下降超过 5% 且绝对值超过 2000 tokens，则判定为缓存中断，并归因到具体原因

可检测的缓存中断原因：

  • System prompt 变化（含字符增量）
  • Tool schema 变化（具体到哪个工具的 schema 变了）
  • Model 切换
  • Fast mode 切换
  • Beta header 变化
  • Effort 值变化
  • Extra body params 变化
  • 5min / 1h TTL 过期
  • 服务端路由/驱逐（无客户端变化，<5min 间隔）
Auto mode / Overage 状态抖动修复：源码注释指出，AFK_MODE_BETA_HEADER 和 overage 状态 曾经因为会话中途变化导致缓存中断。修复方法是 "sticky-on latching" —— 一旦在会话中打开，就在整个会话中保持，即使状态条件不再满足。

工具结果存储与大结果截断

当 Grep 返回 400KB 的搜索结果时，直接塞进上下文窗口是灾难。 toolResultStorage 实现了磁盘持久化 + 预览截断 + 按消息批量预算控制。

toolResultStorage.ts 是上下文窗口管理的关键组件。它解决的核心问题： 如何在不丢失信息的前提下，防止工具结果撑爆 context window。

三层防线

  1. Per-tool 阈值：每个工具声明自己的 maxResultSizeChars， 与全局 50K 默认值取 Math.min()。GrowthBook flag tengu_satin_quoll 可以按工具名 override
  2. 磁盘持久化：超过阈值的结果写入 {projectDir}/{sessionId}/tool-results/{toolUseId}.{txt|json}， 替换为 2KB 预览 + 文件路径引用
  3. Per-message 批量预算：防止 N 个并行工具各自在阈值内但聚合超标 （例如 10 x 40K = 400K）

令牌估算优化

精确计数代价太高，粗估不够准确。Claude Code 用分层策略平衡速度与精度： 精确 API 计数做基准，粗估补差值，文件类型感知调整比率。

tokenEstimation.ts 和 tokens.ts 构成了令牌计数的核心。 最重要的函数是 tokenCountWithEstimation() —— 它是所有阈值检查（autocompact、session memory 等） 的唯一正确方式。

分层估算策略

export function tokenCountWithEstimation(messages): number {
  // 1. 找到最后一个有 usage 的 assistant 消息
  // 2. 回溯到同一 API 响应的第一个分片（处理并行工具调用）
  // 3. 用精确 usage 做基准
  // 4. 对后续新消息做粗估算补差
  return getTokenCountFromUsage(usage)
    + roughTokenCountEstimationForMessages(messages.slice(i + 1));
}

粗估算的核心是 roughTokenCountEstimation：字符数 / 4 ≈ token 数。 但对于 JSON 文件，这个比率不够准确：

文件类型感知的比率调整
function bytesPerTokenForFileType(ext: string): number {
  switch (ext) {
    case 'json':
    case 'jsonl':
    case 'jsonc':
      return 2;  // JSON 有大量单字符 token: { } : , "
    default:
      return 4;  // 通用文本的经验值
  }
}

注释原文："Dense JSON has many single-character tokens which makes the real ratio closer to 2 rather than the default 4"

并行工具调用的精确处理

当模型发出并行工具调用时，流式代码为每个 content block 生成独立的 AssistantMessage， 但它们共享同一个 message.id 和 usage。消息数组看起来像：

// [..., assistant(id=A), user(result), assistant(id=A), user(result), ...]

tokenCountWithEstimation 会回溯到同一 message.id 的第一个分片， 确保所有交错的 tool_result 都被计入估算范围 —— 否则只会估算最后一个 result，严重低估。

const shutdownTelemetry = async () => {

    process.env.CLAUDE_CODE_OTEL_SHUTDOWN_TIMEOUT_MS || '2000'

流式执行与投机推理

StreamingToolExecutor 实现了并发安全的流式工具执行， 投机推理系统在用户思考时提前运行只读操作， 自动压缩在上下文快满时优雅降级。

StreamingToolExecutor

StreamingToolExecutor.ts 是工具并行执行的核心。它实现了一个带优先级的并发调度器：

  • Concurrent-safe 工具（如 Read、Grep、Glob）可以互相并行执行
  • Non-concurrent 工具（如 Bash、Edit）需要独占执行
  • 结果按工具接收顺序缓冲和发射，保证消息序列一致性
  • 任一 Bash 工具出错时，通过子 AbortController 取消所有兄弟工具
投机推理系统 (speculation.ts)
