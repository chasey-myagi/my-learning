1 System Prompt 整体架构

Claude Code 的 system prompt 并非一个静态字符串，而是一个由多个模块协作、动态拼装的 SystemPrompt 类型 -- 本质上是一个品牌化的 readonly string[] 数组。每一个元素代表提示词的一个逻辑区段。

类型定义

src/utils/systemPromptType.ts
export type SystemPrompt = readonly string[] & {
  readonly __brand: 'SystemPrompt'
}

export function asSystemPrompt(value: readonly string[]): SystemPrompt {
  return value as SystemPrompt
}

品牌化类型的设计确保了 system prompt 数组不能被意外替换为普通字符串数组，所有的构建、拼接都必须通过 asSystemPrompt() 工厂函数。

核心入口：getSystemPrompt()

位于 src/constants/prompts.ts:444，这是构建默认系统提示词的主函数。它的返回值是 string[]，后续被 buildEffectiveSystemPrompt() 包装为 SystemPrompt。

src/constants/prompts.ts
export async function getSystemPrompt(
  tools: Tools,
  model: string,
  additionalWorkingDirectories?: string[],
  mcpClients?: MCPServerConnection[],
): Promise<string[]> {
  // 简化模式：只返回 CWD 和日期
  if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
    return [`You are Claude Code...\nCWD: ${getCwd()}\nDate: ...`]
  }

  // 正常模式：分为 静态 + 动态 两大区段
  return [
    // --- 静态内容（可缓存） ---
    getSimpleIntroSection(outputStyleConfig),  // 身份定义
    getSimpleSystemSection(),                  // 系统行为规则
    getSimpleDoingTasksSection(),              // 代码任务指引
    getActionsSection(),                       // 操作安全警告
    getUsingYourToolsSection(enabledTools),    // 工具使用规范
    getSimpleToneAndStyleSection(),            // 语气风格
    getOutputEfficiencySection(),              // 输出效率

    // === 动态边界标记 ===
    SYSTEM_PROMPT_DYNAMIC_BOUNDARY,

    // --- 动态内容（每次可能变化） ---
    ...resolvedDynamicSections,  // 由 systemPromptSection 注册
  ].filter(s => s !== null)
}

构建优先级：buildEffectiveSystemPrompt()

位于 src/utils/systemPrompt.ts:41，它决定了最终使用哪套系统提示词。优先级从高到低：

0Override System Promptloop mode 设置，完全替代
|
1Coordinator Prompt协调者模式
|
2Agent System Prompt主线程 Agent 定义的提示词
|
3Custom System Prompt--system-prompt CLI 参数
|
4Default System Prompt标准 Claude Code 提示词

无论选择了哪套主提示词，appendSystemPrompt 总是追加在末尾（除非 override 模式）。

2 CLAUDE.md 多级加载机制

CLAUDE.md 是 Claude Code 最核心的用户指令系统。它实现了一个 四级优先级 的文件加载机制，定义在 src/utils/claudemd.ts 的开头注释中。

四级加载顺序

优先级类型路径说明
1 低Managed/etc/claude-code/CLAUDE.md全局策略指令（企业管理员）
2User~/.claude/CLAUDE.md用户私有全局指令
3ProjectCLAUDE.md, .claude/CLAUDE.md, .claude/rules/*.md项目指令（提交到仓库）
4 高LocalCLAUDE.local.md私有项目指令（不提交）

文件以反优先级顺序加载（Managed 先加载），因为模型对最后出现的内容关注度更高。越靠后 = 越高优先级。

目录遍历逻辑

Project 和 Local 类型通过从 CWD 向上遍历至根目录来发现文件。越接近 CWD 的文件优先级越高：

src/utils/claudemd.ts:851-858
const dirs: string[] = []
const originalCwd = getOriginalCwd()
let currentDir = originalCwd

while (currentDir !== parse(currentDir).root) {
  dirs.push(currentDir)
  currentDir = dirname(currentDir)
}
// 从 root 向下处理（root -> CWD），确保 CWD 层级最后加载
for (const dir of dirs.reverse()) { ... }

Rules 目录系统

除了 CLAUDE.md 文件，每级还支持 .claude/rules/*.md 目录，通过 processMdRules() 递归加载所有 .md 文件。Rules 文件还支持 条件匹配机制 -- 通过 frontmatter 的 paths 字段指定 glob 模式：

.claude/rules/react-components.md
---
paths:
  - src/components/**
  - src/pages/**
---
Use React functional components with hooks.
Always add proper TypeScript types.

@include 指令

Memory 文件支持 @path 语法来引用其他文件，最大嵌套深度为 5 层。路径解析通过 extractIncludePathsFromTokens() 基于 marked lexer 实现，仅处理文本节点中的 @ 引用，跳过代码块：

src/utils/claudemd.ts:459
const includeRegex = /(?:^|\s)@((?:[^\s\\]|\\ )+)/g
// 支持: @path, @./relative, @~/home, @/absolute

HTML 注释剥离

加载时会自动剥离 CLAUDE.md 中的块级 HTML 注释（<!-- ... -->），允许用户在指令文件中添加不被注入的注释。这通过 stripHtmlComments() 使用 marked 的 lexer 在块级别精确识别。

内容格式化输出

getClaudeMds() 函数负责将所有加载的文件格式化为最终字符串，每个文件加上路径和类型描述：

src/utils/claudemd.ts:1153-1194
export const getClaudeMds = (memoryFiles: MemoryFileInfo[]): string => {
  for (const file of memoryFiles) {
    const description =
      file.type === 'Project'
        ? ' (project instructions, checked into the codebase)'
        : file.type === 'User'
          ? " (user's private global instructions for all projects)"
          : ...
    memories.push(`Contents of ${file.path}${description}:\n\n${content}`)
  }
  return `${MEMORY_INSTRUCTION_PROMPT}\n\n${memories.join('\n\n')}`
}

其中 MEMORY_INSTRUCTION_PROMPT 是一段关键的覆盖指令：

"Codebase and user instructions are shown below. Be sure to adhere
to these instructions. IMPORTANT: These instructions OVERRIDE any
default behavior and you MUST follow them exactly as written."

3 Memory 注入系统

Claude Code 拥有两套记忆系统：CLAUDE.md 指令文件（上一章）和自动记忆（Auto Memory / Team Memory）。两者都通过 system prompt 注入，但路径不同。

自动记忆（memdir）

自动记忆的核心位于 src/memdir/ 目录。入口点是 MEMORY.md 文件，存储在 ~/.claude/projects/<slug>/memory/ 目录下。

src/memdir/memdir.ts:34-38
export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000

MEMORY.md 是一个索引文件，每条记忆指向独立的主题文件。超过 200 行或 25KB 的内容会被截断，并附加警告信息。

记忆注入到 System Prompt

loadMemoryPrompt() 是记忆注入到系统提示词的关键函数，它作为一个 systemPromptSection 注册在动态区段中：

src/constants/prompts.ts:495
systemPromptSection('memory', () => loadMemoryPrompt()),

这意味着记忆内容在会话中只计算一次，之后被缓存直到 /clear 或 /compact。

记忆类型分类系统

buildMemoryLines() 定义了一个四类型分类法（user / feedback / project / reference），并生成包含创建/搜索指南的完整行为指令。模型被指导将记忆写入独立文件，并在 MEMORY.md 中维护索引。

MEMORY.md 通过 CLAUDE.md 管道注入

一个关键的设计：MEMORY.md 的内容同时通过 CLAUDE.md 管道以 AutoMem 类型加载（claudemd.ts:980-992），这意味着它也作为 getUserContext() 的一部分被注入到 <system-reminder> 中。

src/utils/claudemd.ts:980-992
if (isAutoMemoryEnabled()) {
  const { info: memdirEntry } = await safelyReadMemoryFileAsync(
    getAutoMemEntrypoint(), 'AutoMem',
  )
  if (memdirEntry) {
    result.push(memdirEntry)  // 加入 memoryFiles 数组
  }
}

Team Memory

团队记忆（TeamMem 类型）通过 teamMemPaths 模块管理，内容用 <team-memory-content source="shared"> 标签包裹，以区分来源。

4 工具声明注入策略

Claude Code 中的工具声明不是嵌入在 system prompt 文本中，而是通过 Anthropic Messages API 的 tools 参数独立传递。但 system prompt 中包含了大量关于工具使用的行为指引。

工具声明序列化

工具使用指引在 System Prompt 中

getUsingYourToolsSection() 根据当前启用的工具集动态生成指引。核心理念是引导模型优先使用专用工具而非 Bash：

src/constants/prompts.ts:291-301
const providedToolSubitems = [
  `To read files use ${FILE_READ_TOOL_NAME} instead of cat`,

工具延迟加载（Deferred Tools）

通过 defer_loading 标记，部分工具（如 MCP 工具）可以在初始请求中仅声明名称，由 ToolSearch 工具在运行时动态获取完整 schema。这减少了初始 prompt 大小。变更通过 deferred_tools_delta 附件增量通知。

Swarm 字段过滤

当 Agent Swarm 未启用时，filterSwarmFieldsFromSchema() 会从工具 schema 中移除 swarm 相关字段（如 launchSwarm、teammateCount），避免混淆模型。

5 system-reminder 动态插入

<system-reminder> 是 Claude Code 在运行时向模型注入上下文信息的核心载体。它不是系统提示词的一部分，而是作为消息流中的用户消息注入的。

注入机制：prependUserContext()

位于 src/utils/api.ts:449，这个函数将 getUserContext() 的内容包装为 <system-reminder> 标签，作为消息列表的第一条消息（元消息）：

src/utils/api.ts:449-474
export function prependUserContext(
  messages: Message[],
  context: { [k: string]: string },
): Message[] {
  return [
    createUserMessage({
      content: `<system-reminder>
As you answer the user's questions, you can use the following context:
${Object.entries(context)
  .map(([key, value]) => `# ${key}\n${value}`)
  .join('\n')}

      IMPORTANT: this context may or may not be relevant to your tasks.
      You should not respond to this context unless it is highly
      relevant to your task.
</system-reminder>`,
      isMeta: true,
    }),
    ...messages,
  ]
}

userContext 的内容来源

getUserContext()（位于 src/context.ts:155）返回一个 key-value 对象，包含：

  • claudeMd -- 所有 CLAUDE.md 内容的合并输出
  • currentDate -- 当前日期（Today's date is YYYY-MM-DD.）

systemContext 的追加方式

与 userContext 不同，getSystemContext() 的内容直接追加到 system prompt 数组的末尾：

src/query.ts:449-451

  appendSystemContext(systemPrompt, systemContext),

// appendSystemContext 只是将 key:value 对拼接成一个字符串追加到数组末尾

systemContext 包含 gitStatus（初始 git 状态快照）和可选的 cacheBreaker（调试用注入）。

system prompt 声明标签的作用

System prompt 本身也告诉模型如何理解 <system-reminder>：

src/constants/prompts.ts:132-133

8 用户指令优先级体系

理解 Claude Code 中不同指令源的优先级是关键。整个系统可以从两个维度来看：System Prompt 层和消息流层。

System Prompt 层的优先级

在 system prompt 数组中，越靠后的内容越被重视。getSystemPrompt() 的返回顺序：

// 1. 静态行为指引（模型身份、安全规则、工具使用） getSimpleIntroSection --> 模型身份定义 getSimpleSystemSection --> 系统行为规则 getSimpleDoingTasksSection --> 代码任务指引 getActionsSection --> 操作安全（可逆性考量） getUsingYourToolsSection --> 工具使用规范 getOutputEfficiencySection --> 输出效率要求 // === SYSTEM_PROMPT_DYNAMIC_BOUNDARY === // // 2. 动态内容（会话特定、用户特定） session_guidance --> 会话特定引导 memory --> 自动记忆指令 (loadMemoryPrompt) env_info --> 环境信息（CWD、Git、OS） language --> 语言偏好 mcp_instructions --> MCP 服务器指令 scratchpad --> 暂存目录 frc --> 函数结果清理 // 3. systemContext 追加 gitStatus --> Git 状态快照

消息流层的优先级

在消息流中，CLAUDE.md 的内容以 <system-reminder> 形式作为第一条用户消息注入。关键文本指出：

"These instructions OVERRIDE any default behavior and you MUST follow
them exactly as written."

这意味着 CLAUDE.md 中的指令优先级高于系统提示词中的默认行为。

CLAUDE.md 内部优先级

加载顺序类型实际优先级原因
1Managed最低（但不可覆盖）企业策略，位置最前
2User (~/.claude/CLAUDE.md)中低全局个人偏好
3Project (CLAUDE.md)中高项目级指令
4Local (CLAUDE.local.md)最高位置最后，模型关注度最高

设置源控制

每级加载都受到设置源开关的控制：

  • isSettingSourceEnabled('userSettings') -- 控制 User 级别
  • isSettingSourceEnabled('projectSettings') -- 控制 Project 级别
  • isSettingSourceEnabled('localSettings') -- 控制 Local 级别
  • claudeMdExcludes 设置 -- 通过 glob 模式排除特定文件

排除机制

isClaudeMdExcluded() 支持通过 picomatch 模式匹配排除特定 CLAUDE.md 文件，并处理了 macOS 上 /tmp -> /private/tmp 的符号链接问题。

9 Prompt 缓存与性能优化

系统提示词的构建涉及多层缓存策略，这对控制 API 开销和延迟至关重要。

静态/动态边界（Dynamic Boundary）

系统提示词被一个特殊标记 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分为两段：

src/constants/prompts.ts:114
export const SYSTEM_PROMPT_DYNAMIC_BOUNDARY =
  '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'

这个标记在 splitSysPromptPrefix() 中被识别，将 prompt 分为最多 4 个缓存块：

Global Cache 模式（1P 且有边界标记）
块cacheScope说明
Attribution Headernull计费头
System Prompt PrefixnullCLI 标识前缀
Static Content'global'边界前的所有内容，跨 org 可缓存
Dynamic Contentnull边界后的内容，包含用户特定数据

'global' 缓存范围意味着这些内容可以在所有组织间共享缓存，极大减少重复计算。

systemPromptSection 缓存

动态区段通过 systemPromptSections.ts 提供两种注册方式：

src/constants/systemPromptSections.ts
// 普通区段：计算一次后缓存，直到 /clear 或 /compact
systemPromptSection(name, computeFn)

// 危险区段：每轮重新计算，会破坏 prompt 缓存
DANGEROUS_uncachedSystemPromptSection(name, computeFn, reason)

缓存存储在 getSystemPromptSectionCache()（位于 bootstrap state 中），/clear 和 /compact 时通过 clearSystemPromptSections() 重置。

getUserContext / getSystemContext 缓存

这两个函数都使用 lodash.memoize() 实现单次计算缓存：

  • getUserContext() -- CLAUDE.md 内容 + 日期，会话内稳定
  • getSystemContext() -- Git 状态 + 注入，会话内稳定

缓存失效通过显式调用 .cache.clear?.() 实现。

getMemoryFiles 缓存

getMemoryFiles() 也使用 memoize()，确保目录遍历和文件读取只在会话启动时执行一次。通过 resetGetMemoryFilesCache() 在压缩后重置，并带有 InstructionsLoaded hook 触发机制。

工具 Schema 缓存

toolSchemaCache 确保工具的 JSON Schema 在会话中只序列化一次，防止 GrowthBook 特性开关翻转导致的中途变化破坏缓存。

MCP 指令增量策略

MCP 服务器指令从每轮在 system prompt 中重复（DANGEROUS_uncachedSystemPromptSection）改为增量附件模式（mcp_instructions_delta），只在服务器连接/断开时发送变更，大幅减少了缓存失效。

设计原则总结

1. 分层覆盖（Layered Override）

系统默认行为 < CLAUDE.md 指令 < 用户实时输入。每一层都可以被上层覆盖，但下层提供了合理的默认值。

2. 缓存优先（Cache-First）

通过 Dynamic Boundary 标记、memoize 缓存、增量附件等手段，最大化 prompt 的可缓存比例。静态内容用 global scope 跨组织缓存。

3. 关注点分离（Separation of Concerns）

工具声明走 API tools 参数，用户指令走 system-reminder 消息，系统行为走 system prompt，动态上下文走 attachments。每个通道有独立的缓存和更新策略。

4. 安全声明前置（Safety First）

CYBER_RISK_INSTRUCTION 在身份定义之后立即出现，操作安全（getActionsSection）独立为一个完整区段，确保安全规则在模型的注意力高峰期被读取。
