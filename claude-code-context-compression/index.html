01

为什么需要上下文压缩

LLM 对话的根本矛盾：模型的上下文窗口有限，但软件工程任务天然需要长对话。一次复杂的代码重构可能涉及数十次文件读取、Bash 执行、搜索查询，每次工具调用的结果都会吃掉大量 token。

200K
标准上下文窗口
13K
自动压缩缓冲区
20K
摘要预留 Token
3
最大连续失败容忍

Claude Code 的解决方案不是简单地截断历史消息，而是设计了一套三层递进的压缩体系，在保证上下文连贯性的同时，将 token 使用量控制在安全范围内。

核心洞察：压缩不是一次性操作，而是一条管道 — 消息在到达 API 之前，会依次经过 snip 裁剪、微压缩、上下文折叠、自动压缩四道关卡，每道关卡解决不同层次的问题。
02

三层压缩架构

压缩系统按成本和侵入性从低到高分为三层，每层有明确的职责边界：

LAYER 1 — MICRO COMPACT
微压缩：清除工具结果
零 API 调用。直接删除旧的工具结果内容（file_read、bash、grep 等），只保留工具调用的结构。成本最低，速度最快，每次查询前自动运行。
LAYER 2 — SESSION MEMORY COMPACT
会话内存压缩：利用已有摘要
零额外 API 调用。如果后台的会话内存提取器已经生成了摘要，直接复用该摘要替换旧消息。比完整压缩便宜得多，优先尝试。
LAYER 3 — FULL COMPACT
完整压缩：AI 生成摘要
需要一次完整的 API 调用。将整段对话发送给模型，让它生成结构化的详细摘要，然后用摘要替换原始消息。成本最高，但信息保留最完整。
优先级策略：自动压缩时，系统总是先尝试 Layer 2（会话内存），失败后才回退到 Layer 3（完整压缩）。微压缩（Layer 1）作为前置步骤始终运行。

触发机制与令牌预算

自动压缩的触发基于精确的令牌计算，而非简单的消息条数。核心公式如下：

autoCompact.ts — 阈值计算
// 有效上下文窗口 = 模型上下文窗口 - 摘要预留空间
const effectiveWindow = contextWindow - Math.min(
  maxOutputTokens,
  20_000  // MAX_OUTPUT_TOKENS_FOR_SUMMARY
)

// 自动压缩阈值 = 有效窗口 - 缓冲区
const threshold = effectiveWindow - 13_000
// AUTOCOMPACT_BUFFER_TOKENS

系统维护四个递进的阈值级别，每个级别触发不同的行为：

阈值 缓冲区 触发行为
自动压缩阈值 -13,000 静默触发自动压缩，用户无感知
警告阈值 -20,000 UI 显示黄色警告，提示上下文即将用尽
错误阈值 -20,000 UI 显示红色错误，强烈建议手动压缩
阻断限制 -3,000 阻止发送请求，返回合成 413 错误

shouldAutoCompact() 函数包含关键的递归防护逻辑，防止压缩过程中的 fork 代理触发新的压缩：

微压缩：零成本的第一道防线

微压缩的核心思想极其简单：工具调用的结果在被模型看过一次之后，就不再需要完整保留。删除旧的工具结果内容，不影响模型对对话上下文的理解。

时间基础微压缩

  • 超过配置的时间间隔后触发
  • 适用于用户长时间离开后返回的场景

  • 仅在内部构建（ant）中启用

05

完整压缩：AI 驱动的摘要生成

当微压缩和会话内存都无法将 token 降到阈值以下时，系统启动完整压缩 — 发起一次专用的 API 调用，让模型对整段对话生成结构化摘要。

1. 预处理消息
剥离图像（替换为 [image] 占位符）、移除会重新注入的附件（技能列表等），减少发送给压缩模型的 token 量。
2. 执行前置 Hook
触发 pre_compact 钩子，允许用户通过 settings.json 配置的脚本注入自定义压缩指令。Hook 返回的 newCustomInstructions 会合并到压缩提示词中。
3. 流式调用摘要 API
通过 streamCompactSummary() 发起 API 调用。优先使用 fork 代理（复用主对话的缓存前缀），失败后回退到常规流式路径。thinking 模式被禁用以节省 token。
4. Prompt-too-long 重试
如果 API 返回 prompt-too-long 错误，按 API 轮次分组消息，从最旧的组开始逐步删除，最多重试 3 次。删除量基于错误响应中的 token 间隙计算，无法解析时回退到删除 20%。
5. 格式化摘要
提取 <summary> 标签内容，去除 <analysis> 思考过程（节省 token），清理多余空行。
6. 清理状态 + 恢复文件
清空 readFileState 缓存和嵌套内存路径，然后从压缩前的文件读取状态中恢复最多 5 个最近的文件作为附件重新注入。
7. 构建压缩结果
生成边界标记（compact_boundary 类型的系统消息），携带压缩前 token 数、触发方式、已发现的工具列表等元数据。所有后续查询只会看到边界之后的消息。
06

摘要提示词工程

压缩质量的核心在于提示词设计。Claude Code 使用了精心设计的提示词模板，强制模型生成结构化、不遗漏的摘要。

关键约束：压缩提示词开头有一段极其严厉的 "NO_TOOLS_PREAMBLE"，用全大写强调模型绝对不能调用任何工具，必须纯文本输出。这是因为压缩 API 调用如果触发了工具调用，会破坏整个流程。

强制摘要结构（9 个必填部分）

1. Primary Request
捕获用户所有的显式请求和意图，详细记录
2. Key Concepts
列出所有重要的技术概念、技术栈和框架
3. Files & Code
枚举具体的文件和代码段，包含完整代码片段和变更原因
4. Errors & Fixes
记录所有遇到的错误及修复方式，特别关注用户反馈
5. Problem Solving
记录已解决的问题和正在进行的排查
6. All User Messages
列出所有非工具结果的用户消息 — 理解用户反馈和意图变化的关键
7. Pending Tasks
列出明确被要求完成但尚未完成的任务
8. Current Work
详细描述压缩前最近的工作内容，包含文件名和代码片段
9. Next Step
列出与最近工作相关的下一步行动（可选）

分析-摘要双阶段输出

提示词要求模型先在 <analysis> 标签中按时间顺序逐条分析对话内容，然后在 <summary> 标签中输出最终摘要。分析阶段的存在是为了提高摘要质量（类似 Chain of Thought），但格式化时会被去除以节省 token。

会话内存压缩：更便宜的替代路径

会话内存压缩是一条"投机取巧"的路径：如果后台的会话内存提取器（一个独立运行的 fork 代理）已经生成了对话摘要，就直接复用它，避免昂贵的完整压缩 API 调用。

10K
最小保留 Token
5
最少保留文本消息
40K
最大保留 Token

消息保留算法

calculateMessagesToKeepIndex() 是会话内存压缩的核心算法，它决定哪些消息需要保留在摘要之后：

定位最后已摘要的消息
通过 lastSummarizedMessageId 找到摘要覆盖的最后一条消息的索引
从后向前计算保留范围
从 lastSummarizedIndex 之后的消息开始计算 token 数和文本消息数，直到同时满足 minTokens（10K）和 minTextBlockMessages（5）
不足时向前扩展
如果摘要之后的消息不够，向前扩展（但不超过上一个压缩边界），直到满足最小值或达到 maxTokens（40K）
调整索引保持 API 不变量
确保 tool_use / tool_result 配对完整，thinking 块不被分割 — 这是最关键的一步，防止 API 调用失败
API 不变量保护：adjustIndexToPreserveAPIInvariants() 会向前扫描找到所有孤立的 tool_result，然后回溯找到它们对应的 tool_use 所在的 assistant 消息，将整个配对包含在保留范围内。同样的逻辑也适用于 thinking 块和 message.id 的一致性。

压缩后恢复：重建上下文

压缩不只是生成摘要 — 更关键的是压缩后如何恢复必要的上下文，让模型能够无缝继续工作。

压缩后消息重建

compact.ts — buildPostCompactMessages
function buildPostCompactMessages(result) {
  return [
    result.boundaryMarker,     // 压缩边界标记
    ...result.summaryMessages,  // AI 生成的摘要
    ...(result.messagesToKeep), // 保留的原始消息
    ...result.attachments,      // 恢复的文件和工具
    ...result.hookResults,      // Hook 执行结果
  ]
}

查询管道完整集成

每次用户发送消息后，消息在到达 API 之前会经过一条精密的处理管道。压缩逻辑散布在这条管道的多个节点中：

  用户输入
      │
      ▼
  getMessagesAfterCompactBoundary()  ← 只取最后边界之后的消息
      │
      ▼
  applyToolResultBudget()              ← 强制单消息工具结果大小限制
      │
      ▼
  snipCompactIfNeeded()               ← HISTORY_SNIP: 裁剪最旧的消息
      │                                        记录 snipTokensFreed
      ▼
  microcompactMessages()              ← Layer 1: 清除旧工具结果内容
      │
      ▼
  contextCollapse.applyIfNeeded()      ← 可选: 上下文折叠（CONTEXT_COLLAPSE）
      │
      ▼
  shouldAutoCompact()                 ← tokenCount - snipTokensFreed > threshold?
      │
      ├── YES ──▶ autoCompactIfNeeded()
      │            ├── trySessionMemoryCompaction()  ← Layer 2: 优先
      │            └── compactConversation()         ← Layer 3: 回退
      │
      └── NO
      │
      ▼
  normalizeMessagesForAPI()            ← 规范化消息格式
      │
      ▼
  queryModelWithStreaming()            ← 发送到 Anthropic API

边界标记的作用

压缩边界（compact_boundary）是整个系统的锚点。它是一条特殊的系统消息，携带丰富的元数据：

元数据字段 类型 用途
trigger 'manual' | 'auto' 标识压缩是用户手动触发还是自动触发
preTokens number 压缩前的 token 总数，用于分析和监控
preCompactDiscoveredTools string[] 压缩前已通过 ToolSearch 发现的工具列表
preservedSegment {head, anchor, tail} 部分压缩时保留的消息段的 UUID 范围

getMessagesAfterCompactBoundary() 函数通过 findLastCompactBoundaryIndex() 找到最后一个边界标记，只返回其后的消息 — 这意味着边界之前的所有原始消息对后续查询来说都是"不可见"的。

容错与弹性设计

生产环境中的压缩不可能总是成功。Claude Code 的压缩系统包含多层容错机制：

断路器模式
MAX_CONSECUTIVE_FAILURES = 3
连续 3 次自动压缩失败后停止重试，防止无限循环浪费 API 调用。据内部数据，此前有会话累计达 3,272 次失败，每天全球浪费约 250K API 调用
Prompt-too-long 重试
MAX_PTL_RETRIES = 3
API 返回 413 时，按轮次分组逐步删除最旧的消息。计算删除量基于响应中的 token 间隙，无法解析时删除 20% 的组
流式重试
MAX_STREAMING_RETRIES
流式响应中断时自动重试，带指数退避延迟。通过 feature flag 控制是否启用
缓存共享降级
fork agent fallback
fork 代理（复用缓存前缀）失败时，自动降级到常规流式路径，确保压缩总能完成

压缩后清理

runPostCompactCleanup() 在每次成功压缩后执行全局状态清理：

resetMicrocompactState()        // 重置微压缩追踪

设计哲学：每次压缩等同于一次"软重启"。系统不仅替换消息，还重置几乎所有模块级状态，确保压缩后的行为与新会话开始时一致。这解释了为什么压缩后会重新运行 session_start hook。

总结与评价

设计亮点

  • 三层递进策略，成本从零到高逐级升级
  • 会话内存优先，避免不必要的 API 调用
  • 结构化摘要提示词，信息保留率极高
  • API 不变量保护，确保工具对完整性
  • 断路器 + 重试组合，兼顾弹性和成本
  • 压缩后六类恢复注入，上下文无缝延续
  • 边界标记元数据丰富，支持分析和调试

复杂度代价

  • 4000+ 行压缩代码，认知负担极高
  • 多个 feature flag 交织，测试矩阵庞大
  • 缓存微压缩仅限内部构建，外部用户无法受益
  • 完整压缩的 API 调用本身消耗大量 token
  • 摘要质量依赖模型能力，存在信息损失风险
  • 断路器触发后无自动恢复，需用户干预

核心判断

Claude Code 的上下文压缩系统是目前业界最成熟的 LLM 对话状态管理实现之一。它不是简单的"截断 + 摘要"，而是一套完整的状态机 — 从令牌感知的智能触发、三层递进的压缩策略、到精心设计的上下文恢复，每个环节都经过深思熟虑的工程优化。

对于想要在自己的 AI 应用中实现类似机制的开发者，最值得借鉴的三个设计：1) 微压缩作为零成本的第一道防线；2) 结构化摘要提示词的 9 部分框架；3) 压缩后作为"软重启"的状态恢复逻辑。
