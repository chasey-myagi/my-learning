01

六种权限模式：从只读到全自动

Claude Code 的权限控制核心是 PermissionMode 类型。定义于 src/types/permissions.ts， 它包含六种模式，按安全等级从高到低排列。用户通过 Shift+Tab 循环切换，由 getNextPermissionMode() 控制流转顺序。

模式 符号 行为 适用场景
plan ⏸ 只规划不执行，所有工具调用均被阻止 复杂任务前的方案评审
default - 读操作自动放行，写操作需用户确认 日常开发的默认状态
acceptEdits ⏵⏵ 文件编辑 + 特定文件系统命令（mkdir/rm/cp/mv/sed/touch）在工作目录内自动放行 批量重构、文件整理
bypassPermissions ⏵⏵ 所有工具调用自动放行，跳过全部权限检查 可信环境中的完全信任模式
auto ⏵⏵ AI 分类器自动判定是否允许，无需用户交互 高效自动化工作流（需 feature gate）
dontAsk ⏵⏵ 不弹出任何权限提示，所有 ask 转为 deny CI/CD 管道中的静默拒绝

模式切换流转图

外部用户通过 Shift+Tab 按以下顺序循环：

default → acceptEdits → plan → bypassPermissions* → auto* → default

* bypassPermissions 需 isBypassPermissionsModeAvailable 为 true；auto 需 feature gate 开启且 model 支持

内部用户差异： Anthropic 内部用户（USER_TYPE=ant）的循环路径不同 —— 从 default 直接跳到 bypassPermissions（跳过 acceptEdits 和 plan），因为 auto 模式已取代了中间态。
02

权限决策流水线：从工具调用到最终裁决

每次 AI 发起工具调用时，权限系统执行一条严格的决策链。核心入口是 hasPermissionsToUseTool() （src/utils/permissions/permissions.ts），它先调用 hasPermissionsToUseToolInner() 获得初始判定， 再叠加模式特有的后处理逻辑。

决策链概览

工具调用 → 模式检查 → Deny 规则 → Allow 规则 → tool.checkPermissions() → 后处理

后处理阶段根据模式的不同走不同分支：

  • dontAsk 模式： ask → deny（静默拒绝），附带 DONT_ASK_REJECT_MESSAGE
  • auto 模式： ask → 先尝试 acceptEdits 快速路径 → 安全工具白名单 → AI 分类器
  • headless 代理： 运行 PermissionRequest hooks → 无 hook 决策则 auto-deny
  • 交互模式： 弹出权限对话框，等待用户操作

三种决策结果

ALLOW 允许

工具立即执行。可附带 updatedInput（修改后的参数）和 decisionReason（决策原因追踪）。

DENY 拒绝

工具不执行，返回拒绝消息给 AI。包含 message 和 decisionReason。

ASK 询问

弹出交互式权限对话框。附带 suggestions（推荐的永久规则）和 pendingClassifierCheck（异步分类器检查）。

决策原因类型系统

PermissionDecisionReason 是一个判别式联合类型，精确记录每个决策的来源：

// src/types/permissions.ts (概括)
type PermissionDecisionReason =
  | { type: 'rule'; rule: PermissionRule }       // 匹配了 allow/deny/ask 规则
  | { type: 'mode'; mode: PermissionMode }       // 当前模式决定
  | { type: 'classifier'; classifier: 'auto-mode' } // AI 分类器裁决
  | { type: 'hook'; hookName: string }          // Hook 拦截
  | { type: 'safetyCheck'; classifierApprovable: boolean }
  | { type: 'subcommandResults' }               // 复合命令中子命令分析
  | { type: 'asyncAgent' }                       // 后台代理无法弹窗
  | { type: 'sandboxOverride' }                  // 沙箱外运行
  | { type: 'workingDir' }                       // 工作目录限制
  | { type: 'other'; reason: string }            // 通用兜底
03

权限规则引擎：七层来源的优先级体系

权限规则可以来自七种不同的来源，每种来源的优先级和作用域不同。规则引擎在 src/utils/permissions/permissionsLoader.ts 中加载，在 permissions.ts 中匹配。

来源 (Source) 优先级 作用域 典型配置位置
policySettings 最高 企业级组织策略 IT 管理推送
projectSettings 高 项目级 (git tracked) .claude/settings.json
localSettings 中高 项目本地 (gitignored) .claude/settings.local.json
userSettings 中 用户全局 ~/.claude/settings.json
flagSettings 中低 运行时 CLI flag --settings path
cliArg 低 运行时参数 --allowedTools
session / command 运行时 会话内临时 用户在权限对话框中选择

规则格式详解

每条规则由三部分组成：

type PermissionRule = {
  source: PermissionRuleSource      // 来自哪个配置层
  ruleBehavior: 'allow' | 'deny' | 'ask'  // 行为
  ruleValue: {
    toolName: string               // 工具名, 如 "Bash"
    ruleContent?: string            // 可选的细粒度内容
  }
}

规则值解析

permissionRuleParser.ts 负责将字符串规则解析为结构化对象。它支持转义括号以处理复杂命令：

// 解析示例
"Bash"                       → { toolName: "Bash" }
"Bash(npm install)"          → { toolName: "Bash", ruleContent: "npm install" }
"Bash(python -c \"print\\(1\\)\")" → { toolName: "Bash", ruleContent: 'python -c "print(1)"' }
"mcp__server1"               → 匹配该 MCP 服务器的所有工具
"mcp__server1__*"            → 通配符匹配 server1 的所有工具

组织策略控制

allowManagedPermissionRulesOnly： 当企业通过 policySettings 开启此标志时， 只有来自组织策略的权限规则生效。用户自定义的 allow 规则被忽略，权限对话框中的"Always Allow"选项被隐藏。 这保证了企业安全策略的绝对执行力。
04

Bash 命令权限：Shell 规则三种匹配模式

Bash 命令的权限检查是整个系统中最复杂的部分。核心逻辑在 src/tools/BashTool/bashPermissions.ts 和 src/utils/permissions/shellRuleMatching.ts 中实现。

三种匹配模式

精确匹配 (Exact)

Bash(npm install) — 只匹配完全相同的命令字符串

前缀匹配 (Prefix)

Bash(git commit:*) — 匹配以 git commit 开头的所有命令。使用 :* 后缀语法

通配符匹配 (Wildcard)

Bash(docker * build *) — 支持 * 通配任意字符序列，\* 转义为字面星号

命令前处理

在匹配规则之前，命令会经过一系列安全的预处理：

  • 安全环境变量剥离： SAFE_ENV_VARS 中列出的 30+ 个安全变量（如 NODE_ENV、RUST_LOG、LANG）会被从命令前缀中剥离。这样 NODE_ENV=prod npm run build 可以匹配 npm run:* 规则
  • 注释行剥离： stripCommentLines() 移除纯注释行
  • 子命令提取： getSimpleCommandPrefix() 提取前两个有效 token 作为命令前缀（如 git commit），跳过安全包装器

复合命令处理

当用户运行 npm install && npm test 这样的复合命令时，系统使用 splitCommand_DEPRECATED() 拆分为子命令，每个子命令独立检查权限。如果任一子命令需要 ask，则整个命令需要用户确认。

性能保护： MAX_SUBCOMMANDS_FOR_SECURITY_CHECK = 50。超过 50 个子命令的复合命令直接降级为 ask， 防止恶意或意外的超长命令链导致 CPU 饱和。规则建议数也被限制为 MAX_SUGGESTED_RULES_FOR_COMPOUND = 5。

裸 Shell 前缀黑名单

BARE_SHELL_PREFIXES 集合阻止生成过于宽泛的规则建议：

const BARE_SHELL_PREFIXES = new Set([
  'sh', 'bash', 'zsh', 'fish', 'csh', 'tcsh', 'ksh', 'dash',
  'cmd', 'powershell', 'pwsh',
  'env', 'xargs',               // 包装器
  'nice', 'stdbuf', 'nohup', 'timeout', 'time',  // 安全包装器
  'sudo', 'doas', 'pkexec',     // 提权命令
])

这防止了 Bash(bash:*) 或 Bash(sudo:*) 这样等价于 Bash(*) 的危险规则被自动建议。

05

Tree-sitter AST 安全解析：Fail-Closed 架构

Bash 命令的安全分析引擎（src/utils/bash/ast.ts）基于 tree-sitter-bash 构建了一套 显式白名单 + 失败关闭的 AST 解析架构。这是整个权限系统中最核心的安全组件之一。

设计哲学

"Any node type not in the allowlist causes the entire command to be classified as 'too-complex'" — 源码注释。这不是沙箱，不阻止危险命令运行。它只回答一个问题： "能否为这个命令产生可信赖的 argv[] 数组？" 如果能，下游代码可以安全匹配规则；如果不能，就让用户来决定。

解析结果类型

type ParseForSecurityResult =
  | { kind: 'simple'; commands: SimpleCommand[] }  // 成功解析
  | { kind: 'too-complex'; reason: string }      // 超出白名单 → ask
  | { kind: 'parse-unavailable' }                // tree-sitter 不可用

白名单节点类型

AST 中只有以下节点类型被允许通过：

结构节点

program, list, pipeline, redirected_statement — 用于递归找到叶子 command 节点

分隔符

&&, ||, |, ;, &, |&, \n — 命令间无数据的分隔

命令组件

command 节点下的 word, string, raw_string, variable_assignment, file_redirect 等

SimpleCommand 结构

type SimpleCommand = {
  argv: string[]           // argv[0] = 命令名, 其余为参数
  envVars: { name: string; value: string }[]  // 前置环境变量
  redirects: Redirect[]     // 重定向操作
  text: string              // 原始源码文本
}

命令替换 $() 中的命令会被递归提取，外层 argv 中用 __CMDSUB_OUTPUT__ 占位符替代。 内层命令独立进行权限检查。

语义检查 (checkSemantics)

checkSemantics() 在 AST 解析成功后进行进一步的语义分析。它检查安全包装器（nice/stdbuf/nohup/timeout/time）、 环境变量赋值等模式，确保规则匹配的准确性。

07

Auto Mode 智能分类器：AI 守门人

Auto 模式（src/utils/permissions/yoloClassifier.ts）是权限系统中最先进的组件。 它使用一个独立的 AI 分类器来自动判定每个工具调用是否安全——无需用户交互。

分类器架构

工具调用 (ask) → acceptEdits 快速路径 → 安全白名单 → AI 分类器 API → allow / block

快速路径（Fast Paths）

在调用昂贵的分类器 API 前，系统执行三级快速判断：

  1. acceptEdits 仿真： 以 acceptEdits 模式重新调用 tool.checkPermissions()。 如果该模式下会放行（如工作目录内的文件编辑），直接 allow，跳过分类器。但 Agent 和 REPL 工具被排除。
  2. 安全白名单： SAFE_YOLO_ALLOWLISTED_TOOLS 中列出的 20+ 种只读/元数据工具直接放行， 包括 FileRead、Grep、Glob、LSP、TodoWrite、Sleep 等。
  3. 不可分类器审批的安全检查： safetyCheck 且 classifierApprovable=false 的决策 不受分类器影响，必须回退到用户交互。

分类器 API 调用

// 分类器工具 Schema
const YOLO_CLASSIFIER_TOOL_SCHEMA = {
  name: 'classify_result',
  input_schema: {
    properties: {
      thinking: { type: 'string' },        // 逐步推理过程
      shouldBlock: { type: 'boolean' },    // true=阻止, false=放行
      reason: { type: 'string' },           // 决策理由
    }
  }
}

上下文构建

buildTranscriptEntries() 从对话历史中提取精简的上下文记录。关键设计： 只包含用户文本消息和 AI 的 tool_use 块，排除 AI 文本（因为 AI 文本可能被精心构造以影响分类器决策）。

用户可配置规则

AutoModeRules 允许用户通过 settings.autoMode 自定义三类规则：

type AutoModeRules = {
  allow: string[]       // 允许的操作描述
  soft_deny: string[]   // 软拒绝的操作描述
  environment: string[] // 环境上下文描述
}

容错与熔断

Iron Gate (失败关闭)

当分类器 API 不可用时，tengu_iron_gate_closed feature gate 决定行为。默认 fail-closed：拒绝所有操作并附带重试指引。

连续拒绝限制

maxConsecutive: 3，maxTotal: 20。达到限制后回退到用户交互式权限提示，防止 AI 陷入拒绝死循环。

上下文溢出

当对话历史超过分类器的上下文窗口时，自动回退到手动审批模式。在无头模式下直接中止代理。

熔断器

autoModeCircuitBroken 标志。当 GrowthBook 的 tengu_auto_mode_config.enabled 变为 disabled 时触发，全局阻止 auto 模式。

危险权限检测与 Auto Mode 准入

进入 Auto 模式时，permissionSetup.ts 中的 isDangerousBashPermission() 和 isDangerousPowerShellPermission() 会扫描所有现存的 allow 规则，剥离可能绕过分类器的危险规则。

危险 Bash 模式

DANGEROUS_BASH_PATTERNS 定义了 20+ 种跨平台代码执行入口点：

const CROSS_PLATFORM_CODE_EXEC = [
  // 解释器
  'python', 'python3', 'node', 'deno', 'tsx', 'ruby', 'perl', 'php', 'lua',
  // 包管理器执行器
  'npx', 'bunx', 'npm run', 'yarn run', 'pnpm run', 'bun run',
  // Shell
  'bash', 'sh',
  // 远程执行
  'ssh',
]

// Bash 额外：
'zsh', 'fish', 'eval', 'exec', 'env', 'xargs', 'sudo'

检测逻辑

对每个危险模式，检查五种变体：

  1. 精确匹配： python
  2. 前缀语法： python:*
  3. 尾部通配： python*
  4. 空格通配： python *
  5. 标志通配： python -c* 等 python -开头 + *结尾

PowerShell 特有危险

PowerShell 在上述跨平台模式之外，还检测 25+ 种 PS 特有的危险 cmdlet：

// 嵌套 Shell     'pwsh', 'powershell', 'cmd', 'wsl'
// 字符串执行     'iex', 'invoke-expression', 'icm', 'invoke-command'
// 进程创建       'start-process', 'saps', 'start-job', 'start-threadjob'
// .NET 逃逸     'add-type', 'new-object'
// 事件注册       'register-objectevent', 'register-engineevent'

PowerShell 额外处理 .exe 后缀变体（如 python.exe、npm.exe run）， 因为 Windows 上的实际二进制名包含 .exe。

Agent 工具保护

isDangerousTaskPermission： 任何 Agent（子代理）的 allow 规则都被视为危险规则。 原因：子代理在执行前需要由分类器评估其 prompt，auto-approve 会绕过这一安全层。

三层权限处理器

src/hooks/toolPermission/handlers/ 下有三个不同场景的处理器：

interactiveHandler

集群工作者模式。处理分布式代理的权限请求，通过消息传递与主代理协调。

权限解释器 (Permission Explainer)

src/utils/permissions/permissionExplainer.ts 使用一个独立的 AI 侧查询（sideQuery）来为用户解释命令的含义和风险。 它返回三级风险评估（LOW / MEDIUM / HIGH）加上自然语言解释： 

type PermissionExplanation = {
  riskLevel: 'LOW' | 'MEDIUM' | 'HIGH'
  explanation: string   // 命令做什么 (1-2 句)
  reasoning: string     // 为什么要运行 ("I need to...")

影子规则检测

shadowedRuleDetection.ts 检测不可达的权限规则。例如，如果项目配置了 Bash: deny， 那么用户配置的 Bash(npm install): allow 永远不会被触发。系统会发出警告并建议修复方案。

10

命令语义分析与架构哲学

src/tools/BashTool/commandSemantics.ts 实现了命令退出码的语义解读—— 一个看似不起眼但对用户体验影响深远的系统。

语义覆盖

不同命令的退出码含义不同。默认行为是 exit 0 = 成功，其他 = 错误。但某些命令有特殊语义：

命令Exit 0Exit 1Exit 2+
grep / rg找到匹配未找到匹配（非错误）真正的错误
diff无差异有差异（非错误）真正的错误
find成功部分目录不可访问（非错误）真正的错误
test / [条件为真条件为假（非错误）真正的错误

这确保了 AI 不会把 grep 返回 1（未找到匹配）误判为执行失败，从而避免不必要的重试或错误报告。

架构总结：纵深防御

Claude Code 的权限系统体现了经典的纵深防御（Defense in Depth）思想：

第一层：模式控制

六种模式从全局层面决定了权限的基本姿态——从 plan（只读）到 bypass（全放行）

第二层：规则引擎

七层来源的 allow/deny/ask 规则，支持精确、前缀、通配符三种匹配，组织策略具有最高优先级

第三层：AST 安全解析

基于 tree-sitter 的 fail-closed 解析器，40+ 种危险模式检测，确保命令语义正确理解

第四层：AI 分类器

Auto 模式下的独立 AI 守门人，带连续拒绝限制、上下文溢出回退、熔断器机制

第五层：交互式确认

当上述层次都无法自动决策时，回退到用户交互，附带风险解释和规则建议

第六层：沙箱隔离

SandboxManager 提供进程级隔离，即使命令被允许执行，也在受限环境中运行

设计哲学： 权限系统的每一层都遵循"fail-closed"原则——在不确定时拒绝。 从 AST 解析器遇到未知节点类型时返回 too-complex，到分类器 API 不可用时触发 Iron Gate 拒绝， 再到复合命令超过 50 个子命令时降级为 ask。系统宁可多问用户一次，也不会放过一个未经审查的操作。
